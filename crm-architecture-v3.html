L2

API Gateway — 5 Middleware (sequential)

gateway pipeline
  Request ──▶ Load Balancer ──▶ Tenant Resolver ──▶ Auth Guard ──▶ Rate Limiter ──▶ Route Matcher
               TLS terminate     subdomain/header    JWT validate    sliding window    URL → plugin
               round-robin       → Redis cache        + blacklist     Redis ZADD        + method
               pick instance     → Metadata DB        + RBAC check    per-tier limit    + permission
                                  (on miss)            + tenant match                    check

  Mỗi middleware có thể REJECT request → trả lỗi ngay, không đi tiếp.

  ┌──────────────────────────────────────────────────────────────────────────────┐
  │ Route Matcher output:                                                       │
  │   plugin: "customer-care", method: "createSale", permission: "sales:write" │
  │   → L3 biết chính xác plugin nào xử lý, method nào gọi.                   │
  └──────────────────────────────────────────────────────────────────────────────┘
L3

Business Logic — Plugin System + Context + Services

L3 components
PLUGIN MANAGEMENT (system-level, singleton):

  Plugin Manager ────── lifecycle: deploy, upgrade, rollback, canary
  Plugin Registry ───── manifests, routes, events, hooks, permissions, deps
  Sandbox Engine ────── VM isolate, resource limits (5s/50MB/50 queries)
  Event Bus ─────────── sync in-process + async MQ publish
  Hook Registry ──────── before/after hooks, runtime filter by enabledPlugins

PLUGIN CORES (singleton, stateless — code chạy trong sandbox):

  CustomerDataCore ──── customers, companies, contacts (tables owned)
  CustomerCareCore ──── sales, call_logs, tickets (tables owned)
  AnalyticsCore ──────── report_snapshots, dashboards (tables owned)
  AutomationCore ────── rules, email_templates
  MarketingCore ─────── campaigns, segments

  Plugin cores KHÔNG gọi nhau trực tiếp.
  Tương tác qua: events, hooks, hoặc đọc table nhau qua ctx.db.

CONTEXT BUILDING:

  Tenant Context Manager ── LRU cache max 1000, TTL 2h
                            Chứa: tier, enabledPlugins, dbPool, quotas
                            Source: Metadata DB (on cache miss)

  Execution Context Builder ── tạo per-request (~15KB)
                               Wrap: DB conn, cache, logger, eventBus
                               Acquire từ Pool Manager → release sau response

SHARED SERVICES (singleton, called via ctx.services):

  Email ──── template engine + MQ async delivery
  Storage ── S3 wrapper, auto tenant-prefix /tenants/{id}/
  Notification ── WebSocket + Push
  Search ── Elasticsearch wrapper, auto tenant index
  Scheduler ── cron, delayed tasks (tenant-aware)
  MQ Publisher ── publish to RabbitMQ exchanges
L4

Data Access — 4 thực thể, cổng duy nhất tới data

L4 detail
QUERY INTERCEPTOR (tạo per-request, wrap quanh DB connection)
│
├── TenantInjector:    auto-add tenant_id vào SELECT/INSERT/UPDATE/DELETE
├── QueryValidator:    6 safety checks (tenant_id? parameterized? allowed table?)
├── QueryCounter:      [n/50] limit (sandbox), block khi vượt
└── QueryLogger:       slow query >100ms → WARN, >500ms → ALERT

  Double safety: Interceptor (L4) + PostgreSQL RLS (L5)
  → Cần CẢ HAI bị compromised mới leak data.


POOL MANAGER (singleton, tạo khi bootstrap)
│
├── PoolRegistry:        metadata(20), shared(200), vip-initech(30), vip-stark(30)
├── PoolRouter:          tenantCtx.dbPool → đúng pool instance
├── ConnectionConfigurator: SET/RESET app.current_tenant per connection
├── HealthChecker:       30s ping, dead conn → replace
└── TenantQuotaEnforcer: per-tenant cap 20 conns (shared pool) → chống starvation


CACHE MANAGER (singleton, creates TenantScopedCache per-request)
│
├── L1: In-memory (per instance, 100MB, TTL 60s, LRU)       ~0.01ms
├── L2: Redis cluster (shared, TTL 5-15min, LZ4 compressed) ~1-2ms
├── L3: Database (source of truth)                            ~5-20ms
└── Invalidation: write → delete L1+L2 → Pub/Sub → all instances clear L1


MIGRATION ENGINE (chạy khi admin operation, KHÔNG per-request)
│
├── Schema Migrations (DDL): CREATE TABLE IF NOT EXISTS, ADD COLUMN IF NOT EXISTS
│   → Chạy 1 lần cho toàn hệ thống khi plugin deploy/upgrade
│   → Track: schema_migrations (plugin, version) — không có tenant
│   → Backward compatible bắt buộc (shared tables!)
│
├── Tenant Migrations (DML): seed data, RLS policies, default configs
│   → Chạy per-tenant mỗi lần enable plugin
│   → Track: tenant_plugin_migrations (tenant, plugin, version)
│
├── Advisory Lock: pg_advisory_lock per operation → no race condition
└── Dependency enforcement: cannot disable plugin X if Y depends on X
L5

Infrastructure — 5 stateful services

L5 detail
POSTGRESQL (3 loại instance)
│
├── Metadata DB (#1): tenants, tenant_plugins, plugin_registry,
│   plugin_migrations, schema_migrations, audit_log, webhooks, system_config
│   → LOW traffic (cached), HIGH write (audit_log only, partitioned monthly)
│
├── Shared Data DB (#2): business tables của MỌI standard tenants
│   + 3 Read Replicas (analytics, search indexing, hot standby)
│   + PgBouncer (200 app conns → 50 DB conns)
│   → RLS on every table, Query Interceptor double safety
│
└── VIP Data DBs (#3, #4...): 1 instance per VIP tenant
    Same schema, custom tuning, own backup schedule


REDIS CLUSTER (3 Master + Sentinel — 6 vai trò)
│
├── ❶ L2 Cache:          t:{tenantId}:{plugin}:{entity}:{hash}, TTL 5-15min
├── ❷ Domain Map:         domain-map:{subdomain} → tenantId, TTL 5min
├── ❸ Rate Limit:         rl:{tenantId} (Sorted Set, sliding window)
├── ❹ Token Blacklist:    blacklist:{jti}, TTL = token remaining lifetime
├── ❺ Pub/Sub:            cache:invalidate, context:invalidate, plugin:deployed
└── ❻ Realtime Counters:  t:{tenantId}:stats:{metric} (INCR, periodic sync to DB)


RABBITMQ CLUSTER (3 nodes, quorum queues)
│
├── Exchange crm.events (topic): routing key {entity}.{action}
│   → Bindings: customer.* → notification, *.created → search.index, etc
│   → 1 message → multiple queues (fan-out by routing key)
│
├── Exchange crm.system (fanout): plugin.deployed, config.updated
│
├── Queues: notification(3w), search.index(5w), webhook(3w), analytics(2w), email(3w)
│   → Per-queue DLQ, 3 retries with exponential backoff (5s→30s→2min)
│
└── VIP Priority: messages with priority 10 (VIP) / 5 (standard) / 1 (basic)


S3/MINIO: /tenants/{id}/uploads|exports|backups, /system/plugin-artifacts|audit-archives
ELASTICSEARCH: per-tenant indices ({tenantId}-customers, {tenantId}-sales), ~1-2s lag
L6+L7

Cross-cutting + Observability

L6 + L7
L6 CROSS-CUTTING (vertical, xuyên suốt L1-L5):

  Security ───────── Encryption (AES-256 rest, TLS 1.3 transit), PII masking,
                     tenant isolation enforcement, secret mgmt (Vault/KMS)
  Error Framework ── Typed errors, circuit breaker, retry, bulkhead, fallback
                     Plugin error boundary (no cascade crash)
  Config ─────────── Global/Tier/Tenant levels, hot reload, feature flags
  Dependency Mgmt ── Plugin dependency resolver + disable chain enforcement
                     (v2 gộp vào Plugin Manager — v3 tách ra L6 vì xuyên tầng)


L7 OBSERVABILITY (6 components, passive, ~1ms overhead):

  Logging ─────── ScopedLogger per-request, auto {requestId, tenantId, userId, plugin}
                 JSON → stdout → Promtail → Loki/ELK. PII redacted. 5 levels.
  Tracing ─────── OpenTelemetry spans per operation → Jaeger. Context propagation
                 qua HTTP → Exec Context → DB comments → MQ headers → workers.
                 Sampling: 5% random + 100% errors + 100% slow. VIP 20%.
  Metrics ─────── Prometheus scrape /metrics 15s. Catalog: gateway (4), business (5),
                 data-access (7), infrastructure (7). Histograms for P50/P95/P99.
  Alerting ────── Alertmanager → PagerDuty/Slack/Email. 3 severity levels.
                 Escalation: 0min→5min→15min→30min→60min. VIP timers ×50%.
  Health ──────── Liveness (process), Readiness (dependencies), Dependency (detail).
                 K8s probes: live 10s, ready 5s. Fail ready → remove from LB.
  Audit Trail ── Immutable append-only. Data mutations only. before/after state.
                 Async batched (1s buffer). Partitioned monthly → S3 archive.
                 ≠ Logging (Audit = compliance/legal, Logging = debug/monitor).

│  ┌─────────────────────────────────────────────────────────────┐

corrections & additions
SỬA SAI (v2 mô tả sai so với thiết kế chi tiết):

  ① Enterprise tier: "Shared (Schema)" → sửa thành "Shared (RLS)"
     v2 ghi Enterprise dùng "Shared (Schema)" — ngụ ý schema-level isolation.
     Thực tế Enterprise dùng shared tables + RLS, giống Premium.
     Chỉ VIP có dedicated DB.

  ② Migration Engine: "per plugin, transactional" → sửa thành 2 loại migration
     v2 mô tả migration chỉ chạy "on plugin enable, transactional".
     Thực tế: Schema DDL (toàn DB, 1 lần) + Tenant DML (per-tenant, mỗi enable).
     Shared tables = KHÔNG THỂ DROP TABLE khi 1 tenant disable.

  ③ Migration "Down migrations on disable" → sửa thành soft disable (mặc định)
     v2 ghi "Down migrations on disable" → ngụ ý DROP TABLE.
     Thực tế: soft disable = chỉ set enabled=false. Tables + data giữ nguyên.
     Hard purge (DELETE rows, không DROP TABLE) chỉ khi tường minh yêu cầu.

  ④ VIP: "Separate Redis per VIP tenant" → sửa thành "Redis shared, dedicated optional"
     v2 Deployment Topology ghi VIP có "Separate PostgreSQL + Redis".
     Thực tế: VIP isolation chủ yếu ở PostgreSQL (bottleneck chính).
     Redis/MQ shared mặc định, dedicated chỉ khi workload justify.

  ⑤ VIP: "dedicated Redis, higher memory" → inconsistent
     Multi-tenancy panel ghi VIP = "dedicated instance, higher memory" cho Redis.
     Tier comparison bảng thì ghi "Shared*". Mâu thuẫn → thống nhất: Shared*.


BỔ SUNG (v2 thiếu, v3 thêm):

  ⑥ L2 Gateway: thêm Route Matcher
     v2 có 4 middleware. Thực tế 5: Load Balancer → Tenant Resolver →
     Auth Guard → Rate Limiter → Route Matcher (URL → plugin + method + perm).

  ⑦ L3: thêm Context Building section
     v2 không mô tả Tenant Context Manager + Execution Context Builder.
     Đây là thành phần quan trọng, đã có deep dive riêng.

  ⑧ L4 Query Interceptor: thêm 4 sub-components
     v2 chỉ ghi "auto tenant_id". Thực tế: TenantInjector, QueryValidator,
     QueryCounter (sandbox limit), QueryLogger (slow query alert).

  ⑨ L4 Pool Manager: thêm TenantQuotaEnforcer
     v2 không mô tả per-tenant connection cap (20 conns max trong shared pool).
     Quan trọng để chống 1 tenant starvation.

  ⑩ L4 Migration Engine: thêm Dependency enforcement
     Cannot disable customer-data nếu customer-care vẫn enabled.
     v2 không đề cập → thêm vào L6 Cross-cutting.

  ⑪ L5 Redis: mở rộng từ 3 vai trò → 6 vai trò
     v2 ghi "L2 cache, Pub/Sub, Sessions". Thực tế: L2 cache, domain map,
     rate limit (sorted set), token blacklist, Pub/Sub, realtime counters.

  ⑫ L5 RabbitMQ: thêm exchange topology
     v2 chỉ ghi "Async events, DLQ". Thực tế: crm.events (topic exchange) +
     crm.system (fanout), routing key patterns, VIP priority queues,
     3-retry + exponential backoff → DLQ flow.

  ⑬ L6: thêm Dependency Management
     Plugin dependency resolution + disable chain enforcement.
     Xuyên tầng (L3 Plugin Manager + L4 Migration Engine) → thuộc L6.

  ⑭ L7: mở rộng từ 4 → 6 components
     v2 có: Logging, Tracing, Metrics+Alerting (gộp).
     v3 tách: Logging, Tracing, Metrics, Alerting, Health Checks, Audit Trail.
     Audit Trail trước nằm ở L6 → chuyển về L7 (observability concern).
     Health Checks v2 không đề cập → thêm (Liveness, Readiness, Dependency).

  ⑮ Database Topology: thêm schema_migrations + tenant_plugin_migrations
     2 bảng tracking migration mới, phản ánh dual migration design.

  ⑯ Tenant Context: thêm dbPool field
     Field quyết định Pool Manager route tới Shared hay VIP DB.
     v2 Tenant Context code sample không có field này.

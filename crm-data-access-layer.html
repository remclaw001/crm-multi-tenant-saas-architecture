Layer 4 — Data Access

Data Access Layer

4 thực thể cốt lõi: Query Interceptor, Pool Manager, Cache Manager, Migration Engine — cấu trúc nội tại, liên kết xuyên tầng, và hành vi trong mỗi request.

00

Bản đồ tổng quan L4 — 4 thực thể và vị trí

Double Safety — Interceptor + RLS

double safety
TẠI SAO CẦN 2 TẦNG BẢO VỆ?

  Query Interceptor (L4) và PostgreSQL RLS (L5) làm CÙNG VIỆC
  — nhưng ở 2 tầng khác nhau. Đây là defense in depth:


  Tầng 1: Query Interceptor (application level)

    Plugin: ctx.db.query('sales').where({ status: 'new' })

    Interceptor biến thành:
      SELECT * FROM sales
      WHERE status = 'new'
      AND tenant_id = 'acme-123'

    ✓ Chặn ở tầng ứng dụng TRƯỚC KHI gửi tới DB.
    ✓ Nhanh — không cần DB round-trip để reject.
    ✗ Nếu Interceptor có bug → leak.


  Tầng 2: PostgreSQL RLS (database level)

    Connection đã SET: app.current_tenant = 'acme-123'
    RLS policy: USING (tenant_id = current_setting('app.tenant_id'))

    Kể cả query vượt qua Interceptor mà thiếu tenant_id filter:
      SELECT * FROM sales WHERE status = 'new'
      → RLS THÊM: AND tenant_id = 'acme-123'
      → Kết quả vẫn đúng, vẫn chỉ data ACME.

    ✓ Backup nếu Interceptor fail.
    ✓ Database-level — không thể bypass từ application.
    ✗ Chậm hơn (DB must evaluate policy mỗi row).


  SCENARIO: Interceptor có bug

  ┌────────────────────────────────────────────────────────────────────┐
  │                                                                    │
  │  Giả sử bug: Interceptor QUÊN inject tenant_id cho 1 edge case.  │
  │                                                                    │
  │  Query gửi tới PostgreSQL:                                         │
  │    SELECT * FROM sales WHERE status = 'new'                        │
  │    (thiếu AND tenant_id = ...)                                     │
  │                                                                    │
  │  Không có RLS:                                                    │
  │    → Trả về sales CỦA MỌI TENANT. Data leak nghiêm trọng.       │
  │                                                                    │
  │  Có RLS:                                                          │
  │    → PostgreSQL apply policy: tenant_id = 'acme-123'              │
  │    → Chỉ trả sales CỦA ACME. An toàn.                            │
  │                                                                    │
  │  RLS là lớp bảo vệ cuối cùng — "safety net".                     │
  │                                                                    │
  └────────────────────────────────────────────────────────────────────┘


  SCENARIO: RLS có bug (policy sai)

  ┌────────────────────────────────────────────────────────────────────┐
  │                                                                    │
  │  Giả sử RLS policy sai config trên 1 table.                      │
  │                                                                    │
  │  Interceptor vẫn inject tenant_id vào mọi query.                  │
  │  → Query đã có WHERE tenant_id = 'acme-123'                       │
  │  → RLS bug không ảnh hưởng vì filter đã đúng.                    │
  │                                                                    │
  │  Interceptor là lớp bảo vệ đầu tiên — "first wall".              │
  │                                                                    │
  └────────────────────────────────────────────────────────────────────┘

  Kết luận: cần CẢ HAI bị compromised cùng lúc mới leak data.
  Xác suất: rất thấp → defense in depth hiệu quả.

│  Plugin gọi: ctx.cache.get("customers:list:active")                       │
│  Thực tế key: "t:acme-123:customers:list:active"  (auto-prefix)          │
│                                                                            │
│                                                                            │
│  ┌─ 3-Level Cache Architecture ─────────────────────────────────────────┐ │
│  │                                                                      │ │
│  │                                                                      │ │
│  │  L1: IN-MEMORY (per app instance)                                    │ │
│  │  │                                                                   │ │
│  │  │  Storage:     process memory (Map / LRU)                          │ │
│  │  │  Max size:    100 MB per instance                                 │ │
│  │  │  TTL:         60 seconds                                          │ │
│  │  │  Eviction:    LRU (Least Recently Used)                           │ │
│  │  │  Latency:     ~0.01ms  (memory read)                             │ │
│  │  │                                                                   │ │
│  │  │  Nhanh nhất nhưng nhỏ nhất.                                    │ │
│  │  │  Mỗi instance có L1 RIÊNG — không share.                       │ │
│  │  │  Instance 1 cache ≠ Instance 2 cache.                           │ │
│  │  │                                                                   │ │
│  │  │                     MISS                                         │ │
│  │  │                       │                                           │ │
│  │  │                       ▼                                           │ │
│  │                                                                      │ │
│  │  L2: REDIS CLUSTER (shared across instances)                          │ │
│  │  │                                                                   │ │
│  │  │  Storage:     Redis cluster (3 nodes + Sentinel)                  │ │
│  │  │  Max size:    per policy (no hard limit)                          │ │
│  │  │  TTL:         5–15 minutes (configurable per data type)           │ │
│  │  │  Compression: LZ4 for values > 1KB                               │ │
│  │  │  Latency:     ~1-2ms  (network + Redis)                          │ │
│  │  │                                                                   │ │
│  │  │  Shared — tất cả instances đọc/ghi cùng Redis.                 │ │
│  │  │  Instance 1 ghi → Instance 2 đọc được ngay.                    │ │
│  │  │                                                                   │ │
│  │  │  Key format: t:{tenantId}:{plugin}:{entity}:{hash}               │ │
│  │  │  Example:    t:acme-123:care:sales:status=new:page=1              │ │
│  │  │                                                                   │ │
│  │  │                     MISS                                         │ │
│  │  │                       │                                           │ │
│  │  │                       ▼                                           │ │
│  │                                                                      │ │
│  │  L3: DATABASE (source of truth)                                        │ │
│  │  │                                                                   │ │
│  │  │  PostgreSQL (via Query Interceptor → Pool Manager)                │ │
│  │  │  Latency:     ~5-20ms  (query execution + network)                │ │
│  │  │                                                                   │ │
│  │  │  Luôn đúng, luôn mới nhất.                                       │ │
│  │  │  Nhưng chậm nhất → nên cache ở L1/L2 khi có thể.                │ │
│  │  │                                                                   │ │
│  └──────────────────────────────────────────────────────────────────────┘ │
│                                                                            │
│                                                                            │
│  ┌─ Luồng READ ────────────────────────────────────────────────────────┐ │
│  │                                                                      │ │
│  │  ctx.cache.get("customers:list:active")                              │ │
│  │       │                                                              │ │
│  │       ▼                                                              │ │
│  │  L1 memory: t:acme-123:customers:list:active                        │ │
│  │       │                                                              │ │
│  │  ┌────┴──────┐                                                       │ │
│  │  │ L1 HIT   │→ return value. Done. (~0.01ms)                      │ │
│  │  │ L1 MISS  │→ query L2                                             │ │
│  │  └───────────┘                                                       │ │
│  │       │                                                              │ │
│  │       ▼                                                              │ │
│  │  L2 Redis: GET t:acme-123:customers:list:active                     │ │
│  │       │                                                              │ │
│  │  ┌────┴──────┐                                                       │ │
│  │  │ L2 HIT   │→ decompress → store in L1 → return. (~2ms)           │ │
│  │  │ L2 MISS  │→ query L3 (database)                                  │ │
│  │  └───────────┘                                                       │ │
│  │       │                                                              │ │
│  │       ▼                                                              │ │

│  │       │                                                              │ │
│  │       ▼                                                              │ │
│  │  Result → compress → store in L2 Redis (TTL 5min)                   │ │
│  │        → store in L1 memory (TTL 60s)                               │ │
│  │        → return to plugin                                (~15ms)    │ │
│  │                                                                      │ │
│  └──────────────────────────────────────────────────────────────────────┘ │
│                                                                            │
│                                                                            │

│  │                                                                      │ │

Migration Engine — Schema Versioning

Khác 3 thực thể kia: Migration Engine KHÔNG tham gia xử lý request. Nó chỉ chạy khi: plugin được enable/disable cho tenant, hoặc plugin version được upgrade. Đây là admin operation, không phải request-time operation.
migration engine anatomy
┌─ MIGRATION ENGINE ──────────────────────────────────────────────────────────┐
│                                                                              │
│  Trách nhiệm:                                                             │
│  Quản lý schema evolution per plugin, per tenant.                            │
│  Mỗi plugin có migration files riêng.                                        │
│  Mỗi tenant chạy migrations khi enable plugin.                               │
│                                                                              │
│                                                                              │
│  ┌─ Thành phần bên trong ──────────────────────────────────────────────┐   │
│  │                                                                      │   │
│  │  MigrationRegistry                                                 │   │
│  │  │                                                                   │   │
│  │  │  Theo dõi: plugin nào đã chạy migration nào cho tenant nào.      │   │
│  │  │                                                                   │   │
│  │  │  Table: plugin_migrations (trong Metadata DB)                     │   │
│  │  │  ┌──────────────┬────────────────┬──────────┬──────────────┐     │   │
│  │  │  │ tenant_id    │ plugin_name    │ version  │ executed_at  │     │   │
│  │  │  ├──────────────┼────────────────┼──────────┼──────────────┤     │   │
│  │  │  │ acme-123     │ customer-data  │ 001      │ 2025-06-15   │     │   │
│  │  │  │ acme-123     │ customer-data  │ 002      │ 2025-06-15   │     │   │
│  │  │  │ acme-123     │ customer-data  │ 003      │ 2025-06-15   │     │   │
│  │  │  │ acme-123     │ customer-care  │ 001      │ 2025-07-01   │     │   │
│  │  │  │ acme-123     │ customer-care  │ 002      │ 2025-07-01   │     │   │
│  │  │  │ globex-456   │ customer-data  │ 001      │ 2025-08-10   │     │   │
│  │  │  │ globex-456   │ customer-data  │ 002      │ 2025-08-10   │     │   │
│  │  │  │ globex CHƯA enable care → không có care migrations     │     │   │
│  │  │  └──────────────┴────────────────┴──────────┴──────────────┘     │   │
│  │                                                                      │   │
│  │                                                                      │   │
│  │  MigrationExecutor                                                  │   │
│  │  │                                                                   │   │
│  │  │  Chạy migration trong TRANSACTION:                                │   │
│  │  │                                                                   │   │
│  │  │  ┌────────────────────────────────────────────────────────────┐  │   │
│  │  │  │  BEGIN TRANSACTION                                         │  │   │
│  │  │  │                                                            │  │   │
│  │  │  │  -- Acquire lock (1 tenant, 1 plugin tại 1 thời điểm)     │  │   │
│  │  │  │  SELECT pg_advisory_lock(hash('acme-123:customer-care'))   │  │   │
│  │  │  │                                                            │  │   │
│  │  │  │  -- Check: đã chạy tới version nào?                       │  │   │
│  │  │  │  SELECT MAX(version) FROM plugin_migrations               │  │   │
│  │  │  │  WHERE tenant_id = 'acme-123' AND plugin = 'customer-care' │  │   │
│  │  │  │  → result: NULL (chưa chạy)                               │  │   │
│  │  │  │                                                            │  │   │
│  │  │  │  -- Chạy migrations chưa executed:                        │  │   │
│  │  │  │  RUN 001_create_sales.sql                                 │  │   │
│  │  │  │  RUN 002_create_call_logs.sql                              │  │   │
│  │  │  │  RUN 003_create_tickets.sql                                │  │   │
│  │  │  │  RUN 004_add_indexes.sql                                   │  │   │
│  │  │  │                                                            │  │   │
│  │  │  │  -- Record migrations executed                             │  │   │
│  │  │  │  INSERT INTO plugin_migrations (tenant, plugin, version)   │  │   │
│  │  │  │  VALUES ('acme-123', 'customer-care', '001'), ...          │  │   │
│  │  │  │                                                            │  │   │
│  │  │  │  -- Apply RLS policies to new tables                       │  │   │
│  │  │  │  ALTER TABLE sales ENABLE ROW LEVEL SECURITY               │  │   │
│  │  │  │  CREATE POLICY tenant_iso ON sales USING (...)             │  │   │
│  │  │  │                                                            │  │   │
│  │  │  │  COMMIT                                                    │  │   │
│  │  │  │  (hoặc ROLLBACK nếu BẤT KỲ step nào fail)                │  │   │
│  │  │  └────────────────────────────────────────────────────────────┘  │   │
│  │  │                                                                   │   │
│  │  │  Fail → ROLLBACK toàn bộ. Không partial state.                    │   │
│  │  │  Plugin KHÔNG được enable. Admin thấy error message.               │   │
│  │                                                                      │   │
│  │                                                                      │   │
│  │  RollbackExecutor                                                   │   │
│  │  │                                                                   │   │
│  │  │  Khi tenant disable plugin:                                       │   │
│  │  │                                                                   │   │
│  │  │  Option A: Soft disable (default)                                  │   │
│  │  │    → Giữ nguyên tables + data                                     │   │
│  │  │    → Plugin routes bị remove → user không truy cập được           │   │
│  │  │    → Data vẫn ở đó nếu re-enable sau                             │   │
│  │  │                                                                   │   │
│  │  │  Option B: Hard disable (explicit request)                          │   │
│  │  │    → RUN DOWN migrations (reverse order)                          │   │
│  │  │    → 004_down: DROP INDEXES                                       │   │
│  │  │    → 003_down: DROP TABLE tickets                                 │   │
│  │  │    → 002_down: DROP TABLE call_logs                               │   │
│  │  │    → 001_down: DROP TABLE sales                                   │   │
│  │  │    → Remove plugin_migrations records                             │   │
│  │  │    → DATA PERMANENTLY DELETED                                    │   │
│  │  │                                                                   │   │
│  │  │  Configurable per tenant. VIP có thể chọn.                      │   │
│  │                                                                      │   │
│  └──────────────────────────────────────────────────────────────────────┘   │
│                                                                              │
│                                                                              │
│  ĐẶC BIỆT: Migration chạy trên DB NÀO?                                   │
│                                                                              │
│  ┌────────────────────────────────────────────────────────────────────────┐ │
│  │                                                                        │ │
│  │  Standard tenant (ACME, Premium):                                      │ │
│  │  → Migrations chạy trên Shared Data DB                                │ │
│  │  → Tables tạo ra nằm cạnh tables của tenants khác                     │ │

│  │                                                                        │ │

cross-layer relationships
L4 NHẬN INPUT TỪ
═══════════════════════════════════════════════════════════════════════

L2 API Gateway  →  L4
│
├── Tenant Resolver cung cấp: tenantId
│   → Pool Manager dùng để route connection
│   → Query Interceptor dùng để inject tenant_id
│
└── Auth Guard cung cấp: userId, roles
    → Query Logger dùng cho audit context


L3 Business Logic  →  L4
│
├── Plugin Cores gọi: ctx.db.query(), ctx.db.insert(), ctx.cache.get()
│   → Query Interceptor xử lý mọi DB operation
│   → Cache Manager xử lý mọi cache operation
│
├── Execution Context Builder cung cấp: tenantId, dbPool, cachePrefix
│   → Pool Manager dùng dbPool để chọn pool
│   → Cache Manager dùng cachePrefix cho key generation
│
├── Sandbox Engine cung cấp: maxQueries limit
│   → Query Counter enforce giới hạn
│
├── Plugin Manager gọi: Migration Engine
│   → Khi enable/disable plugin cho tenant
│   → Khi upgrade plugin version
│
└── Tenant Context Manager gọi: Pool Manager
    → Acquire metadata DB connection
    → Query tenant_plugins table


L6 Cross-cutting  →  L4
│
├── Config Manager cung cấp: pool sizes, TTLs, health check intervals
│   → Pool Manager dùng để configure pools
│   → Cache Manager dùng để set TTLs
│
└── Error Handling Framework cung cấp: Circuit Breaker
    → Pool Manager dùng: pool exhausted → circuit open → 503



L4 CUNG CẤP CHO
═══════════════════════════════════════════════════════════════════════

L4  →  L3 Business Logic
│
├── Query Interceptor → TenantScopedDB interface
│   Cho plugin: ctx.db (safe, tenant-scoped DB access)
│
├── Cache Manager → TenantScopedCache interface
│   Cho plugin: ctx.cache (auto-prefixed cache access)
│
└── Migration Engine → schema ready confirmation
    Cho Plugin Manager: plugin tables exist → safe to enable


L4  →  L5 Infrastructure
│
├── Pool Manager → managed connections TO:
│   ├── Metadata DB PostgreSQL
│   ├── Shared Data DB PostgreSQL
│   └── VIP Data DB PostgreSQL(s)
│
└── Cache Manager → managed keys TO:
    └── Redis Cluster


L4  →  L7 Observability
│
├── Query Interceptor emits:
│   ├── Metrics: query_duration, query_count, slow_query_count
│   └── Traces: span per DB query (table, operation, duration)
│
├── Pool Manager emits:
│   ├── Metrics: pool_active, pool_idle, pool_waiting, pool_exhausted
│   └── Alerts: pool > 80% → WARNING, pool = 100% → CRITICAL
│
└── Cache Manager emits:
    ├── Metrics: cache_hit_rate, cache_miss_rate per level
    └── Traces: span per cache operation

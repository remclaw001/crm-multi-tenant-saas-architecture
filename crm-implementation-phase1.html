Vấn đề: Table owner bypass RLS
PostgreSQL default behavior Theo mặc định, table owner không bị RLS chặn. Nếu chỉ dùng ENABLE ROW LEVEL SECURITY, user crm (owner của tất cả tables) vẫn thấy toàn bộ data dù không set app.tenant_id. Đây là lỗ hổng nghiêm trọng trong môi trường dev — dev không nhận ra và tưởng RLS hoạt động.
Giải pháp: FORCE ROW LEVEL SECURITY
migration — RLS setup sql 
-- ENABLE: bật RLS, nhưng owner vẫn bypass
ALTER TABLE users ENABLE ROW LEVEL SECURITY;

-- FORCE: áp dụng cả với owner → query "trần" trả về 0 rows
ALTER TABLE users FORCE ROW LEVEL SECURITY;

-- Tương tự cho roles và user_roles
ALTER TABLE roles ENABLE ROW LEVEL SECURITY;
ALTER TABLE roles FORCE ROW LEVEL SECURITY;

ALTER TABLE user_roles ENABLE ROW LEVEL SECURITY;
ALTER TABLE user_roles FORCE ROW LEVEL SECURITY;
Policy: USING vs WITH CHECK
migration — RLS policy sql 
CREATE POLICY tenant_isolation ON users
  USING      (tenant_id = current_setting('app.tenant_id', true)::uuid)  SELECT UPDATE DELETE
  WITH CHECK (tenant_id = current_setting('app.tenant_id', true)::uuid);  INSERT UPDATE
ClauseÁp dụng choHành vi
USING SELECT · UPDATE · DELETE Filter rows — chỉ trả về/xử lý row có tenant_id khớp session var
WITH CHECK INSERT · UPDATE Validate row mới — không cho INSERT row có tenant_id không khớp session var
Nếu thiếu WITH CHECK Có thể INSERT {tenant_id: 'tenant-b'} trong khi session đang set app.tenant_id = 'tenant-a'. Row sẽ được tạo thành công nhưng sau đó vô hình với cả hai tenant — data corruption khó phát hiện.

src/config/env.ts — Zod validation
File này là single source of truth cho toàn bộ environment configuration. Mọi service trong ứng dụng import config từ đây thay vì đọc process.env trực tiếp — đảm bảo type safety và validate một lần duy nhất lúc startup.
backend/src/config/env.ts typescript 
import 'dotenv/config';        // load .env trước mọi thứ khác
import { z } from 'zod';

const envSchema = z.object({
  NODE_ENV: z.enum(['development', 'test', 'production']).default('development'),

  // Shared pool — tất cả standard tenants
  DATABASE_URL:            z.string().min(1, 'DATABASE_URL is required'),
  DATABASE_POOL_MAX:        z.coerce.number().int().positive().default(200),

  // Metadata pool — migrations, tenant lookup (optional, fallback = DATABASE_URL)
  DATABASE_METADATA_URL:    z.string().min(1).optional(),
  DATABASE_METADATA_POOL_MAX: z.coerce.number().int().positive().default(20),

  REDIS_URL:    z.string().min(1, 'REDIS_URL is required'),
  RABBITMQ_URL: z.string().min(1, 'RABBITMQ_URL is required'),
  PORT:         z.coerce.number().int().positive().default(3000),
});

// safeParse thay vì parse() → kiểm soát được output error
const result = envSchema.safeParse(process.env);

if (!result.success) {
  console.error('\n❌  Invalid / missing environment variables:\n');
  const formatted = result.error.format();
  for (const [key, val] of Object.entries(formatted)) {
    if (key === '_errors') continue;
    const errors = (val as { _errors: string[] })._errors;
    if (errors.length) console.error(`  ${key}: ${errors.join(', ')}`);
  }
  console.error('\nCopy .env.example to .env and fill in all required values.\n');
  process.exit(1);                // crash ngay, không để app start ở trạng thái thiếu config
}

export const config = result.data;
export type Config = typeof config;
z.coerce.number() — tại sao? process.env luôn là string. z.number() sẽ fail cho "200". z.coerce.number() tự động convert "200" → 200 trước khi validate. Không cần parseInt(process.env.DATABASE_POOL_MAX) rải khắp code.
knexfile.ts — afterCreate pool hook
Phần quan trọng nhất không phải là config connection mà là hook afterCreate — giải quyết vấn đề context leak giữa các request:
backend/knexfile.ts typescript 
import 'dotenv/config';          // phải đứng đầu — CommonJS require() chạy theo thứ tự
import type { Knex } from 'knex';
import { config } from './src/config/env';

const knexConfig: Knex.Config = {
  client: 'postgresql',
  connection: config.DATABASE_URL,
  pool: {
    min: 2,
    max: config.DATABASE_POOL_MAX,

    // Reset tenant context mỗi khi connection được trả về pool
    afterCreate(conn, done) {
      conn.query("SELECT set_config('app.tenant_id', NULL, false)", (err) => {
        done(err, conn);
      });
    },
  },

  migrations: {
    directory: './src/db/migrations',
    extension: 'ts',
    tableName: 'knex_migrations',
  },
  seeds: {
    directory: './src/db/seeds',
    extension: 'ts',
  },
};

export default knexConfig;
Context leak — nếu không có afterCreate Request A (tenant: acme) → lấy connection X → SET app.tenant_id = 'acme' → xử lý xong → trả connection X về pool.
Request B (tenant: globex) → lấy đúng connection X từ pool → app.tenant_id vẫn là 'acme' → RLS filter nhầm → data leak.
set_config — tham số thứ 3 set_config('app.tenant_id', NULL, false) — false nghĩa là session-level (tồn tại qua nhiều transaction). true sẽ là transaction-local (bị reset sau mỗi commit). Dùng false ở đây để reset sạch toàn bộ session state của connection trước khi trả về pool.

Vấn đề: Seed với FORCE RLS bật
Nếu INSERT vào users mà không set app.tenant_id WITH CHECK policy sẽ fail vì tenant_id != NULL. Seed script cần một cách bypass RLS hoặc set context đúng trước khi insert.

Giải pháp 2: SET LOCAL trong transaction (insert)
seeds/01_tenants.ts — SET LOCAL pattern typescript 
await knex.transaction(async (trx) => {
  // 1. Insert tenants — không cần SET (tenants không có RLS)
  const [acme, globex, initech] = await trx('tenants').insert([...]).returning('*');

  // 2. Seed từng tenant
  for (const tenant of [acme, globex, initech]) {
    // SET LOCAL: chỉ có hiệu lực trong transaction này
    // Khi transaction commit/rollback → setting tự động hủy
    await trx.raw(`SET LOCAL "app.tenant_id" = ?`, [tenant.id]);

    // RLS WITH CHECK pass vì tenant_id insert = app.tenant_id session var
    const [adminRole] = await trx('roles').insert([
      { tenant_id: tenant.id, name: 'admin' },
    ]).returning('*');

    await trx('users').insert([
      { tenant_id: tenant.id, email: `admin@${tenant.subdomain}.example.com`, ... },
    ]);
  }
});
SET LOCAL vs SET SET — session-level, tồn tại sau khi transaction commit. Nguy hiểm nếu dùng trong seed vì setting sẽ "rò" ra các query sau.
SET LOCAL — transaction-level, tự reset khi transaction kết thúc. An toàn và đúng cách để inject context tạm thời.

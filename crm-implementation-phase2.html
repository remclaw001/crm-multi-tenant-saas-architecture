Tại sao AsyncLocalStorage an toàn với async/await AsyncLocalStorage dùng Node.js execution context — mỗi async chain có context riêng. Khác với global variable, context của Promise chain A không bị ghi đè bởi chain B dù chúng chạy đồng thời trên cùng một thread. Node.js runtime tự động propagate context qua await, setTimeout, EventEmitter, và stream callbacks.
TenantStore structure
src/dal/context/TenantContext.ts — TenantStore interface typescript 
export type TenantTier = 'standard' | 'vip' | 'enterprise';

export interface TenantStore {
  readonly tenantId: string;          // UUID — immutable trong suốt request
  readonly tenantTier: TenantTier;    // standard | vip | enterprise
  queryCount: number;                  // mutable — tăng mỗi lần query
}

// Implementation — static class wrapping AsyncLocalStorage
const storage = new AsyncLocalStorage<TenantStore>();

export class TenantContext {
  static run<T>(store: Omit<TenantStore, 'queryCount'>, fn: () => T): T {
    return storage.run({ ...store, queryCount: 0 }, fn);  // queryCount bắt đầu từ 0
  }

  static getTenantId(): string | undefined {
    return storage.getStore()?.tenantId;
  }

  static requireTenantId(): string {
    const id = storage.getStore()?.tenantId;
    if (!id) throw new Error('TenantContext.requireTenantId() called outside of TenantContext.run()');
    return id;
  }

  static incrementQueryCount(): number {
    const store = storage.getStore();
    if (store) store.queryCount++;
    return store?.queryCount ?? 0;
  }
}

Vấn đề: afterCreate không đủ
afterCreate chỉ chạy khi connection được TẠO MỚI afterCreate của Knex (Phase 1) reset app.tenant_id = NULL chỉ khi connection lần đầu được tạo ra từ pool. Khi connection được trả về pool và lấy ra lần thứ hai, afterCreate KHÔNG chạy lại.
Connection X tạo mới → afterCreate() reset app.tenant_id = NULL ✓ Request A (tenant: acme) → lấy connection X từ pool → SET app.tenant_id = 'acme' → query → xong Connection X trả về pool → afterCreate() KHÔNG chạy lại (chỉ chạy khi tạo mới!) ✗ Request B (tenant: globex) → lấy đúng connection X → app.tenant_id vẫn là 'acme' → DATA LEAK!
Giải pháp: Wrap release()
Thay vì dựa vào afterCreate, PoolRegistry wrap hàm release() của mỗi client — đảm bảo reset TRƯỚC KHI trả connection về pool, bất kể connection đó được lấy ra lần thứ bao nhiêu.
src/dal/pool/PoolRegistry.ts — acquire pattern typescript 
async acquireConnection(tenantId: string): Promise<PoolClient> {
  assertValidUuid(tenantId);                                // SQL injection guard
  const pool = this.selectPool(tenantId);
  const client = await pool.connect();

  // Bọc release() để luôn reset TRƯỚC KHI trả về pool
  const _release = client.release.bind(client);
  (client as any).release = async (err?: Error | boolean) => {
    try {
      await client.query(`SET "app.tenant_id" = ''`);  // reset TRƯỚC KHI release
    } catch { /* connection broken — pool sẽ discard, không cần làm gì */ }
    _release(err);                                          // trả về pool
  };

  // Set tenant context cho connection này
  await client.query(`SET "app.tenant_id" = '${tenantId}'`);

  return client;
}
UUID validation — ngăn SQL injection
src/dal/pool/PoolRegistry.ts — assertValidUuid typescript 
// SET command không hỗ trợ parameterized query ($1)
// → phải interpolate trực tiếp vào string
// → PHẢI validate format trước

function assertValidUuid(id: string): void {
  if (!/^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i.test(id)) {
    throw new Error(`PoolRegistry: invalid tenant_id format: "${id}"`);
  }
}
// UUID chỉ chứa hex chars và dashes — không có SQL injection vector
PostgreSQL SET command — không có parameterized query SET "app.tenant_id" = $1 là cú pháp không hợp lệ trong PostgreSQL. SET chỉ nhận literal values, không nhận placeholders. Regex validation đảm bảo chỉ UUID hợp lệ ([0-9a-f]{8}-...) được interpolate — UUID chỉ gồm hex characters và dashes, không có ký tự SQL đặc biệt nào.
Pool selection logic
src/dal/pool/PoolRegistry.ts — selectPool typescript 
private selectPool(tenantId: string): Pool {
  // VIP tenant → dedicated pool nếu đã đăng ký
  const vipPool = this.vipPools.get(tenantId);
  if (vipPool) return vipPool;           // dedicated pool — không dùng shared

  // Standard tenant → shared pool (200 connections)
  return this.sharedPool;
}

// Metadata pool được truy cập riêng — không qua acquireConnection
getMetadataPool(): Pool { return this.metadataPool; }

// VIP tenant registration — gọi khi tenant config load
registerVipPool(tenantId: string, connectionString: string): void {
  if (this.vipPools.has(tenantId)) return;   // idempotent — không tạo lại nếu đã có
  const pool = new Pool({
    connectionString,
    max: 30,                               // VIP hard limit
  });
  this.vipPools.set(tenantId, pool);
}
Tại sao monkey-patch acquireConnection?
Knex không expose một "before query" hook sạch ở public API. acquireConnection là điểm duy nhất có thể intercept: xảy ra sau khi connection được lấy từ pool, trước khi bất kỳ query nào chạy trên connection đó.
Knex.query(builder) └── client.acquireConnection() ← ĐÂY là điểm hook duy nhất đúng thời điểm └── pool.connect() └── pg connection
Hai lớp bảo vệ — defense in depth
Lớp 1: QueryInterceptor (Knex level) acquireConnection → SET app.tenant_id releaseConnection → SET app.tenant_id = '' Lớp 2: PoolRegistry (pg level) acquireConnection → SET app.tenant_id release() wrapper → SET app.tenant_id = '' Lớp 3: afterCreate (pool level, Phase 1) mỗi khi connection được TẠO MỚI → set_config(app.tenant_id, NULL)
Belt AND suspenders — defense-in-depth Nếu một lớp bị lỗi hoặc bị bypass, lớp kia vẫn bảo vệ. Ví dụ: nếu code gọi thẳng pg.Pool.connect() bỏ qua Knex, PoolRegistry vẫn reset. Nếu code bỏ qua PoolRegistry và dùng Knex trực tiếp, QueryInterceptor vẫn bảo vệ.
Implementation — monkey-patch pattern
src/dal/interceptor/QueryInterceptor.ts typescript 
export function applyQueryInterceptor(knexInstance: Knex): void {
  // Lưu các method gốc
  const originalAcquire = knexInstance.client.acquireConnection.bind(knexInstance.client);
  const originalRelease = knexInstance.client.releaseConnection.bind(knexInstance.client);

  // Override acquireConnection — chạy TRƯỚC mọi query
  knexInstance.client.acquireConnection = async () => {
    const conn = await originalAcquire();

    const tenantId = TenantContext.getTenantId();
    if (tenantId) {
      await conn.query(`SET "app.tenant_id" = '${tenantId}'`);  // scope connection
      QueryCounter.increment(false);                         // đếm, không throw
    }

    return conn;
  };

  // Override releaseConnection — chạy SAU khi query hoàn tất
  knexInstance.client.releaseConnection = async (conn: any) => {
    try {
      await conn.query(`SET "app.tenant_id" = ''`);  // reset context
    } catch { /* connection broken — ignore */ }
    return originalRelease(conn);
  };
}
Tích hợp QueryCounter
acquireConnection cũng gọi QueryCounter.increment(false) — tracking số lượng query mà không throw. Phase 6 Sandbox sẽ gọi increment(true) để enforce hard limit 50 queries/request.
SET vs SET LOCAL
SETSET LOCAL
Scope Session (toàn bộ connection) Transaction (chỉ trong BEGIN...COMMIT)
Dùng trong Request lifecycle (nhiều query) Seed script (single transaction)
Reset khi Manual, hoặc khi connection đóng Transaction kết thúc (tự động)
Tại sao dùng SET (session) ở interceptor Một request có thể thực hiện nhiều query trong nhiều transaction nhỏ — ví dụ: đọc user (transaction 1) → ghi audit log (transaction 2) → update cache (không có transaction). SET LOCAL sẽ bị reset sau mỗi transaction, khiến RLS mất hiệu lực từ query thứ hai trở đi.

Key pattern — tenant isolation qua prefix
t : <tenantId> : <resourceType> : <id> │ │ │ │ │ UUID tenant 'customer' resource ID │ 'deal' │ 'tenant-config' └── prefix cố định — dễ scan, dễ invalidate theo tenant
key pattern examples text

t:cccc-dddd:customer:cust-001        → customer của tenant cccc-dddd (khác key!)

CacheManager — auto key construction
src/dal/cache/CacheManager.ts typescript 
export class CacheManager implements ICacheManager {
  constructor(private redis: Redis) {}

  // Key pattern enforcement — developer không thể tạo key sai format
  private buildKey(resource: string, id: string): string {
    const tenantId = TenantContext.requireTenantId();  // throw nếu ngoài context
    return `t:${tenantId}:${resource}:${id}`;
  }

  async get<T>(resource: string, id: string): Promise<T | null> {
    const raw = await this.redis.getBuffer(this.buildKey(resource, id));
    if (!raw) return null;
    return msgpack.decode(raw) as T;                    // MessagePack decode
  }

  async set<T>(resource: string, id: string, value: T, ttlSeconds = 300): Promise<void> {
    const encoded = msgpack.encode(value);                // MessagePack encode
    await this.redis.setex(this.buildKey(resource, id), ttlSeconds, Buffer.from(encoded));
  }
}

SCAN vs KEYS — non-blocking invalidation
src/dal/cache/CacheManager.ts — invalidateByTenant typescript 
// ❌ KEYS — block Redis trong khi scan, O(N) với N = tổng số keys
const keys = await redis.keys(`t:${tenantId}:customer:*`);
// → nếu Redis có 10 triệu keys, lệnh này block toàn bộ server trong vài giây

// ✓ SCAN — cursor-based, không block, trả về từng batch
async invalidateByTenant(tenantId: string, resource: string): Promise<number> {
  const pattern = `t:${tenantId}:${resource}:*`;
  let cursor = '0';
  let total = 0;

  do {
    const [nextCursor, batch] = await redis.scan(cursor, 'MATCH', pattern, 'COUNT', 100);
    cursor = nextCursor;
    if (batch.length) {
      await redis.del(...batch);
      total += batch.length;
    }
  } while (cursor !== '0');  // cursor = '0' → scan hoàn thành

  // Redis tiếp tục phục vụ requests khác trong khi SCAN chạy
  return total;
}
QueryCounter — thiết kế cho Phase 6
src/dal/middleware/QueryCounter.ts typescript 
export class QueryLimitExceededError extends Error {
  constructor(public readonly count: number, public readonly limit: number) {
    super(`Query limit exceeded: ${count}/${limit}`);
    this.name = 'QueryLimitExceededError';
  }
}

export const QUERY_LIMIT = 50;  // Sandbox hard limit (Phase 6)

export class QueryCounter {
  // enforce = false → đếm, không throw (QueryInterceptor, toàn bộ system)
  // enforce = true  → đếm VÀ throw khi > 50 (Sandbox Engine, Phase 6)
  static increment(enforce: boolean = false): number {
    const count = TenantContext.incrementQueryCount();   // per-request, via AsyncLocalStorage
    if (enforce && count > QUERY_LIMIT) {
      throw new QueryLimitExceededError(count, QUERY_LIMIT);   SANDBOX ONLY
    }
    return count;
  }

  static getCount(): number {
    return TenantContext.getQueryCount();
  }
}
Phase 6 Sandbox sẽ gọi QueryCounter.increment(true) Khi plugin chạy trong isolated-vm, mọi DB query đều đi qua context bridge giữa sandbox và host. Đó là điểm Phase 6 sẽ gọi increment(true) — nếu plugin cố tình chạy vòng lặp query, nó sẽ bị dừng sau query thứ 51. Counter sống trong AsyncLocalStorage nên per-request và không cần lock.

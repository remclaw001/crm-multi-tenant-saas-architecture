Security — L2 Gateway

Authentication & Authorization Flow

Toàn bộ luồng xác thực: đăng nhập → JWT → xác thực mỗi request → refresh → revoke → phân quyền RBAC.

Complete authentication flow: login → JWT → per-request verification → refresh → revoke → RBAC authorization.

00

Bản đồ Auth — Hai luồng chínhAuth Map — Two Main Flows

Nguyên tắc cốt lõi: JWT là nguồn sự thật duy nhất về danh tính. Tenant được xác định từ header/body (TenantResolver), sau đó được xác nhận khớp với tenant_id trong JWT (AuthGuard). Hai bước này tách biệt để tránh tenant impersonation.
Core principle: JWT is the single source of truth for identity. The tenant is resolved from the header/body (TenantResolver), then verified to match the tenant_id inside the JWT (AuthGuard). These two steps are separate to prevent tenant impersonation.
auth architecture
  LUỒNG 1: ĐĂNG NHẬP — Lấy token

  Browser/Client
      │  POST /auth/login  { tenantSlug, email, password }
      ▼
  L2 Gateway  ─── bỏ qua AuthGuard (@Public endpoint)
      │
      ▼
  AuthService  (L3 Business Logic)
      ├── Resolve tenant từ tenantSlug (body) → Metadata DB
      ├── Verify password với bcrypt (cost 12)
      ├── Load roles từ user_roles → roles[]
      └── Issue JWT (HS256 dev / RS256 JWKS prod)
      │
      ▼
  Response: { token, user: { id, name, email, roles }, tenant: { id, subdomain, name, tier } }


  LUỒNG 2: GỌI API — Dùng token

  Browser/Client
      │  GET /api/v1/plugins/customer-data/customers
      │  Authorization: Bearer <token>
      │  X-Tenant-ID: acme-123  (hoặc X-Tenant-Slug: acme)
      ▼
  L2 Gateway
      │
      ├── ① TenantResolver:  X-Tenant-ID header → resolvedTenant
      ├── ② AuthGuard:       verify JWT signature → extract claims
      │                       cross-check: jwt.tenant_id == resolvedTenant.id  ← tenant isolation
      ├── ③ RateLimiter:     check per-tenant limit (ThrottlerGuard)
      └── ④ Controller:      check ctx.enabledPlugins.includes(plugin) → 403 nếu disabled
      │
      ▼
  L3 Business Logic (SandboxService → plugin core)


  LUỒNG 3: ADMIN LOGIN — Super admin

  Browser/Client
      │  POST /api/v1/admin/auth/login  { email, password }
      │  Không có tenant header — admin routes bypass TenantResolverMiddleware
      ▼
  AdminAuthService
      ├── Query user trong tenant 'system' (Metadata DB)
      ├── Verify password bcrypt + kiểm tra role 'super_admin'
      └── Issue JWT (bắt buộc dùng JWT_SECRET_FALLBACK / HS256)
      │
      ▼
  Response: { token, user: { id, email, role: "super_admin" } }

Đăng nhập — Login FlowLogin — Login Flow

Endpoint public: POST /auth/login không qua AuthGuard (@Public()). Tenant được resolve từ tenantSlug trong body — AuthService tự query Metadata DB, không phụ thuộc vào TenantResolverMiddleware.
Public endpoint: POST /auth/login bypasses AuthGuard (@Public()). The tenant is resolved from tenantSlug in the request body — AuthService queries the Metadata DB directly, independent of TenantResolverMiddleware.
login flow
POST /auth/login
Body: { tenantSlug: "acme", email: "alice@acme.com", password: "..." }
Không cần tenant header — tenantSlug nằm trong body


BƯỚC 1: Resolve Tenant từ Body  (~2ms)
│
│  SELECT id, subdomain, name, tier FROM tenants
│  WHERE subdomain = 'acme' AND is_active = true
│  → { id: "acme-123", tier: "premium", ... }
│
│  Không tìm thấy / is_active = false → 401 Unauthorized


BƯỚC 2: Activate RLS + Load User & Roles  (~3ms)
│
│  SELECT set_config('app.tenant_id', 'acme-123', true)  ← bật RLS trong session
│
│  SELECT u.id, u.email, u.name, u.password_hash, u.is_active,
│         COALESCE(array_agg(r.name) FILTER (WHERE r.name IS NOT NULL), '{}') AS roles
│  FROM users u
│  LEFT JOIN user_roles ur ON ur.user_id = u.id AND ur.tenant_id = 'acme-123'
│  LEFT JOIN roles r ON r.id = ur.role_id AND r.tenant_id = 'acme-123'
│  WHERE u.email = 'alice@acme.com' AND u.tenant_id = 'acme-123'
│  GROUP BY u.id
│
│  → { id: "user-5", password_hash: "$2b$12$...", is_active: true, roles: ["admin"] }


BƯỚC 3: Verify Password (bcrypt)  (~100ms, CPU-bound)
│
│  Constant-time: luôn chạy bcrypt kể cả khi user không tồn tại (dùng DUMMY_HASH)
│  DUMMY_HASH = '$2b$12$invalidhashusedtoblindtimingXXX...'
│  hashToCompare = user?.password_hash ?? DUMMY_HASH
│
│  valid = await bcrypt.compare(inputPassword, hashToCompare)
│
│  if (!user || !user.is_active || !valid)
│    → 401 { message: "Invalid credentials" }  ← thông báo chung, không tiết lộ lý do


BƯỚC 4: Issue JWT  (~1ms)
│
│  token = JWT.sign({
│    sub:       "user-5",
│    tenant_id: "acme-123",   ← snake_case
│    email:     "alice@acme.com",
│    roles:     ["admin"],
│  }, secret)
│
│  Dev mode:  HS256 + JWT_SECRET_FALLBACK (symmetric)
│  Prod mode: RS256 + JWKS endpoint (Keycloak / asymmetric)


BƯỚC 5: Response
│
│  200 OK
│  {
│    token:  "eyJ...",
│    user:   { id: "user-5", name: "Alice", email: "alice@acme.com", roles: ["admin"] },
│    tenant: { id: "acme-123", subdomain: "acme", name: "Acme Corp", tier: "premium" }
│  }

JWT & Claims — Cấu trúc TokenToken Structure

JWT là source of truth: Mọi thông tin cần thiết cho một request đều nằm trong JWT — không cần query DB để biết userId, tenant_id, roles. AuthGuard chỉ cần verify signature và cross-check tenant.
JWT is the source of truth: All information needed for a request is embedded in the JWT — no DB query is required to know userId, tenant_id, or roles. AuthGuard only needs to verify the signature and cross-check the tenant.
jwt structure
JWT TOKEN — Dev: HS256 / Prod: RS256

  eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9   ← dev (HS256)
  eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9   ← prod (RS256 via JWKS)
  .eyJzdWIiOiJ1c2VyLTUiLCJ0ZW5hbnRfaWQiOiJhY21lLTEyMyIsLi4ufQ
  .<signature>

  ┌─────────────────────────────────────────────────────────────────────┐
  │ HEADER                                                              │
  │   alg: "HS256"  (dev)  ← JWT_SECRET_FALLBACK, symmetric           │
  │   alg: "RS256"  (prod) ← JWKS endpoint (Keycloak), asymmetric     │
  │   typ: "JWT"                                                        │
  └─────────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────────┐
  │ PAYLOAD (claims thực tế được issue)                                 │
  │                                                                     │
  │   sub:       "user-5"          ← userId (standard claim)           │
  │   jti:       "uuid-v4-..."     ← JWT ID: dùng cho blacklist revoke   │
  │   tenant_id: "acme-123"        ← bắt buộc: tenant scope (snake_case) │
  │   email:     "alice@acme.com"  ← email user                        │
  │   roles:     ["admin"]         ← RBAC roles từ user_roles          │
  │                                                                     │
  │   iat:  1708000000   ← issued at (Unix timestamp, auto)            │
  │   exp:  <theo JWT_EXPIRES_IN hoặc default NestJS JwtModule>        │
  └─────────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────────┐
  │ JwtClaims interface (TypeScript — req.user sau khi guard chạy)     │
  │                                                                     │
  │   sub:       string    ← required, validate trong JwtStrategy      │
  │   jti:       string    ← JWT ID (UUID v4), dùng để blacklist       │
  │   tenant_id: string    ← required, cross-check với resolvedTenant  │
  │   roles:     string[]  ← default [] nếu không có                   │
  │   email?:    string    ← optional                                   │
  │   iat?:      number                                                 │
  │   exp?:      number                                                 │
  └─────────────────────────────────────────────────────────────────────┘

Xác thực Request — AuthGuard PipelineVerify Request — AuthGuard Pipeline

Chạy trên mỗi request (trừ @Public): JwtAuthGuard extend Passport AuthGuard('jwt'), chạy sau TenantResolverMiddleware. Nếu bất kỳ bước nào fail, request bị reject ngay, không forward xuống L3.
Runs on every request (except @Public): JwtAuthGuard extends Passport AuthGuard('jwt'), runs after TenantResolverMiddleware. If any step fails, the request is rejected immediately without being forwarded to L3.
authguard pipeline
Mỗi request có Authorization: Bearer <token>


BƯỚC 1: Check @Public()  (~0ms)
│
│  JwtAuthGuard.canActivate() kiểm tra Reflector metadata:
│  isPublic = reflector.getAllAndOverride(IS_PUBLIC_KEY, [handler, class])
│
│  if isPublic → bypass hoàn toàn, không verify JWT
│  Áp dụng cho: POST /auth/login, POST /api/v1/admin/auth/login, GET /health, GET /metrics


BƯỚC 2: Verify Signature (Passport JwtStrategy)  (~0.5ms, CPU)
│
│  ExtractJwt.fromAuthHeaderAsBearerToken() → lấy token từ Authorization header
│
│  Dev mode:  JWT.verify(token, JWT_SECRET_FALLBACK, { algorithms: ['HS256'] })
│  Prod mode: JWT.verify(token, publicKey,           { algorithms: ['RS256'] })
│             publicKey từ JWKS cache (10 phút, tối đa 5 entries)
│
│  Nếu token thiếu hoặc signature không hợp lệ:
│    → 401 { message: "Missing or invalid authentication token" }
│
│  ignoreExpiration: false → Passport tự check exp, reject nếu hết hạn → 401


BƯỚC 3: Validate Claims (JwtStrategy.validate)  (~0ms)
│
│  Kiểm tra required claims:
│  if typeof payload.sub !== 'string' || !payload.sub
│    → 401 { message: "JWT missing required claim: sub" }
│
│  if typeof payload.tenant_id !== 'string' || !payload.tenant_id
│    → 401 { message: "JWT missing required claim: tenant_id" }
│
│  Return: JwtClaims { sub, tenant_id, roles, email?, iat?, exp? }
│          → trở thành req.user


BƯỚC 4: Cross-check Tenant  (~0ms)
│
│  Đây là bước bảo vệ tenant isolation quan trọng nhất (trong handleRequest):
│  resolvedTenant = req.resolvedTenant   ← từ TenantResolverMiddleware
│
│  if resolvedTenant && claims.tenant_id !== resolvedTenant.id
│    → 403 ForbiddenException:
│         "JWT tenant mismatch: token belongs to tenant {claims.tenant_id},
│          but request targets tenant {resolvedTenant.id}"
│
│  Ngăn user của tenant A dùng token hợp lệ để gọi API trên subdomain tenant B.
│  Admin routes không có resolvedTenant → bước này bị skip (if resolvedTenant).


BƯỚC 5: Kiểm tra JTI Blacklist (Redis)  (~0.3ms, I/O)
│
│  Sau khi tenant cross-check pass, kiểm tra token có bị revoke không:
│  key = "auth:blacklist:" + claims.jti
│
│  Redis.EXISTS(key)
│    → exists → 401 UnauthorizedException: "Token has been revoked"
│    → not found → PASS
│
│  Blacklist entry được tạo khi user logout (POST /auth/logout).
│  TTL = remaining lifetime của JWT → entry tự xóa khi token đã hết hạn.
│
│  → PASS → req.user = JwtClaims → tiếp tục vào controller

authguard pipeline
Every request with Authorization: Bearer <token>


STEP 1: Check @Public()  (~0ms)
│
│  JwtAuthGuard.canActivate() checks Reflector metadata:
│  isPublic = reflector.getAllAndOverride(IS_PUBLIC_KEY, [handler, class])
│
│  if isPublic → fully bypass, no JWT verification
│  Applies to: POST /auth/login, POST /api/v1/admin/auth/login, GET /health, GET /metrics


STEP 2: Verify Signature (Passport JwtStrategy)  (~0.5ms, CPU)
│
│  ExtractJwt.fromAuthHeaderAsBearerToken() → extract token from Authorization header
│
│  Dev mode:  JWT.verify(token, JWT_SECRET_FALLBACK, { algorithms: ['HS256'] })
│  Prod mode: JWT.verify(token, publicKey,           { algorithms: ['RS256'] })
│             publicKey from JWKS cache (10 min, max 5 entries)
│
│  If token is missing or signature is invalid:
│    → 401 { message: "Missing or invalid authentication token" }
│
│  ignoreExpiration: false → Passport auto-checks exp, rejects if expired → 401


STEP 3: Validate Claims (JwtStrategy.validate)  (~0ms)
│
│  Check required claims:
│  if typeof payload.sub !== 'string' || !payload.sub
│    → 401 { message: "JWT missing required claim: sub" }
│
│  if typeof payload.tenant_id !== 'string' || !payload.tenant_id
│    → 401 { message: "JWT missing required claim: tenant_id" }
│
│  Return: JwtClaims { sub, tenant_id, roles, email?, iat?, exp? }
│          → becomes req.user


STEP 4: Cross-check Tenant  (~0ms)
│
│  This is the most critical tenant isolation protection step (inside handleRequest):
│  resolvedTenant = req.resolvedTenant   ← from TenantResolverMiddleware
│
│  if resolvedTenant && claims.tenant_id !== resolvedTenant.id
│    → 403 ForbiddenException:
│         "JWT tenant mismatch: token belongs to tenant {claims.tenant_id},
│          but request targets tenant {resolvedTenant.id}"
│
│  Prevents tenant A's user from using a valid token to call the API on tenant B's subdomain.
│  Admin routes have no resolvedTenant → this step is skipped (if resolvedTenant).


STEP 5: JTI Blacklist Check (Redis)  (~0.3ms, I/O)
│
│  After tenant cross-check passes, verify the token has not been revoked:
│  key = "auth:blacklist:" + claims.jti
│
│  Redis.EXISTS(key)
│    → exists → 401 UnauthorizedException: "Token has been revoked"
│    → not found → PASS
│
│  Blacklist entry is created when the user calls POST /auth/logout.
│  TTL = remaining JWT lifetime → entry auto-expires when the token would have expired anyway.
│
│  → PASS → req.user = JwtClaims → proceed to controller

Refresh & Revoke — Token Lifecycle

Đã được implement đầy đủ: Refresh token rotation, logout với JTI blacklist, và bảng refresh_tokens đều đã có trong codebase. JWT access token ngắn hạn — refresh token 7 ngày — dùng 1 lần (rotation).
Fully implemented: Refresh token rotation, logout with JTI blacklisting, and the refresh_tokens table are all present in the codebase. Short-lived JWT access tokens — 7-day refresh tokens — single-use (rotation).
refresh flow
POST /auth/refresh  — @Public, không cần tenant header
Body: { refreshToken: "<96-char hex>" }


BƯỚC 1: Hash token đầu vào  (~0ms)
│
│  tokenHash = SHA-256(refreshToken)  ← hàm crypto.createHash('sha256')
│  Chỉ lưu hash trong DB, không bao giờ lưu raw token → an toàn khi DB bị leak


BƯỚC 2: Tra cứu trong DB  (~2ms)
│
│  SELECT rt.*, u.email, u.is_active, u.tenant_id, array_agg(r.name) AS roles
│  FROM refresh_tokens rt
│  JOIN users u ON u.id = rt.user_id
│  LEFT JOIN user_roles ur ON ...
│  WHERE rt.token_hash = tokenHash
│
│  Không tìm thấy → 401 Unauthorized
│  rt.revoked_at IS NOT NULL → 401 "Refresh token has been revoked"
│  rt.expires_at < NOW() → 401 "Refresh token has expired"
│  u.is_active = false → 401 Unauthorized


BƯỚC 3: Token Rotation (atomic)  (~1ms)
│
│  1. Revoke old token:
│     UPDATE refresh_tokens SET revoked_at = NOW() WHERE token_hash = tokenHash
│
│  2. Generate new refresh token:
│     newRawToken = crypto.randomBytes(48).toString('hex')  ← 96-char hex
│     newHash = SHA-256(newRawToken)
│     INSERT INTO refresh_tokens (token_hash, user_id, tenant_id, expires_at)
│     VALUES (newHash, userId, tenantId, NOW() + 7 days)
│
│  3. Issue new JWT (với jti mới):
│     token = JWT.sign({ sub, jti: uuidv4(), tenant_id, email, roles }, secret)
│
│  Dùng 1 lần: refresh token cũ bị revoke ngay khi được dùng.
│  Nếu hacker dùng trước → token hợp lệ của user sẽ không hoạt động → detect replay.


BƯỚC 4: Response
│
│  200 OK
│  {
│    token:        "eyJ...",      ← JWT access token mới (short-lived)
│    refreshToken: "<96-char hex>" ← Refresh token mới (7 ngày)
│  }


DATABASE: bảng refresh_tokens

  CREATE TABLE refresh_tokens (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    token_hash  TEXT NOT NULL UNIQUE,     ← SHA-256 hash của raw token
    user_id     UUID NOT NULL,
    tenant_id   UUID NOT NULL,
    expires_at  TIMESTAMPTZ NOT NULL,     ← NOW() + 7 ngày
    revoked_at  TIMESTAMPTZ,              ← NULL = còn hiệu lực
    created_at  TIMESTAMPTZ DEFAULT NOW()
  );

Logout — JTI Blacklist

Logout ngay lập tức: POST /auth/logout yêu cầu JWT hợp lệ. JTI của access token được ghi vào Redis blacklist với TTL bằng thời gian còn lại của token — không cần query DB thêm. Mọi request sau đó với token đó sẽ bị từ chối tại AuthGuard.
Immediate invalidation: POST /auth/logout requires a valid JWT. The access token's JTI is written to the Redis blacklist with TTL equal to the token's remaining lifetime — no extra DB query needed. All subsequent requests using that token are rejected at the AuthGuard level.
logout flow
POST /auth/logout  — yêu cầu JWT hợp lệ + X-Tenant-ID header
Authorization: Bearer <token>
X-Tenant-ID: acme-123


BƯỚC 1: AuthGuard chạy bình thường
│
│  → Verify signature, validate claims, check blacklist, cross-check tenant
│  → req.user = { sub, jti, tenant_id, roles, exp, ... }


BƯỚC 2: Tính TTL còn lại  (~0ms)
│
│  now = Math.floor(Date.now() / 1000)   ← Unix timestamp seconds
│  ttl = claims.exp - now                ← số giây còn lại trước khi JWT hết hạn
│
│  if ttl <= 0 → token đã hết hạn, không cần blacklist


BƯỚC 3: Ghi vào Redis Blacklist  (~0.3ms)
│
│  key = "auth:blacklist:" + claims.jti
│  Redis.SET(key, "1", "EX", ttl)   ← TTL = remaining lifetime
│
│  JTI là UUID v4 duy nhất → key collision không thể xảy ra.
│  Entry tự xóa khi JWT hết hạn → Redis không tích lũy entries vô thời hạn.


BƯỚC 4: Response
│
│  204 No Content   ← thành công, không có body


KẾT QUẢ:

  Token bị logout:
  ├── Redis: auth:blacklist:<jti> = "1"  TTL = còn N giây
  └── Mọi request sau đó → AuthGuard bước 5 → 401 "Token has been revoked"

  Sau khi JWT hết hạn tự nhiên:
  └── Redis entry tự xóa → không còn gì để check
logout flow
POST /auth/logout  — requires valid JWT + X-Tenant-ID header
Authorization: Bearer <token>
X-Tenant-ID: acme-123


STEP 1: AuthGuard runs normally
│
│  → Verify signature, validate claims, check blacklist, cross-check tenant
│  → req.user = { sub, jti, tenant_id, roles, exp, ... }


STEP 2: Calculate remaining TTL  (~0ms)
│
│  now = Math.floor(Date.now() / 1000)   ← Unix timestamp in seconds
│  ttl = claims.exp - now                ← seconds remaining before JWT expires
│
│  if ttl <= 0 → token already expired, no blacklist entry needed


STEP 3: Write to Redis Blacklist  (~0.3ms)
│
│  key = "auth:blacklist:" + claims.jti
│  Redis.SET(key, "1", "EX", ttl)   ← TTL = remaining lifetime
│
│  JTI is UUID v4 — collision impossible.
│  Entry self-expires when the JWT would have expired anyway → no accumulation.


STEP 4: Response
│
│  204 No Content   ← success, no body


RESULT:

  Token after logout:
  ├── Redis: auth:blacklist:<jti> = "1"  TTL = N seconds remaining
  └── All subsequent requests → AuthGuard step 5 → 401 "Token has been revoked"

  After JWT naturally expires:
  └── Redis entry auto-deletes → nothing left to check

Security — L2 Gateway

Authentication & Authorization Flow

Toàn bộ luồng xác thực: đăng nhập → JWT → xác thực mỗi request → refresh → revoke → phân quyền RBAC.

Complete authentication flow: login → JWT → per-request verification → refresh → revoke → RBAC authorization.

00

Bản đồ Auth — Hai luồng chínhAuth Map — Two Main Flows

Nguyên tắc cốt lõi: JWT là nguồn sự thật duy nhất về danh tính. Tenant được xác định từ subdomain (TenantResolver), sau đó được xác nhận khớp với tenantId trong JWT (AuthGuard). Hai bước này tách biệt để tránh tenant impersonation.
Core principle: JWT is the single source of truth for identity. The tenant is resolved from the subdomain (TenantResolver), then verified to match the tenantId inside the JWT (AuthGuard). These two steps are separate to prevent tenant impersonation.
auth architecture
  LUỒNG 1: ĐĂNG NHẬP — Lấy tokens

  Browser/Client
      │  POST /auth/login  { email, password, subdomain }
      ▼
  L2 Gateway  ─── bỏ qua AuthGuard (endpoint public)
      │
      ▼
  AuthService  (L3 Business Logic)
      ├── Verify credentials → Metadata DB
      ├── Issue Access Token  (JWT RS256, TTL 15 phút)
      ├── Issue Refresh Token (opaque token, TTL 7 ngày)
      └── Store Refresh Token → Metadata DB
      │
      ▼
  Response: { accessToken, refreshToken, expiresIn: 900 }


  LUỒNG 2: GỌI API — Dùng access token

  Browser/Client
      │  POST /api/v1/sales
      │  Authorization: Bearer <accessToken>
      │  Host: acme.crm.com
      ▼
  L2 Gateway
      │
      ├── ① TenantResolver:  acme.crm.com → tenantId="acme-123"
      ├── ② AuthGuard:       verify JWT → extract userId, roles
      │                       cross-check: jwt.tenantId == "acme-123"  ← bảo vệ tenant isolation
      ├── ③ RateLimiter:     check per-tenant limit
      ├── ④ RouteMatcher:    POST /api/v1/sales → plugin="customer-care", method="createSale"
      └── ⑤ PermissionCheck: user có quyền sales:write trong tier premium không?
      │
      ▼
  L3 Business Logic (plugin execution)


  LUỒNG 3: TOKEN HẾT HẠN — Refresh

  Browser/Client
      │  POST /auth/refresh  { refreshToken }
      ▼
  AuthService
      ├── Verify refresh token còn hạn và chưa bị revoke
      ├── Issue Access Token mới  (TTL 15 phút)
      ├── Issue Refresh Token mới (TTL 7 ngày) ← rotation
      └── Revoke refresh token cũ
      │
      ▼
  Response: { accessToken, refreshToken, expiresIn: 900 }

01

Đăng nhập — Login FlowLogin — Login Flow

Endpoint public: POST /auth/login là endpoint duy nhất không qua AuthGuard. TenantResolver vẫn chạy để xác định tenant từ subdomain trước khi tìm user trong đúng tenant scope.
Public endpoint: POST /auth/login is the only endpoint that bypasses AuthGuard. TenantResolver still runs to identify the tenant from the subdomain before looking up the user in the correct tenant scope.
login flow
POST /auth/login
Host: acme.crm.com
Body: { email: "alice@acme.com", password: "..." }


BƯỚC 1: TenantResolver  (~1ms)
│
│  Subdomain "acme" → Redis GET domain-map:acme.crm.com → "acme-123"
│  tenant = { tenantId: "acme-123", tier: "premium", status: "active" }
│  status != active → 403 Forbidden  (nếu suspended/offboarded)
│  → PASS


BƯỚC 2: Validate Credentials  (~5ms)
│
│  SELECT * FROM users
│  WHERE email='alice@acme.com' AND tenant_id='acme-123'
│  → { userId: "user-5", passwordHash: "$argon2id$...", status: "active" }
│
│  Argon2id.verify(inputPassword, passwordHash)
│  → match
│
│  user.status != active → 401 Unauthorized  (nếu bị khóa)


BƯỚC 3: Load Roles & Permissions  (~2ms)
│
│  SELECT r.name, p.key FROM user_roles ur
│  JOIN roles r ON ur.role_id = r.id
│  JOIN role_permissions rp ON r.id = rp.role_id
│  JOIN permissions p ON rp.permission_id = p.id
│  WHERE ur.user_id='user-5' AND ur.tenant_id='acme-123'
│
│  → roles: ["sales_manager"]
│  → permissions: ["sales:read","sales:write","customer:read","customer:write"]


BƯỚC 4: Issue Access Token  (~1ms, CPU — RSA sign)
│
│  jti = UUID()  ← unique token ID cho blacklist
│  accessToken = JWT.sign({
│    header:  { alg: "RS256", typ: "JWT" },
│    payload: { sub:         "user-5",
│               tenantId:    "acme-123",
│               tier:        "premium",
│               roles:       ["sales_manager"],
│               permissions: ["sales:read","sales:write","customer:read","customer:write"],
│               iat:         1708000000,
│               exp:         1708000900,   ← iat + 900s (15 phút)
│               jti:         "tok-abc-123" }
│  }, privateKey)    ← RSA-2048 private key (rotated monthly)


BƯỚC 5: Issue Refresh Token  (~1ms)
│
│  refreshToken = crypto.randomBytes(48).toString('hex')  ← opaque, không phải JWT
│  refreshTokenHash = SHA256(refreshToken)
│
│  INSERT INTO refresh_tokens:
│  { tokenHash:  refreshTokenHash,
│    userId:     "user-5",
│    tenantId:   "acme-123",
│    expiresAt:  now() + 7d,
│    createdAt:  now(),
│    deviceInfo: request.userAgent }


BƯỚC 6: Response
│
│  200 OK
│  {
│    accessToken:  "eyJ...",
│    tokenType:    "Bearer",
│    expiresIn:    900,
│    refreshToken: "a3f9b2..."   ← opaque hex string
│  }
│
│  Client lưu:
│  - accessToken  → memory (không localStorage — tránh XSS)
│  - refreshToken → httpOnly cookie (tránh XSS)


TIMING:

  ┌──────────────────────────────┬────────┬─────────────────────────────────┐
  │ Bước                         │ Time   │ Note                            │
  ├──────────────────────────────┼────────┼─────────────────────────────────┤
  │ TenantResolver               │ ~1ms   │ Redis GET (cache hit)           │
  │ Validate credentials (Argon2)│ ~50ms  │ CPU-bound, intentionally slow   │
  │ Load roles/permissions       │ ~2ms   │ Metadata DB query               │
  │ Sign JWT (RSA-2048)          │ ~1ms   │ CPU — asymmetric crypto         │
  │ Store refresh token          │ ~1ms   │ Metadata DB INSERT              │
  ├──────────────────────────────┼────────┼─────────────────────────────────┤
  │ TOTAL                        │ ~55ms  │ Argon2 chiếm phần lớn (đúng)   │
  └──────────────────────────────┴────────┴─────────────────────────────────┘

  Argon2id chậm là cố ý — brute-force protection. Login không cần <1ms.
02

JWT & Claims — Cấu trúc TokenToken Structure

JWT là source of truth: Mọi thông tin cần thiết cho một request đều nằm trong JWT — không cần query DB để biết userId, tenantId, roles, permissions. AuthGuard chỉ cần verify signature và check blacklist.
JWT is the source of truth: All information needed for a request is embedded in the JWT — no DB query is required to know userId, tenantId, roles, or permissions. AuthGuard only needs to verify the signature and check the blacklist.
jwt structure
ACCESS TOKEN — JWT (RS256)

  eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9
  .eyJzdWIiOiJ1c2VyLTUiLCJ0ZW5hbnRJZCI6ImFjbWUtMTIzIi4uLn0
  .<RSA-2048 signature>

  ┌─────────────────────────────────────────────────────────────────────┐
  │ HEADER                                                              │
  │   alg: "RS256"   ← Asymmetric — verify bằng public key (safe)     │
  │   typ: "JWT"                                                        │
  └─────────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────────┐
  │ PAYLOAD                                                             │
  │                                                                     │
  │   sub:         "user-5"              ← userId (standard claim)     │
  │   tenantId:    "acme-123"            ← bắt buộc: tenant scope       │
  │   tier:        "premium"             ← để Permission Check nhanh   │
  │   roles:       ["sales_manager"]     ← RBAC roles                  │
  │   permissions: ["sales:read",        ← expanded permissions        │
  │                 "sales:write",           (denormalized từ roles)    │
  │                 "customer:read",                                    │
  │                 "customer:write"]                                   │
  │                                                                     │
  │   iat:  1708000000   ← issued at (Unix timestamp)                  │
  │   exp:  1708000900   ← iat + 900s = 15 phút                        │
  │   jti:  "tok-abc-123"← JWT ID — dùng làm blacklist key             │
  └─────────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────────┐
  │ SIGNATURE                                                           │
  │   RSA-2048 sign(base64(header) + "." + base64(payload), privateKey)│
  │   Verify: RSA verify(signature, publicKey)                         │
  │   Public key được cache tại mỗi app instance — không cần DB call  │
  └─────────────────────────────────────────────────────────────────────┘


REFRESH TOKEN — Opaque String

  a3f9b2c1d8e4f7a0b3c5d2e8f1a4b7c0d3e6f9a2b5c8d1e4f7a0b3c6d9e2

  ├── 48 random bytes → hex string (96 chars)
  ├── Không phải JWT — không chứa claims
  ├── Lưu dưới dạng SHA256 hash trong Metadata DB
  ├── TTL: 7 ngày (604800 giây)
  └── Xóa sau khi dùng (rotation) — mỗi refresh sinh token mới


SO SÁNH ACCESS vs REFRESH TOKEN:

  ┌──────────────────────┬─────────────────────────┬──────────────────────────┐
  │                      │ Access Token             │ Refresh Token            │
  ├──────────────────────┼─────────────────────────┼──────────────────────────┤
  │ Format               │ JWT (RS256)              │ Opaque hex string        │
  │ TTL                  │ 15 phút (900s)           │ 7 ngày (604800s)         │
  │ Lưu ở client        │ Memory (JS variable)     │ httpOnly cookie          │
  │ Chứa claims          │ Có (userId, roles, ...)  │ Không                    │
  │ Verify               │ Signature + exp + jti    │ DB lookup + exp check    │
  │ Revoke               │ Redis blacklist (jti)    │ DELETE từ DB             │
  │ Dùng để             │ Mọi API request          │ Lấy access token mới     │
  └──────────────────────┴─────────────────────────┴──────────────────────────┘


KEY ROTATION:

  RSA keypair được rotate định kỳ (monthly):
  ├── Sinh keypair mới
  ├── Public key mới được deploy lên tất cả instances (config reload)
  ├── Tokens ký bằng key cũ vẫn còn hạn → giữ old public key thêm 15 phút
  └── Sau 15 phút: xóa old public key → tokens cũ không verify được nữa
  (15 phút = max TTL của access token → đảm bảo không có token cũ nào còn hạn)
03

Xác thực Request — AuthGuard PipelineVerify Request — AuthGuard Pipeline

Chạy trên mỗi request (trừ /auth/*): AuthGuard là middleware thứ 3 trong L2 Gateway, sau TenantResolver và trước RateLimiter. Nếu bất kỳ bước nào fail, request bị reject ngay, không forward xuống L3.
Runs on every request (except /auth/*): AuthGuard is the 3rd middleware in the L2 Gateway, after TenantResolver and before RateLimiter. If any step fails, the request is rejected immediately without being forwarded to L3.
authguard pipeline
Mỗi request có Authorization: Bearer <token>


BƯỚC 1: Extract Token  (~0ms)
│
│  header = request.headers['Authorization']
│  if !header || !header.startsWith('Bearer ')
│    → 401 { code: "MISSING_TOKEN", message: "Authorization header required" }
│
│  token = header.slice(7)  ← "Bearer " = 7 chars


BƯỚC 2: Decode & Verify Signature  (~0.5ms, CPU)
│
│  decoded = JWT.verify(token, publicKey, { algorithms: ['RS256'] })
│
│  Nếu signature không hợp lệ:
│    → 401 { code: "INVALID_TOKEN" }
│
│  Public key được cache in-memory tại mỗi instance.
│  Không cần network call để verify signature.


BƯỚC 3: Kiểm tra Expiry  (~0ms)
│
│  if decoded.exp < Math.floor(Date.now() / 1000)
│    → 401 { code: "TOKEN_EXPIRED", message: "Token đã hết hạn, vui lòng refresh" }


BƯỚC 4: Cross-check Tenant  (~0ms)
│
│  Đây là bước bảo vệ tenant isolation quan trọng nhất:
│  resolvedTenantId = request.tenant.tenantId   ← từ TenantResolver (bước trước)
│
│  if decoded.tenantId !== resolvedTenantId
│    → 403 { code: "TENANT_MISMATCH" }
│
│  Tại sao cần bước này?
│  Ngăn user của tenant A dùng token hợp lệ để gọi API trên subdomain tenant B.
│  Nếu không check: alice@acme.com với valid JWT có thể gọi beta.crm.com.


BƯỚC 5: Kiểm tra Blacklist  (~0.3ms, Redis)
│
│  result = Redis.EXISTS('blacklist:' + decoded.jti)
│  if result === 1
│    → 401 { code: "TOKEN_REVOKED", message: "Token đã bị thu hồi" }
│
│  JTI được đưa vào blacklist khi: logout, force-logout, password change.
│  Redis key có TTL = thời gian còn lại của token → tự cleanup.


BƯỚC 6: Attach vào Request Context  (~0ms)
│
│  request.auth = {
│    userId:      decoded.sub,          ← "user-5"
│    tenantId:    decoded.tenantId,     ← "acme-123"
│    tier:        decoded.tier,         ← "premium"
│    roles:       decoded.roles,        ← ["sales_manager"]
│    permissions: decoded.permissions,  ← ["sales:read", "sales:write", ...]
│    jti:         decoded.jti           ← "tok-abc-123"
│  }
│  → PASS → tiếp tục sang RateLimiter


TIMING BREAKDOWN:

  ┌──────────────────────────────┬─────────┬─────────────────────────────────┐
  │ Bước                         │ Time    │ I/O                             │
  ├──────────────────────────────┼─────────┼─────────────────────────────────┤
  │ Extract + parse              │ ~0ms    │ None                            │
  │ RS256 signature verify       │ ~0.5ms  │ None (in-memory public key)     │
  │ Expiry check                 │ ~0ms    │ None                            │
  │ Tenant cross-check           │ ~0ms    │ None                            │

  │ TOTAL AuthGuard              │ ~1ms    │ 1 Redis call                    │

  Đây là phần đóng góp ~1ms trong tổng ~3.9ms của L2 Gateway.

04

Refresh & Revoke — Token Lifecycle

Refresh token rotation: Mỗi lần refresh đều sinh ra cả access token mới lẫn refresh token mới, đồng thời vô hiệu hóa refresh token cũ. Điều này giúp phát hiện refresh token bị đánh cắp (nếu token cũ được dùng lại sau rotation → alert).
Refresh token rotation: Every refresh issues both a new access token and a new refresh token, while simultaneously invalidating the old refresh token. This enables detection of stolen refresh tokens (if the old token is reused after rotation → alert).
token refresh flow
ACCESS TOKEN HẾT HẠN — Client nhận 401 TOKEN_EXPIRED

POST /auth/refresh
Cookie: refreshToken=a3f9b2...  (httpOnly)
Không cần Authorization header — đây là endpoint public


BƯỚC 1: Tìm Refresh Token  (~2ms)
│
│  tokenHash = SHA256(requestCookie.refreshToken)
│
│  SELECT * FROM refresh_tokens
│  WHERE token_hash = tokenHash AND revoked_at IS NULL
│  → { userId: "user-5", tenantId: "acme-123", expiresAt: ... }
│
│  Nếu không tìm thấy:
│    → 401 { code: "INVALID_REFRESH_TOKEN" }


BƯỚC 2: Kiểm tra Expiry  (~0ms)
│
│  if token.expiresAt < now()
│    DELETE FROM refresh_tokens WHERE token_hash = tokenHash
│    → 401 { code: "REFRESH_TOKEN_EXPIRED", message: "Vui lòng đăng nhập lại" }


BƯỚC 3: Load User + Roles  (~2ms)
│
│  SELECT u.*, r.name as role FROM users u
│  JOIN user_roles ur ON u.id = ur.user_id
│  JOIN roles r ON ur.role_id = r.id
│  WHERE u.id = token.userId AND u.tenant_id = token.tenantId
│
│  if user.status != 'active'
│    → 401 { code: "USER_INACTIVE" }


BƯỚC 4: Rotation — Revoke Old, Issue New  (~2ms)
│
│  Trong một transaction:
│  ├── UPDATE refresh_tokens SET revoked_at=now() WHERE token_hash=tokenHash
│  ├── newRefreshToken = crypto.randomBytes(48).toString('hex')
│  └── INSERT INTO refresh_tokens { newTokenHash, userId, tenantId, expiresAt: now()+7d }
│
│  newAccessToken = JWT.sign({ sub, tenantId, tier, roles, permissions,
│                               iat: now(), exp: now()+900, jti: newJti }, privateKey)


BƯỚC 5: Response
│
│  200 OK
│  Set-Cookie: refreshToken=<newRefreshToken>; HttpOnly; Secure; SameSite=Strict
│  Body: { accessToken: "eyJ...", expiresIn: 900 }


PHÁT HIỆN TOKEN BỊ ĐÁO CẮP (Refresh Token Reuse Detection):

  Nếu refresh token cũ được dùng lại sau khi đã rotation:
  │  Token cũ đã bị revoked (revoked_at IS NOT NULL)
  │  → Đây là dấu hiệu token bị đánh cắp hoặc replay attack
  │
  │  Action:
  │  ├── Revoke TẤT CẢ refresh tokens của user này trong tenant đó
  │  ├── Đưa TẤT CẢ active JTIs của user vào blacklist
  │  ├── Gửi email cảnh báo cho user
  │  └── Log security event (audit trail)
  └── → 401 { code: "TOKEN_REUSE_DETECTED" }
revoke & logout
LOGOUT — Thu hồi token hiện tại

POST /auth/logout
Authorization: Bearer <accessToken>
Cookie: refreshToken=a3f9b2...

│  ① Revoke access token: Redis SETEX 'blacklist:{jti}' {remainingTTL} 1
│     remainingTTL = decoded.exp - now()
│     Key tự xóa sau remainingTTL giây → Redis không tích lũy entries cũ.
│
│  ② Revoke refresh token: UPDATE refresh_tokens SET revoked_at=now()
│     WHERE token_hash=SHA256(cookie)
│
│  ③ Response: 200 OK
│              Set-Cookie: refreshToken=; expires=Thu, 01 Jan 1970... (xóa cookie)


FORCE LOGOUT — Admin thu hồi tất cả sessions của một user

DELETE /system/users/{userId}/sessions  (System Admin hoặc Tenant Admin)

│  ① Revoke tất cả refresh tokens:
│     UPDATE refresh_tokens SET revoked_at=now()
│     WHERE user_id=userId AND tenant_id=tenantId AND revoked_at IS NULL
│
│  ② Blacklist tất cả active JTIs:
│     Không thể enumerate JTIs từ DB (chúng nằm trong JWT payload).
│     Thay vào đó: thêm user-level revocation record.
│     Redis SETEX 'user-revoked:{userId}' {maxTTL} {timestamp}
│     AuthGuard kiểm tra thêm: nếu key tồn tại và iat < timestamp → reject
│     maxTTL = 7 ngày (max access token TTL nếu user có nhiều tokens cũ)
│
│  ③ Response: 200 OK — user phải đăng nhập lại trên mọi thiết bị


PASSWORD CHANGE — Tự động force logout

  Khi user đổi password:
  ├── Hash password mới → lưu vào DB
  ├── Trigger Force Logout flow ở trên (tất cả sessions bị kill)
  └── Issue token mới cho session hiện tại (nếu user muốn ở lại)
05

Phân quyền — RBAC & Permission CheckAuthorization — RBAC & Permission Check

3 lớp kiểm tra độc lập: (1) Plugin có được enable cho tenant không? (2) User có role/permission không? (3) Tier của tenant có hỗ trợ feature này không? Cả 3 phải PASS thì request mới được forward xuống L3.
3 independent check layers: (1) Is the plugin enabled for the tenant? (2) Does the user have the required role/permission? (3) Does the tenant's tier support this feature? All 3 must PASS before the request is forwarded to L3.
rbac model
MÔ HÌNH RBAC — Tenant-scoped

  ┌─────────────────────────────────────────────────────────────────────────┐
  │                                                                         │
  │  User  ──── (trong tenant) ────►  Role  ────►  Permission              │
  │                                                                         │
  │  alice@acme.com  →  sales_manager  →  sales:read, sales:write          │
  │                                        customer:read, customer:write    │
  │                                                                         │
  │  bob@acme.com    →  viewer         →  sales:read, customer:read         │
  │                                                                         │
  │  carol@acme.com  →  admin          →  *.* (tất cả permissions)         │
  │                                                                         │
  └─────────────────────────────────────────────────────────────────────────┘

  Roles là per-tenant: "sales_manager" ở ACME ≠ "sales_manager" ở BETA.
  Mỗi tenant có thể định nghĩa roles tùy theo plugin nào đã enable.


BẢNG ROLES MẶC ĐỊNH (theo tier):

  ┌──────────────────┬────────────────────────────────────────────────────┐
  │ Role             │ Permissions mặc định                               │
  ├──────────────────┼────────────────────────────────────────────────────┤
  │ admin            │ *.* (toàn bộ trong tenant scope)                   │
  │ sales_manager    │ sales:*, customer:*, analytics:read                │
  │ sales_rep        │ sales:read, sales:write, customer:read             │
  │ viewer           │ *.read (mọi plugin, read-only)                     │
  │ support          │ customer:read, customer:write, sales:read          │
  └──────────────────┴────────────────────────────────────────────────────┘


PERMISSION NAMING CONVENTION:

  {plugin}:{action}

  sales:read        ← đọc sale records
  sales:write       ← tạo/sửa sale
  sales:delete      ← xóa sale
  customer:read     ← đọc customer
  customer:write    ← sửa customer
  analytics:view    ← xem dashboard analytics
  marketing:send    ← gửi campaign
  admin:users       ← quản lý users trong tenant
permission check pipeline
PERMISSION CHECK — Sau AuthGuard, tại PermissionCheck middleware (L2)

  Thực hiện sau khi RouteMatcher đã xác định plugin + requiredPermission:
  route = { plugin: "customer-care", method: "createSale",
             requiredPermission: "sales:write" }


KIỂM TRA 1: Plugin có enabled cho tenant không?  (~0ms, in-memory)
│
│  tenantContext = TenantContextManager.get("acme-123")  ← từ LRU cache
│  if !tenantContext.enabledPlugins.includes("customer-care")
│    → 404 { code: "PLUGIN_NOT_FOUND" }
│  → PASS


KIỂM TRA 2: User có permission không?  (~0ms, in-memory)
│
│  userPermissions = request.auth.permissions  ← từ JWT payload
│  if !userPermissions.includes("sales:write")
│    → 403 { code: "PERMISSION_DENIED",
│              message: "Bạn không có quyền sales:write" }
│  → PASS


KIỂM TRA 3: Tier có hỗ trợ feature không?  (~0ms, in-memory)
│
│  tierFeatures = TierConfig.get("premium")  ← static config
│  if !tierFeatures.includes("sales:write")
│    → 403 { code: "FEATURE_NOT_IN_TIER",
│              message: "Feature này không có trong gói Premium" }
│  → PASS


→ TẤT CẢ PASS → Forward xuống L3 với full context

  request.context = {
    tenantId:    "acme-123",
    tier:        "premium",
    userId:      "user-5",
    roles:       ["sales_manager"],
    permissions: ["sales:read","sales:write","customer:read","customer:write"],
    plugin:      "customer-care",
    method:      "createSale"
  }
permissions in sandbox
TRONG SANDBOX — Plugin Code Kiểm tra Permission

  Plugin code chạy trong SandboxEngine (L3) nhận ExecutionContext:

  // CustomerCareCore.createSale(ctx, input)
  //
  // ctx.user.can(permission) → boolean
  // ctx.user.roles          → string[]
  // ctx.user.id             → string

  if (!ctx.user.can('sales:write')) {
    throw new PermissionError('sales:write required');
  }

  Tại sao cần check lại trong plugin nếu L2 đã check?
  Vì một plugin có thể gọi sub-operations yêu cầu permissions khác nhau.
  Ví dụ: createSale cũng ghi audit → cần audit:write.
  L2 chỉ check permission của action chính. Plugin tự check sub-actions.


PERMISSION TRONG EXECUTION CONTEXT:

  ExecutionContext {
    tenantId:  "acme-123",
    db:        TenantScopedDB,       ← auto-scoped, không cần filter thủ công
    cache:     TenantScopedCache,
    logger:    ScopedLogger,
    eventBus:  TenantScopedEventBus,
    user: {
      id:          "user-5",
      roles:       ["sales_manager"],
      permissions: ["sales:read","sales:write","customer:read","customer:write"],
      can: (perm) => this.permissions.includes(perm)
    }
  }

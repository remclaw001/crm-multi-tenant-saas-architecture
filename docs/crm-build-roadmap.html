Build Roadmap

Thứ tự xây dựng hệ thống CRM multi-tenant SaaS — từ nền tảng hạ tầng đến production hardening.

Nguyên tắc thứ tự

Multi-tenancy bake-in từ ngày đầu. Query Interceptor và Connection Pool Registry là xương sống — phải tồn tại trước khi viết bất kỳ dòng business logic nào. Retrofit sau sẽ dẫn đến data leak tiềm ẩn và refactor toàn bộ.

Đi từ trong ra ngoài: Data Layer → Logic Layer → API Layer → UI Layer. Không bao giờ xây UI trước khi API ổn định.

Observability sớm hơn bạn nghĩ — Phase 4, không phải Phase cuối. Debug giai đoạn phát triển không có structured log và trace là cực kỳ tốn thời gian.

01

Local Dev Environment

Thiết lập toàn bộ hạ tầng cục bộ, schema tenant, Row-Level Security và config foundation. Đây là nền móng — mọi phase sau đều chạy trên đây.

Tại sao trước tiên

Không có PostgreSQL với RLS thì không test được multi-tenancy đúng nghĩa. Schema sai ở giai đoạn này = refactor toàn bộ sau. Thiết lập một lần, dùng xuyên suốt.

📋 Danh sách công việc
  • ★
    Docker Compose với PG 16, Redis 7, RabbitMQ — môi trường dev one-command
  • ★
    Tạo bảng tenants (id, name, subdomain, tier, db_url, config, is_active, created_at, updated_at)
  • ★
    Thiết lập Row-Level Security — ENABLE + FORCE RLS, policy USING + WITH CHECK dựa trên app.tenant_id session variable
  • ★
    Connection pool afterCreate hook — reset app.tenant_id = NULL khi connection trả về pool, ngăn context leak giữa các request
  • ✓
    Tạo bảng users, roles, permissions, user_roles, role_permissions với cột tenant_id trên các bảng tenant-scoped
  • ✓
    Viết seed data: 2 standard tenants + 1 VIP tenant để test isolation
  • ✓
    dotenv + Zod schema validate toàn bộ env variables lúc startup
  • ✓
    Knex migration baseline — 20260222000001_init_schema.ts
  • ✓
    Cấu hình pg_stat_statements, pgcrypto, uuid-ossp extensions qua scripts/postgres-init.sql (Docker init hook)
🏁 Deliverables
▸docker-compose.yml — toàn bộ services up với docker compose up
▸RLS hoạt động: query không có app.tenant_id trả về 0 rows — kể cả với table owner (FORCE ROW LEVEL SECURITY)
▸Zod config schema — app crash rõ ràng nếu thiếu env var
▸Knex migration 20260222000001_init_schema.ts chạy được (có up() và down())
▸Seed data với 3 tenants khác tier, đủ roles và users per tenant
▸Pool afterCreate hook đảm bảo không có tenant context leak giữa các connection

Test RLS bằng cách: SET app.tenant_id = '<uuid>' rồi SELECT — chỉ thấy data của tenant đó. Không SET → 0 rows.

02

Query Interceptor + Data Access Layer (L4)

Xây dựng cơ chế tự động scope mọi query theo tenant. Đây là thành phần CRITICAL nhất — toàn bộ tính đúng đắn của multi-tenancy phụ thuộc vào đây.

Tại sao là phase 2 (Critical)

Nếu viết business logic trước khi có interceptor, sẽ có query "trần" không có tenant scope lọt vào codebase. Tìm và sửa lại cực kỳ tốn công và dễ bỏ sót. Interceptor phải là lưới an toàn mặc định.

📋 Danh sách công việc
  • ★
    Knex.js Query Interceptor — hook vào queryBuilder runner, tự động inject WHERE tenant_id = ? cho mọi SELECT/UPDATE/DELETE
  • ★
    AsyncLocalStorage TenantContext — truyền tenant_id xuyên suốt call stack không cần truyền tham số thủ công
  • ★
    Connection Pool Registry — shared pool (max=200), VIP pool (max=30/tenant), metadata pool (max=20)
  • ✓
    Cache Manager — ioredis wrapper enforce key pattern t:<id>:<type>:<id>
  • ✓
    QueryCounterMiddleware — đếm số query/request, chuẩn bị cho sandbox limit (Phase 6)
  • ✓
    Unit test: query từ tenant A không thấy data của tenant B dù không có WHERE thủ công
  • ✓
    Interface IDbContext, ICacheManager — chuẩn bị cho DIP (Phase 7)
🏁 Deliverables
▸Query Interceptor: mọi query tự động có tenant scope, không cần developer nhớ thêm WHERE
▸Pool Registry: trả đúng pool theo tier của tenant (shared vs VIP)
▸Cache Manager: set/get/invalidate với key pattern chuẩn
▸Test cross-tenant isolation pass 100%

03

API Server + Tenant Resolution + Auth (L2–L3)

NestJS server nhận HTTP request, resolve tenant từ subdomain/header, verify JWT và đưa context vào AsyncLocalStorage để L4 tự động scope.

Tại sao sau L4

Nếu xây API server trước khi có L4 interceptor, các handler đầu tiên sẽ query DB không có tenant scope. Đây là lỗi kiến trúc nguy hiểm, khó phát hiện và khó sửa về sau.

📋 Danh sách công việc
  • ★
    Khởi tạo NestJS v10 + TypeScript — cấu hình module, ValidationPipe, global filters
  • ★
    Tenant resolution middleware — parse subdomain (acme.app.com) hoặc header (X-Tenant-ID) → lookup DB → set AsyncLocalStorage
  • ★
    JWT verification — fetch JWKS từ Keycloak, verify signature via JwtAuthGuard (passport-jwt), extract tenant claims
  • ✓
    Rate limiting cơ bản với @nestjs/throttler (Kong thay thế ở Phase 10)
  • ✓
    Health check endpoint GET /health và GET /ready
  • ✓
    Route registration cấu trúc (prefix: /api/v1/:plugin/)
  • ✓
    Integration test: request với JWT hợp lệ → query DB đúng tenant
🏁 Deliverables
▸HTTP server nhận request, resolve tenant, verify JWT trong <5ms overhead
▸Request từ tenant A không thể truy cập data tenant B dù JWT hợp lệ
▸Request không có JWT → 401. JWT sai tenant → 403
▸Health check endpoint trả về status DB + Redis + RabbitMQ

Cài đặt structured logging, distributed tracing và metrics collection trước khi viết business logic. Debug không có log/trace là mù.

Tại sao sớm thế này (Phase 4, không phải Phase cuối)

Retrofit tracing vào code đã viết rất tốn công. Và trong giai đoạn phát triển Phase 5–8, bạn sẽ cần log có cấu trúc để debug plugin execution, query count, cache hit/miss. Thiếu observability = mò mẫm trong bóng tối.

05

Execution Context + Plugin Registry (L3)

Xây dựng cơ chế load plugin, build ExecutionContext và Hook Registry. Đây là trái tim của Plugin System — chưa cần sandbox cứng.

Tại sao trước Sandbox

Cần hiểu plugin chạy như thế nào trong ExecutionContext trước khi nhốt nó vào isolate. Sandbox (Phase 6) bọc xung quanh registry đã có, không thay thế nó.

📋 Danh sách công việc
  • ★
    ExecutionContext builder — load tenant config từ DB (+ cache), danh sách plugin enabled, user claims, permission check
  • ★
    Plugin manifest loader — đọc plugin.manifest.json, validate bằng Zod schema (dependencies, permissions, resource limits)
  • ★
    Plugin Registry — đăng ký plugin cores: CustomerData, CustomerCare, Analytics, Automation, Marketing
  • ★
    Hook Registry — before/after/filter hooks với priority queue (TinyQueue min-heap)
  • ✓
    Route registry — map (method, url) → plugin handler
  • ✓
    Interface IPluginCore, IPluginRegistry, IExecutionContext
  • ✓
    Per-tenant enable/disable plugin (lookup từ tenant config)
🏁 Deliverables
▸Plugin được load, manifest validated, core đăng ký thành công
▸Hook fired đúng thứ tự priority (before → handler → after)
▸Tenant A chỉ thấy plugin mình đã enable
▸ExecutionContext chứa đủ: tenant config, user claims, DB connection, cache client

06

Plugin Sandbox — Hard Resource Limits (L3)

Bọc plugin execution trong V8 isolate độc lập. Enforce limits: 5s timeout, 50MB memory, 50 queries/request. Plugin không thể ảnh hưởng host process.

Tại sao sau Plugin Registry

Sandbox là lớp bảo vệ bọc xung quanh registry đã có. Cần hiểu plugin chạy gì trước khi hạn chế nó. Nếu làm ngược lại, sẽ design sandbox theo giả định sai về cách plugin hoạt động.

📋 Danh sách công việc
  • ★
    Isolated-VM wrapper — mỗi plugin call chạy trong V8 isolate riêng, không share heap với host
  • ★
    Set timeout: 5000 — plugin quá 5 giây bị kill, trả về TimeoutError
  • ★
    Set memoryLimitMb: 50 — plugin dùng quá 50MB bị kill, trả về MemoryError
  • ★
    Tích hợp QueryCounterMiddleware (Phase 2) — throw khi vượt 50 queries/request
  • ✓
    Context bridge — expose safe API (DB query, cache) vào isolate, không expose Node.js globals
  • ✓
    Test: plugin loop vô hạn → bị timeout. Plugin malloc quá lớn → bị kill. Plugin 51 queries → bị reject
  • ✓
    Metrics: sandbox execution time histogram, violation count per limit type
🏁 Deliverables
▸Plugin chạy trong isolate — crash plugin không crash host process
▸3 loại limit đều được enforce và có error message rõ ràng
▸Plugin chỉ truy cập được API được whitelist, không thể require('fs')

07

Cross-Cutting — Error Handling · Security (L6)

Chuẩn hóa error responses, thêm security headers và encryption. NestJS DI đã được wire xuyên suốt từ P3–P6 — Phase này tập trung vào các cross-cutting concerns còn lại chưa được xử lý.

Tại sao sau Plugin System

NestJS IoC container đã wire tất cả service xuyên suốt P3–P6 — không cần DI library bên ngoài. Phase 7 hoàn thiện các lớp bảo vệ còn thiếu: error contract thống nhất, security headers, và encryption cho PII. Làm sớm hơn không có ý nghĩa vì error hierarchy cần biết tất cả loại lỗi từ P2–P6 để map đúng HTTP status.

📋 Danh sách công việc
  • ★
    Error hierarchy — AppError → DomainError → PluginError → ValidationError với HTTP status mapping rõ ràng
  • ★
    NestJS HttpExceptionFilter (APP_FILTER) — chuẩn hóa tất cả error response theo RFC 7807 Problem Details JSON (đã wire ở P3, hoàn thiện mapping ở P7)
  • ✓
    Helmet.js — CSP, HSTS, X-Frame-Options, X-Content-Type-Options
  • ✓
    CORS configuration — per-tenant allowed origins lấy từ tenant config trong DB
  • ✓
    bcrypt hashing cho passwords (cost 12)
  • ✓
    AES-256-GCM encryption cho PII fields trong DB (dùng Node.js built-in crypto)
  • ✓
    Sentry error tracking (hoặc GlitchTip self-hosted) — capture unhandled exceptions với tenant context
🏁 Deliverables
▸Mọi error response có format thống nhất: {"type", "title", "status", "detail", "instance"}
▸Security headers pass Mozilla Observatory scan ≥ B+
▸PII fields (email, phone, address) được encrypted at rest — plain text không bao giờ persist vào DB
▸Unhandled exception tự động xuất hiện trên Sentry dashboard với tenant_id và trace_id đính kèm

08

Async Workers — RabbitMQ · BullMQ (L5)

Tách các tác vụ nặng (email, search indexing, webhooks, audit) ra khỏi request path. Request trả về ngay, worker xử lý async phía sau.

Tại sao sau Cross-cutting

Worker cần error handling chuẩn (Phase 7) để xử lý retry khi fail. NestJS DI đã sẵn sàng từ P3 — worker chỉ cần khai báo thêm module là inject được service. Xây worker trước khi có error hierarchy = mỗi worker tự xử lý lỗi theo cách riêng, không nhất quán.

📋 Danh sách công việc
  • ★
    RabbitMQ exchanges: notifications, search.index, webhooks, audit — với DLX (dead-letter exchange)
  • ★
    Audit writer worker — consume từ audit exchange, write vào audit_logs table bất đồng bộ
  • ✓
    Notification worker — email (Nodemailer/Resend), push, in-app
  • ✓
    Search indexing worker — consume events, push document vào Elasticsearch
  • ✓
    Webhook worker — outbound HTTP calls đến tenant-defined endpoints
  • ✓
    BullMQ cho job queue: retry với exponential backoff, priority, delay
  • ✓
    Bull Board UI — monitor job queue trong môi trường dev
  • ✓
    node-cron jobs: cleanup expired sessions, generate periodic reports
🏁 Deliverables
▸Request path không bị block bởi I/O nặng — latency giảm đáng kể
▸Failed job tự retry với backoff, sau N lần vào DLQ để inspect
▸Audit log được ghi async — không mất event dù request fail
▸Bull Board hiển thị queue depth, throughput, failed jobs

09

Presentation Layer (L1)

Xây giao diện người dùng theo thứ tự ưu tiên: Admin Console → Web App → Module Federation Plugin UI → Mobile. Frontend bind vào API đã ổn định.

Tại sao sau cùng trong backend phases

Frontend thay đổi nhiều nhất trong vòng đời sản phẩm. Xây UI trước khi API ổn định = rebuild nhiều lần. API contract phải stable trước — UI bind vào API, không phải ngược lại.

📋 Thứ tự xây dựng
  • ①
    Admin Console (Next.js + Shadcn/ui) — quản lý tenant, enable/disable plugin, xem metrics. Cần trước để operator quản lý hệ thống
  • ②
    Web App chính (React 18 + Next.js) — CRM UI cho end users: contacts, deals, tasks, activities
  • ③
    Module Federation — tách plugin UI thành Webpack remote, host app lazy-load theo manifest
  • ④
    Mobile (React Native) — sau cùng, khi UX pattern đã được validate trên web
🏁 Deliverables

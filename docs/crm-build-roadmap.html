Build Roadmap

Thứ tự xây dựng hệ thống CRM multi-tenant SaaS — từ nền tảng hạ tầng đến production hardening.

Build order for the multi-tenant SaaS CRM system — from infrastructure foundation to production hardening.

Nguyên tắc thứ tựOrdering Principles

Multi-tenancy bake-in từ ngày đầu. Query Interceptor và Connection Pool Registry là xương sống — phải tồn tại trước khi viết bất kỳ dòng business logic nào. Retrofit sau sẽ dẫn đến data leak tiềm ẩn và refactor toàn bộ.

Multi-tenancy baked in from day one. The Query Interceptor and Connection Pool Registry are the backbone — they must exist before writing any line of business logic. Retrofitting later leads to potential data leaks and a full refactor.

Đi từ trong ra ngoài: Data Layer → Logic Layer → API Layer → UI Layer. Không bao giờ xây UI trước khi API ổn định.

Build from the inside out: Data Layer → Logic Layer → API Layer → UI Layer. Never build the UI before the API is stable.

Observability sớm hơn bạn nghĩ — Phase 4, không phải Phase cuối. Debug giai đoạn phát triển không có structured log và trace là cực kỳ tốn thời gian.

Observability earlier than you think — Phase 4, not the last phase. Debugging during development without structured logs and traces is extremely time-consuming.

Local Dev Environment

Thiết lập toàn bộ hạ tầng cục bộ, schema tenant, Row-Level Security và config foundation. Đây là nền móng — mọi phase sau đều chạy trên đây.

Set up the complete local infrastructure, tenant schema, Row-Level Security, and config foundation. This is the foundation — every subsequent phase runs on top of it.

Tại sao trước tiên

Không có PostgreSQL với RLS thì không test được multi-tenancy đúng nghĩa. Schema sai ở giai đoạn này = refactor toàn bộ sau. Thiết lập một lần, dùng xuyên suốt.

Why First

Without PostgreSQL with RLS you cannot properly test multi-tenancy. A wrong schema at this stage = full refactor later. Set it up once, use it throughout.

02

Query Interceptor + Data Access Layer (L4)

Xây dựng cơ chế tự động scope mọi query theo tenant. Đây là thành phần CRITICAL nhất — toàn bộ tính đúng đắn của multi-tenancy phụ thuộc vào đây.

Build the mechanism that automatically scopes every query per tenant. This is the most CRITICAL component — the entire correctness of multi-tenancy depends on it.

Tại sao là phase 2 (Critical)

Nếu viết business logic trước khi có interceptor, sẽ có query "trần" không có tenant scope lọt vào codebase. Tìm và sửa lại cực kỳ tốn công và dễ bỏ sót. Interceptor phải là lưới an toàn mặc định.

Why Phase 2 (Critical)

If you write business logic before the interceptor exists, "bare" queries without tenant scope will slip into the codebase. Finding and fixing them is extremely costly and easy to miss. The interceptor must be the default safety net.

API Server + Tenant Resolution + Auth (L2–L3)

NestJS server nhận HTTP request, resolve tenant từ subdomain/header, verify JWT và đưa context vào AsyncLocalStorage để L4 tự động scope.

NestJS server receives HTTP requests, resolves the tenant from subdomain/header, verifies JWT, and places context into AsyncLocalStorage for L4 to auto-scope.

Tại sao sau L4

Nếu xây API server trước khi có L4 interceptor, các handler đầu tiên sẽ query DB không có tenant scope. Đây là lỗi kiến trúc nguy hiểm, khó phát hiện và khó sửa về sau.

Why After L4

If you build the API server before the L4 interceptor exists, the first handlers will query the DB without tenant scope. This is a dangerous architectural flaw that is hard to detect and hard to fix later.

Observability — Logging · Tracing · Metrics (L7)

Cài đặt structured logging, distributed tracing và metrics collection trước khi viết business logic. Debug không có log/trace là mù.

Install structured logging, distributed tracing, and metrics collection before writing business logic. Debugging without logs/traces is flying blind.

Tại sao sớm thế này (Phase 4, không phải Phase cuối)

Retrofit tracing vào code đã viết rất tốn công. Và trong giai đoạn phát triển Phase 5–8, bạn sẽ cần log có cấu trúc để debug plugin execution, query count, cache hit/miss. Thiếu observability = mò mẫm trong bóng tối.

Why This Early (Phase 4, Not the Last Phase)

Retrofitting tracing into already-written code is very costly. And during Phases 5–8 development, you will need structured logs to debug plugin execution, query count, cache hit/miss. No observability = groping in the dark.

Xây dựng cơ chế load plugin, build ExecutionContext và Hook Registry. Đây là trái tim của Plugin System — chưa cần sandbox cứng.

Build the plugin loading mechanism, ExecutionContext, and Hook Registry. This is the heart of the Plugin System — no hard sandbox needed yet.

Tại sao trước Sandbox

Why Before Sandbox

You need to understand how a plugin runs in the ExecutionContext before locking it into an isolate. The Sandbox (Phase 6) wraps around the existing registry, it does not replace it.

Plugin Sandbox — Hard Resource Limits (L3)

Bọc plugin execution trong V8 isolate độc lập. Enforce limits: 5s timeout, 50MB memory, 50 queries/request. Plugin không thể ảnh hưởng host process.

Wrap plugin execution in an independent V8 isolate. Enforce limits: 5s timeout, 50MB memory, 50 queries/request. A plugin cannot affect the host process.

Tại sao sau Plugin Registry

Sandbox là lớp bảo vệ bọc xung quanh registry đã có. Cần hiểu plugin chạy gì trước khi hạn chế nó. Nếu làm ngược lại, sẽ design sandbox theo giả định sai về cách plugin hoạt động.

Why After Plugin Registry

The Sandbox is a protective layer that wraps around the existing registry. You need to understand what a plugin does before restricting it. Doing it the other way around means designing the sandbox based on wrong assumptions about how plugins work.

Async Workers — RabbitMQ · BullMQ (L5)

Move heavy tasks (email, search indexing, webhooks, audit) out of the request path. Request returns immediately, worker handles them asynchronously in the background.

Tại sao sau Cross-cutting

Worker cần error handling chuẩn (Phase 7) để xử lý retry khi fail. NestJS DI đã sẵn sàng từ P3 — worker chỉ cần khai báo thêm module là inject được service. Xây worker trước khi có error hierarchy = mỗi worker tự xử lý lỗi theo cách riêng, không nhất quán.

Why After Cross-cutting

Workers need proper error handling (Phase 7) to handle retries on failure. NestJS DI is ready from P3 — a worker only needs to declare an additional module to inject a service. Building workers before the error hierarchy = each worker handles errors its own way, inconsistently.

01

Plugin Manager — Lifecycle Controller

Vai trò: Quản lý toàn bộ vòng đời plugin — deploy, upgrade, enable/disable per-tenant, canary rollout, rollback. Phối hợp với Registry, Sandbox, Migration Engine (L4), và MQ (L5).
Role: Manages the complete plugin lifecycle — deploy, upgrade, enable/disable per-tenant, canary rollout, rollback. Coordinates with Registry, Sandbox, Migration Engine (L4), and MQ (L5).
plugin manager
CẤU TRÚC NỘI BỘ:

  Plugin Manager (singleton)
  ├── LifecycleController
  │   ├── deploy(artifact)        → validate + register + schema migration
  │   ├── upgrade(plugin, ver)    → canary + new schema migration + broadcast
  │   ├── rollback(plugin)        → restore prev version + broadcast
  │   ├── enable(tenant, plugin)  → check deps + tenant migration + activate
  │   ├── disable(tenant, plugin) → check dependents + deactivate
  │   └── undeploy(plugin)        → deprecate 90d → manual cleanup
  │
  ├── DependencyResolver
  │   ├── resolve(plugin)                → topological sort dependencies
  │   ├── checkCanEnable(tenant, plugin) → all deps enabled for tenant?
  │   └── checkCanDisable(tenant, plugin)→ no dependents still enabled?
  │
  ├── CanaryController
  │   ├── start(plugin, 5%)       → route 5% traffic to new version
  │   ├── monitor(plugin)         → watch error rate, latency
  │   ├── promote(plugin)         → 5% → 25% → 100%
  │   └── abort(plugin)           → rollback, restore old version
  │
  └── BroadcastManager
      └── notify(event) → MQ crm.system (fanout) → all instances


DEPLOY PIPELINE:

  1. Dev push → Git
  2. CI/CD: tests + security scan + build artifact → Artifact Registry
  3. System Admin: approve version
  4. Plugin Manager.deploy():
     ├── Pull pre-built artifact (NO npm install trên production)
     ├── Validate checksum + manifest schema
     ├── Parse manifest → register routes, events, hooks, permissions
     ├── Run Schema Migrations (DDL) — CREATE TABLE IF NOT EXISTS (1 lần)
     ├── Instantiate Plugin Core singleton
     ├── Store metadata → Metadata DB (plugin_registry)
     └── Broadcast via MQ: "plugin.deployed: customer-care@2.0.0"
  5. All instances receive → hot-load plugin code
  6. Tenant Admin enable → Tenant Migrations (DML) per-tenant → activated


ENABLE/DISABLE PER-TENANT:

  enable(acme-123, customer-care):
  ① DependencyResolver: customer-care depends on customer-data → OK (enabled)
  ② Migration Engine: Schema DDL → SKIP (done). Tenant DML → RUN for acme-123
  ③ UPDATE tenant_plugins SET enabled=true, version='2.0.0'
  ④ Invalidate Tenant Context cache (Redis Pub/Sub → all instances)

  disable(acme-123, customer-care) (soft — mặc định):
  ① DependencyResolver: any plugin depends on care? → NO → proceed
  ② UPDATE tenant_plugins SET enabled=false. Tables + data: GIỮA NGUYÊN.
  ③ Invalidate cache → routes 404, events SKIP, hooks SKIP for acme-123


CANARY + AUTO-ROLLBACK:

  Upgrade 1.5 → 2.0:
  Phase 1: 5% tenants → monitor 1h
  Phase 2: 25% → monitor 1h
  Phase 3: 100% → stable
  AUTO-ROLLBACK: error >5% in 5min → restore v1.5 → PagerDuty + Slack

Plugin Registry — Metadata Store

Vai trò: Bộ nhớ trung tâm chứa metadata mọi plugin. Gateway cần biết URL thuộc plugin nào, Event Bus cần biết ai lắng nghe, Hook Registry cần biết ai hook — tất cả query Registry.
Role: Central store for all plugin metadata. Gateway needs to know which plugin owns a URL, Event Bus needs to know who is listening, Hook Registry needs to know who hooks — all query the Registry.
registry
REGISTRY CHỨA 5 MAP:

  Plugin Registry (singleton, in-memory — loaded from Metadata DB)
  │
  ├── pluginMap: Map<name, PluginEntry>
  │   └── { name, version, status, coreInstance, manifest, deployedAt }
  │
  ├── routeTable: Map<urlPattern, RouteEntry>
  │   ├── "POST /api/v1/sales" → { plugin: "customer-care",
  │   │     method: "createSale", permission: "sales:write" }
  │   ├── "GET /api/v1/customers" → { plugin: "customer-data",
  │   │     method: "listCustomers", permission: "customers:read" }
  │   └── ...
  │   L2 Route Matcher dùng routeTable này.
  │
  ├── eventMap: Map<eventName, Array<HandlerEntry>>
  │   ├── "sales.created" → [
  │   │     { plugin: "analytics", handler: "onSaleCreated", sync: true },
  │   │     { plugin: "automation", handler: "onSaleAsync",  sync: false }
  │   │   ]
  │   └── Event Bus dùng eventMap.
  │
  ├── hookMap: Map<hookName, Array<HookEntry>>
  │   ├── "filter:customerList" → [
  │   │     { provider: "customer-data", consumer: "customer-care",
  │   │       handler: "enrichWithSalesData", priority: 10 }
  │   │   ]
  │   └── Hook Registry dùng hookMap.
  │

  │   └── L2 Auth Guard dùng permissionMap.
  │

      ├── "customer-care" → ["customer-data"]

Sandbox Engine — Isolation & Resource Limits

Tại sao? Plugin code có thể có bug: vô hạn loop, memory leak, cố truy cập hệ thống. Sandbox ngăn chặn: plugin crash → graceful error, không crash process, không ảnh hưởng requests khác.
Why? Plugin code can have bugs: infinite loops, memory leaks, attempts to access the system. Sandbox prevents: plugin crash → graceful error, no process crash, no impact on other requests.
sandbox
CƠ CHẾ:

  Sandbox.execute(pluginCore, methodName, ctx, args):

  ① Load resource limits (from manifest):
     timeout: 5000ms (VIP: 10000ms), maxMemory: 50MB, maxQueries: 50

  ② Whitelist allowed APIs:
     ALLOWED: ctx.db, ctx.cache, ctx.logger, ctx.eventBus, ctx.services
     BLOCKED: process, fs, net, child_process, os, require(), global state

  ③ Execute trong VM/Worker Thread có giám sát:
     ┌─ VM isolate ──────────────────────────────────┐
     │  pluginCore[methodName](ctx, ...args)         │
     │                                                │
     │  Timer: 5000ms ──────────────────────┐        │
     │  Memory monitor: 50MB ───────────────┤        │
     │  Query counter: [n/50] ──────────────┤        │
     │                                      │        │
     │  Bất kỳ limit bị vượt ─────────────▶│        │
     └──────────────────────────────────────┘        │
                                             │
  ④ Khi vi phạm limit:                      │
     → Terminate NGAY LẬP TỨC               │
     → Log error + metrics: sandbox_limit_reached_total
     → Return: { error: 'SERVICE_UNAVAILABLE', fallback: true }
     → Request trả graceful error, process tiếp tục chạy bình thường

  ⑤ Khi thành công:
     → Return kết quả → Gateway → response cho user
     → Metrics: plugin_execution_duration_seconds


ERROR BOUNDARY:

  Request A → Plugin care: CRASH (unhandled throw)
  ├── Sandbox catches → log → return 500 "Plugin error"
  └── Process tiếp tục chạy

  Request B → Plugin data: Bình thường (không bị ảnh hưởng)
  Request C → Plugin care: Bình thường (mỗi request có context riêng)

Event Bus — Plugin Communication

Vai trò: Cho phép plugins giao tiếp MÀ KHÔNG biết nhau. Plugin A emit event → Event Bus deliver tới handlers — A không biết ai lắng nghe. Loose coupling.
Role: Allows plugins to communicate WITHOUT knowing each other. Plugin A emits event → Event Bus delivers to handlers — A does not know who is listening. Loose coupling.
event bus
2 CHẾ ĐỘ:

  ctx.eventBus.emit("sales.created", { saleId, amount })
  │
  ├── SYNC HANDLERS (in-process, cùng request)
  │   ├── Chạy tuần tự trong cùng process, cùng transaction
  │   ├── analytics.onSaleCreated(event) → sales_count ++
  │   ├── Cùng DB transaction (rollback = rollback handler)
  │   ├── Ảnh hưởng response time (handler chậm = request chậm)
  │   ├── Exception → catch bởi Sandbox → log + skip
  │   └── Dùng cho: data consistency, realtime aggregation
  │
  └── ASYNC HANDLERS (distributed, qua MQ)
      ├── Publish message to RabbitMQ crm.events (topic exchange)
      │   routing key: "sales.created"
      │   payload: { saleId, tenantId, requestId, traceId }
      │   priority: VIP=10, Premium=5, Basic=1
      │
      ├── Workers consume từ bound queues:
      │   notification → email, push
      │   search.index → ES update
      │   webhook → external URLs
      │   analytics → batch reports
      │
      ├── KHÔNG ảnh hưởng response time (fire & forget)
      ├── Retry 3x exponential backoff → DLQ
      └── Carries traceId → linked spans in Jaeger


SCOPED EVENT BUS — Tenant-aware filter:

  ScopedEventBus (per-request) wraps Event Bus singleton:
  ├── emit(): auto-attach tenantId, userId, requestId, traceId
  └── Filter: sync handler "analytics.onSaleCreated":
       ACME enable analytics?  YES → call
       GLOBEX enable analytics? NO → SKIP

  Plugin đăng ký handler 1 lần (system-level).
  ScopedEventBus filter PER-REQUEST dựa trên tenant enabledPlugins.


EVENT vs HOOK:

  Event: Thông báo "đã xảy ra". Read-only payload. Async possible. Loose coupling.
  Hook:  Can thiệp "sắp xảy ra". MODIFY payload. Luôn sync. Tighter coupling.

Hook Registry — Extension Points

Vai trò: Quản lý "điểm mở rộng" giữa plugins. Plugin A CỐ TÌNH tạo hook points — cho phép plugins KHÁC can thiệp vào luồng xử lý mà KHÔNG sửa code gốc.
Role: Manages "extension points" between plugins. Plugin A DELIBERATELY creates hook points — allowing OTHER plugins to intervene in the processing flow WITHOUT modifying the original code.
hooks
3 LOẠI HOOK:

  before:  Chạy TRƯỚC thao tác. Validate, reject, enrich.
           customer-data provides "before:customerSave"
           automation consumes → validateBeforeSave() → nếu invalid → cancel

  after:   Chạy SAU thao tác. Trigger side effects.
           customer-data provides "after:customerSave"
           automation consumes → checkRulesAfterSave() → trigger email

  filter:  TRANSFORM data đang xử lý. Nhận → sửa → trả.
           customer-data provides "filter:customerList"
           customer-care consumes → enrichWithSalesData(customers)
             → thêm lastSaleAmount vào mỗi customer → return modified


EXECUTION PIPELINE:

  Plugin customer-data code:
    let customers = await ctx.db.query('customers').where(filters);
    customers = await ctx.hooks.run('filter:customerList', customers);
    //                   ▲
    //  Hook Registry: lookup hookMap → filter by enabledPlugins
    //  → sort by priority (thấp = chạy trước)
    //  → pipe data qua từng consumer handler

  Input data
    │
    ▼
  ┌────────────────────────────────────────────────────┐
  │ Hook Registry.run("filter:customerList", data)     │
  │                                                     │
  │ ① care.enrichWithSalesData(data)    (priority 10)  │
  │    → data + lastSaleAmount                         │
  │         │                                           │
  │ ② analytics.enrichWithScore(data)   (priority 20)  │
  │    → data + engagementScore                        │
  │    Nếu analytics KHÔNG enabled → SKIP ②              │
  └─────────┬──────────────────────────────────────────┘
            ▼
  Output data (enriched)


AN TOÀN:
  ├── Consumer chạy trong CÙNG Sandbox, cùng resource limits
  ├── Consumer throw → catch, log, skip (KHÔNG cancel thao tác gốc)
  ├── Consumer chạy lâu → đếm vào timeout chung
  └── Consumer query DB → đếm vào maxQueries chung

Plugin Cores — Business Logic Code

Vai trò: Chứa code business logic thực tế — CRUD, validation, business rules. Mỗi core là singleton stateless: instantiate 1 lần, dùng chung cho MỌI tenant. State per-tenant nằm ở database (L4/L5).
Role: Contains the actual business logic code — CRUD, validation, business rules. Each core is a stateless singleton: instantiated once, shared across ALL tenants. Per-tenant state lives in the database (L4/L5).
plugin cores
CẤU TRÚC THƯ MỤC:

  customer-care/
  ├── manifest.json    ← metadata: deps, routes, events, hooks, perms
  ├── core.js          ← business logic (stateless singleton)
  ├── routes.js        ← route handler definitions
  ├── events.js        ← event handler implementations
  ├── hooks.js         ← hook provider/consumer implementations
  ├── migrations/
  │   ├── schema/      ← DDL (system-wide, run once)
  │   └── tenant/      ← DML (per-tenant, run on enable)
  └── tests/


CORE.JS — Code example:

  class CustomerCareCore {

    async createSale(ctx, input) {
      // 1. Validate
      const validated = validateSaleInput(input);

      // 2. Hook: before:saleCreate
      await ctx.hooks.run('before:saleCreate', validated);

      // 3. Check customer (reads table from customer-data plugin)
      const cust = await ctx.db.query('customers')
        .where({ id: validated.customerId }).first();
      // ctx.db auto-inject tenant_id

      if (!cust) throw new BusinessError('CUSTOMER_NOT_FOUND');

      // 4. Insert sale
      const sale = await ctx.db.insert('sales', {
        customerId: validated.customerId,
        title: validated.title,
        amount: validated.amount,
        status: 'new'
      }); // tenant_id auto-injected

      // 5. Update customer stats
      await ctx.db.update('customers',
        { id: validated.customerId },
        { totalSales: cust.totalSales + 1 });

      // 6. Hook: after:saleCreate
      await ctx.hooks.run('after:saleCreate', sale);

      // 7. Event: sales.created
      await ctx.eventBus.emit('sales.created', {
        saleId: sale.id, amount: sale.amount });

      ctx.logger.info('Sale created', { saleId: sale.id });
      return sale;
    }
  }

Context Building — Chuẩn bị thế giới cho PluginContext Building — Preparing the World for Plugins

Vai trò: Giai đoạn GIỮA Gateway và Plugin execution. Tạo Tenant Context (tenant là ai, dùng gì) + Execution Context (request cần gì). Plugin CHỈ nhận Execution Context.
Role: The stage BETWEEN Gateway and Plugin execution. Creates Tenant Context (who the tenant is, what they use) + Execution Context (what the request needs). Plugins ONLY receive the Execution Context.
context building
TENANT CONTEXT MANAGER:

  getOrBuild(tenantId):
  ① LRU cache hit (~95%):  return cached           ~0.01ms
  ② Cache miss → Metadata DB:
     SELECT * FROM tenants WHERE id = $id           ~2ms
     SELECT * FROM tenant_plugins WHERE enabled=true ~2ms
  ③ Assemble:
     {
       tenantId: "acme-123", tier: "premium",
       dbPool: "shared",  ← hoặc "vip-initech"
       enabledPlugins: Map {
         "customer-data": { version, config, coreRef: singleton },
         "customer-care": { version, config, coreRef: singleton }
       },
       quotas: { rateLimit: 500, storage: "50GB", maxConns: 20 }
     }
  ④ Store LRU (max 1000, evict 2h)
  Invalidation: Redis Pub/Sub → all instances evict


EXECUTION CONTEXT BUILDER (per-request, ~2-5ms):

  build(tenantCtx, jwtClaims, requestInfo):
  ① Acquire DB conn: PoolRouter(tenantCtx.dbPool) → conn #14     ~1ms
     SET app.current_tenant = 'acme-123'                          ~0.5ms
  ② Wrap DB: ctx.db = TenantScopedDB(conn) + Query Interceptor
  ③ Cache:   ctx.cache = TenantScopedCache(prefix "t:acme-123")
  ④ Logger:  ctx.logger = ScopedLogger({requestId, tenantId, userId})
  ⑤ Events:  ctx.eventBus = ScopedEventBus(filter by enabledPlugins)
  ⑥ Services: ctx.services = { email, storage, notification, search }
  ⑦ Config:  ctx.pluginConfig = tenant-specific config for plugin

  After response: RESET tenant → release conn → GC context

  Plugin thấy:  ctx.db, ctx.cache, ctx.logger, ctx.eventBus, ctx.services
  Plugin KHÔNG thấy: pool, tenant_id injection, RLS, cache prefix,
                     log scope, routing, tracing, sandbox limits

Shared Services — Utility dùng chungShared Services — Common Utilities

Vai trò: Singletons cung cấp utility cho mọi plugin. Plugin gọi qua ctx.services — không cần biết S3 bucket nào, Redis cluster nào, MQ exchange nào. Tất cả tenant-aware.
Role: Singletons providing utilities to every plugin. Plugins call via ctx.services — no need to know which S3 bucket, Redis cluster, or MQ exchange. All are tenant-aware.
shared services
① Email Service
  ctx.services.email.send({ to, template, data })
  → Render template → publish to MQ email queue (async, no block)
  → Worker → SMTP provider (SendGrid/SES) → retry 3x → DLQ

② Storage Service
  ctx.services.storage.upload(file, path)
  ctx.services.storage.getUrl(path)
  → S3/MinIO wrapper, auto-prefix /tenants/{tenantId}/{path}
  → Plugin: storage.upload(file, "contracts/abc.pdf")
  → Actual: s3://bucket/tenants/acme-123/contracts/abc.pdf
  → Tenant CANNOT access files of other tenants
  → Signed URLs for temp access (15min TTL)

③ Notification Service
  ctx.services.notification.send({ userId, type, payload })
  → WebSocket (real-time) + Push (FCM/APNs) + In-app (DB stored)
  → Publish to MQ notification queue → worker delivers

④ Search Service
  ctx.services.search.index(entity, data)
  ctx.services.search.query(entity, searchText, filters)
  → Elasticsearch wrapper, per-tenant indices: {tenantId}-{entity}
  → Index: async via MQ search.index queue (~1-2s lag)
  → Query: sync direct to ES (tenantId filter enforced)

⑤ Scheduler Service
  ctx.services.scheduler.schedule({ cron, task, data })
  ctx.services.scheduler.delay({ after, task, data })
  → Cron: "0 9 * * 1" (mỗi thứ Hai 9h)
  → Delay: "sau 30 phút gửi follow-up"
  → Tenant-aware, stored in DB, polls → MQ → worker

⑥ MQ Publisher Service
  ctx.services.mq.publish({ exchange, routingKey, payload, priority })
  → RabbitMQ wrapper, auto-attach tenantId, requestId, traceId
  → VIP=10, Premium=5, Basic=1 priority
  → Plugin thường KHÔNG gọi trực tiếp (Event Bus gọi internally)

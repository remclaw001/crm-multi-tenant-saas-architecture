Security — Defense in DepthSecurity — Defense in Depth

Nguyên tắc: Security không phải 1 bức tường — nó là nhiều lớp chồng lên nhau. Mỗi layer có defense riêng. Attacker phá được 1 lớp → vẫn còn nhiều lớp phía sau. Plugin developer KHÔNG CẦN tự implement security — hệ thống enforce tự động.
Principle: Security is not a single wall — it is many layers stacked on top of each other. Each layer has its own defense. An attacker breaking through one layer → still faces many more layers behind. Plugin developers do NOT need to implement security themselves — the system enforces it automatically.

Tenant Isolation — 3-Layer ProtectionTenant Isolation — 3-Layer Protection

Zero cross-tenant data leak là yêu cầu TUYỆT ĐỐI. Một tenant KHÔNG BAO GIỜ nhìn thấy data của tenant khác. Hệ thống enforce tự động ở 3 tầng — plugin developer KHÔNG THỂ bypass dù cố ý hay vô ý.
Zero cross-tenant data leaks are an ABSOLUTE requirement. A tenant will NEVER see another tenant's data. The system enforces this automatically at 3 layers — plugin developers CANNOT bypass it whether intentionally or accidentally.
3-layer tenant isolation
LAYER 1: QUERY INTERCEPTOR (L4 — application level)

  Plugin code:  ctx.db.query('customers').where({ status: 'active' })

  Query Interceptor transforms:
  SELECT * FROM customers WHERE status = 'active'
  →
  SELECT * FROM customers WHERE status = 'active' AND tenant_id = 'acme-123'

  TenantInjector:
  ├── AUTO-inject tenant_id vào mọi SELECT, INSERT, UPDATE, DELETE
  ├── Plugin KHÔNG CẦN (và KHÔNG THỂ) chỉ định tenant_id
  ├── INSERT: auto-add tenant_id = $current vào column list
  └── Cross-table JOIN: validate BOTH tables have tenant_id filter

  QueryValidator (6 safety checks):
  ├── ① Has tenant_id in WHERE?           → guaranteed by injector
  ├── ② No raw SQL without tenant filter? → block if detected
  ├── ③ No SELECT * FROM sys_tables?      → block system table access
  ├── ④ No ALTER/DROP/TRUNCATE?           → block DDL from plugins
  ├── ⑤ No cross-schema access?           → block pg_catalog etc
  └── ⑥ Parameter count matches?          → prevent SQL injection


LAYER 2: POSTGRESQL RLS (L5 — database level)

  Backup layer. Nếu Query Interceptor có bug → RLS vẫn block:

  CREATE POLICY tenant_isolation ON customers
    USING (tenant_id = current_setting('app.current_tenant'));

  ├── Connection: SET app.current_tenant = 'acme-123' (before queries)
  ├── RLS evaluated by PostgreSQL engine, KHÔNG bypass được
  ├── Nếu interceptor quên inject → RLS vẫn filter đúng tenant
  ├── Nếu interceptor inject SAI tenant → RLS override
  └── Apply cho MỌI table có tenant_id column

  Double safety: 2 systems independent, cùng enforce 1 rule.
  Probability cả 2 fail cùng lúc = gần bằng 0.


LAYER 3: AUDIT + ALERT (L7 — detection level)

  Nếu (impossible nhưng) cross-tenant access xảy ra:

  ├── Audit Trail ghi: user_id, tenant_id, resource accessed
  ├── Anomaly detection: query patterns bất thường
  │   → Tenant A user query data with Tenant B tenant_id in result
  │   → IMMEDIATE ALERT: PagerDuty P1
  ├── Forensic: full trace available (request_id → Jaeger → log → audit)
  └── Response: auto-block user, investigate

Error Handling — Resilience FrameworkError Handling — Resilience Framework

Vai trò: Cung cấp framework thống nhất để MỌI layer xử lý lỗi nhất quán: phân loại lỗi (permanent vs transient), pattern phục hồi (retry, circuit breaker, bulkhead, fallback), và error boundary cho plugins. Plugin developer throw error → framework tự phân loại + response đúng HTTP status.
Role: Provides a unified framework so that EVERY layer handles errors consistently: error classification (permanent vs transient), recovery patterns (retry, circuit breaker, bulkhead, fallback), and error boundaries for plugins. Plugin developer throws an error → framework automatically classifies it + responds with the correct HTTP status.

RESILIENCE PATTERNS:

  ┌─ CIRCUIT BREAKER (protect downstream khi service lỗi) ──────────────┐
  │                                                                        │
  │  Áp dụng cho: DB Pool, Redis, Elasticsearch, External APIs           │
  │                                                                        │
  │  ┌──────────┐   fail > threshold   ┌──────────┐   30s timer   ┌─────┐│
  │  │  CLOSED  │ ──────────────────▶  │   OPEN   │ ──────────▶  │HALF ││
  │  │ (normal) │                      │(fast-fail)│              │OPEN ││
  │  └──────────┘  ◀────────────────── └──────────┘              └──┬──┘│
  │       ▲        success in half-open      ▲                      │    │
  │       │                                  │     fail in half-open│    │
  │       └──────────────────────────────────┴──────────────────────┘    │
  │                                                                        │
  │  DB Pool exhausted → Circuit OPEN → 503 fast-fail (no wait)         │
  │  30s later → HALF-OPEN → try 1 request                              │
  │  Success → CLOSED (resume normal). Fail → OPEN again (30s more)    │
  │                                                                        │
  │  Benefit: không queue 1000 requests chờ DB recover              │
  │  → fast-fail + Retry-After cho client biết khi nào retry             │
  └────────────────────────────────────────────────────────────────────────┘

  ┌─ RETRY WITH BACKOFF (tự phục hồi transient errors) ────────────────┐
  │                                                                        │
  │  Áp dụng cho: MQ publish, ES index, External API calls              │
  │                                                                        │
  │  Attempt 1: fail → wait 5s                                           │
  │  Attempt 2: fail → wait 30s                                          │
  │  Attempt 3: fail → wait 2min                                         │
  │  All 3 fail → DLQ (Dead Letter Queue) → manual investigation       │
  │                                                                        │
  │  Jitter: random ±20% delay → prevent thundering herd                │
  │  CHỈ retry SystemError (transient). BusinessError KHÔNG retry.        │
  └────────────────────────────────────────────────────────────────────────┘

  ┌─ BULKHEAD (isolate failure domains) ─────────────────────────────────┐
  │                                                                        │
  │  Per-tenant connection caps (Pool Manager):                          │
  │  ├── Shared Pool: 200 total connections                              │
  │  ├── Per-tenant cap: 20 connections max                              │
  │  │   → 1 tenant spike KHÔNG starve other tenants                    │
  │  └── VIP: own pool, own cap (no shared resource contention)         │
  │                                                                        │
  │  Per-plugin Sandbox limits:                                          │
  │  ├── timeout: 5000ms, maxMemory: 50MB, maxQueries: 50              │
  │  │   → 1 plugin bug KHÔNG affect other plugins                     │
  │  └── Error boundary: plugin crash → only that request fails        │
  │                                                                        │
  │  MQ queue isolation:                                                  │
  │  ├── Separate queues per concern (notification, search, webhook)    │
  │  │   → notification backlog KHÔNG block search indexing             │
  │  └── VIP priority queues (10/5/1)                                   │
  └────────────────────────────────────────────────────────────────────────┘

  ┌─ FALLBACK (degraded nhưng vẫn hoạt động) ───────────────────────────┐
  │                                                                        │
  │  Redis down     → fallback to Metadata DB (chậm hơn ~5ms/call)     │
  │  ES down        → fallback to DB LIKE query (chậm, basic search)   │
  │  MQ down        → log to disk queue → replay khi MQ recover        │
  │  Cache miss     → rebuild from DB → cache for next request         │
  │  Plugin crash   → 500 graceful error, process vẫn chạy            │
  │  CDN down       → origin server serve static files (chậm hơn)     │
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘


ERROR BOUNDARY LAYERS (cascade protection):

  Request →
  ├── L2 Gateway:  reject early (auth, rate limit) → 4xx fast
  ├── L3 Sandbox:  catch plugin crash → 500 graceful, process safe
  ├── L4 Circuit:  catch infra fail → 503 fast-fail, no cascade
  ├── L5 Retry:    catch transient → retry 3x → DLQ
  └── L7 Alert:    detect pattern → alert ops team

  Mỗi layer catch lỗi CỦA NÓ. Không để lỗi leak ra ngoài.
  Plugin crash ≠ process crash. DB down ≠ toàn bộ hệ thống down.

Config Management — 3-Level ConfigurationConfig Management — 3-Level Configuration

Vai trò: Quản lý cấu hình ở 3 cấp: Global (toàn hệ thống), Tier (per-plan), Tenant (per-tenant). Config cascade: Tenant override Tier, Tier override Global. Hỗ trợ hot reload (thay đổi mà không restart) và feature flags (bật/tắt feature không cần deploy).
Role: Manages configuration at 3 levels: Global (system-wide), Tier (per-plan), Tenant (per-tenant). Config cascade: Tenant overrides Tier, Tier overrides Global. Supports hot reload (changes without restart) and feature flags (enable/disable features without deploying).

Dependency Management — Plugin Dependency GraphDependency Management — Plugin Dependency Graph

Vai trò: Quản lý dependency giữa plugins: A depends on B → không enable A nếu B chưa enable, không disable B nếu A vẫn enable. Thuộc L6 vì XUYÊN TẦNG: enforcement diễn ra ở L3 (Plugin Manager), L4 (Migration Engine), và L1 (UI hiển thị).
Role: Manages dependencies between plugins: A depends on B → cannot enable A if B is not yet enabled, cannot disable B if A is still enabled. Belongs to L6 because it is CROSS-LAYER: enforcement occurs at L3 (Plugin Manager), L4 (Migration Engine), and L1 (UI display).

ENABLE ENFORCEMENT (Plugin Manager — L3):

  Tenant ACME muốn enable "automation":
  ┌────────────────────────────────────────────────────────────────────┐
  │                                                                    │
  │  DependencyResolver.checkCanEnable("acme-123", "automation"):     │
  │                                                                    │
  │  deps = ["customer-data", "customer-care"]                        │

  │  ✗ customer-care enabled for ACME?  → NO                         │
  │                                                                    │
  │  → REJECT: "Cannot enable automation: customer-care not enabled" │
  │  → Response: { missingDeps: ["customer-care"],                    │
  │               suggestion: "Enable customer-care first" }          │
  │                                                                    │
  │  Tương tác L1: Admin Console hiển thị:                              │

MIGRATION DEPENDENCY (Migration Engine — L4):

  ┌────────────────────────────────────────────────────────────────────┐
  │                                                                    │
  │  Schema DDL: customer-care creates table "sales"                  │
  │              with FOREIGN KEY (customer_id) → customers.id        │
  │                                                                    │
  │  Enforcement:                                                     │
  │  ├── DDL run ORDER respects dependency graph                      │
  │  │   customer-data DDL first → THEN customer-care DDL            │
  │  ├── Hard purge (DELETE rows): respect FK constraints             │
  │  │   → Cannot purge customer-data if sales rows reference it     │
  │  └── Advisory lock: prevent race condition                        │
  │      (2 tenants enable same plugin → serialize via pg_advisory)  │
  │                                                                    │
  └────────────────────────────────────────────────────────────────────┘


TẠI SAO L6 (KHÔNG PHẢI L3)?

  Dependency enforcement xuyên NHIỀU layer:
  ├── L1: Admin UI hiển thị dep warnings, suggest enable order
  ├── L3: Plugin Manager check deps trước enable/disable
  ├── L4: Migration Engine respect dep order cho DDL/DML
  └── L7: Alert nếu orphaned state detected (edge case)

  → Không thuộc riêng L3 hay L4. Nó là cross-cutting concern.

Layer 4 — Data Access

Data Access Layer

5 thực thể cốt lõi: Query Builder + Pipeline, Pool Manager, Cache Manager, Migration Engine, DIP Boundary — cấu trúc nội tại, liên kết xuyên tầng, và hành vi trong mỗi request.

Bản đồ tổng quan L4 — 5 thực thể và vị trí

Query Builder + Pipeline — Cấu trúc nội tại

query builder pipeline anatomy
┌─ QUERY BUILDER + PIPELINE ─────────────────────────────────────────────────┐
│                                                                              │
│  Được tạo khi nào?                                                        │
│  → Khi build Execution Context (mỗi request)                                │
│  → TenantScopedDB (implements IDbContext) wrap DB connection                 │
│  → Inject ISQLDialect + ITenantStrategy qua DI (constructor)                │
│  → Plugin nhận ctx.db — KHÔNG BIẾT DB nào bên dưới                          │
│                                                                              │
│  Input:  raw DB connection + tenantId + ISQLDialect + ITenantStrategy       │
│  Output: TenantScopedDB — interface mà plugin thấy (ctx.db)                │
│                                                                              │
│  ┌─ Thành phần bên trong ─────────────────────────────────────────────────┐ │
│  │                                                                        │ │
│  │  CrmQueryBuilder «per-query»                                         │ │
│  │  │                                                                     │ │
│  │  │  Fluent API — plugin gọi chain methods:                            │ │
│  │  │  ctx.db.query('sales')                                              │ │
│  │  │    .where({ status: 'new' })                                        │ │
│  │  │    .select(['id', 'title', 'amount'])                               │ │
│  │  │    .orderBy('created_at', 'desc')                                   │ │
│  │  │    .limit(20)                                                       │ │
│  │  │    .execute()     ← terminal method triggers pipeline               │ │
│  │  │                                                                     │ │
│  │  │  .toAST() → QueryAST (value object, DB-agnostic):                  │ │
│  │  │  { table: 'sales', operation: 'SELECT',                            │ │
│  │  │    select: ['id','title','amount'],                                │ │
│  │  │    where: [{ field:'status', op:'=', value:'new' }],               │ │
│  │  │    orderBy: [{ field:'created_at', dir:'desc' }],                  │ │
│  │  │    limit: 20 }                                                      │ │
│  │  │                                                                     │ │
│  │  │  Plugin KHÔNG viết SQL. Builder tạo AST typed → structural safety.  │ │
│  │                                                                        │ │
│  │                                                                        │ │
│  │  ITenantStrategy.injectTenant(ast, tenantId)                          │ │
│  │  │                                                                     │ │
│  │  │  Duyệt AST → thêm tenant_id node vào where conditions             │ │
│  │  │                                                                     │ │
│  │  │  SELECT:                                                            │ │
│  │  │    ast.where prepend:                                               │ │
│  │  │    { field: 'tenant_id', op: '=', value: 'acme-123' }             │ │
│  │  │                                                                     │ │
│  │  │  INSERT:                                                            │ │
│  │  │    ast.insertData prepend: tenant_id: 'acme-123'                    │ │
│  │  │                                                                     │ │
│  │  │  UPDATE / DELETE:                                                   │ │
│  │  │    ast.where prepend: tenant_id = 'acme-123'                        │ │
│  │  │                                                                     │ │
│  │  │  JOIN:                                                              │ │
│  │  │    inject tenant_id cho MỌI joined table (chống cross-tenant leak) │ │
│  │  │                                                                     │ │
│  │  │  PostgresRLSStrategy: inject AST + SET app.current_tenant (RLS)    │ │
│  │  │  MySQLTenantStrategy: inject AST + SET @app_tenant session var     │ │
│  │  │  SQLiteTenantStrategy: inject AST only (test)               │ │
│  │  │                                                                     │ │
│  │  │  Plugin KHÔNG THỂ viết query mà không có tenant_id.                │ │
│  │  │  tenant_id lấy từ ExecutionContext, không phải plugin code.         │ │
│  │                                                                        │ │
│  │                                                                        │ │
│  │  QueryValidator.validate(ast)                                         │ │
│  │  │                                                                     │ │
│  │  │  6 safety checks trên AST (SAU inject, TRƯỚC generate SQL):        │ │
│  │  │                                                                     │ │
│  │  │  ┌────────────────────────────────────────────────────────────┐     │ │
│  │  │  │  ☐ Query có đúng tenant_id sau khi inject?         PASS   │     │ │
│  │  │  │  ☐ Parameterized? (AST values không interpolate)   PASS   │     │ │
│  │  │  │  ☐ Table nằm trong plugin's allowedTables?         PASS   │     │ │
│  │  │  │  ☐ Không có DDL operations (ALTER/DROP/TRUNCATE)?  PASS   │     │ │
│  │  │  │  ☐ Không query system tables (pg_catalog)?         PASS   │     │ │
│  │  │  │  ☐ JOIN tables cũng có tenant_id filter?           PASS   │     │ │
│  │  │  └────────────────────────────────────────────────────────────┘     │ │
│  │  │                                                                     │ │
│  │  │  Bất kỳ check nào FAIL → SecurityError, query BLOCKED.             │ │
│  │                                                                        │ │
│  │                                                                        │ │
│  │  QueryCounter «per-request»                                           │ │
│  │  │                                                                     │ │
│  │  │  Mỗi query đi qua → counter++                                      │ │
│  │  │  Sandbox Engine set giới hạn: maxQueries = 50                       │ │
│  │  │                                                                     │ │
│  │  │  counter = 49 → WARNING log                                        │ │
│  │  │  counter = 50 → BLOCK, throw QueryLimitExceeded                    │ │
│  │  │                                                                     │ │
│  │  │  Ngăn plugin chạy N+1 queries hoặc infinite loops.                │ │
│  │                                                                        │ │
│  │                                                                        │ │
│  │  ISQLDialect.generateSelect(ast)                                      │ │
│  │  │                                                                     │ │
│  │  │  AST → parameterized SQL per DB type:                               │ │
│  │  │                                                                     │ │
│  │  │  PostgresDialect: SELECT ... WHERE tenant_id = $1 AND status = $2 │ │
│  │  │  MySQLDialect:    SELECT ... WHERE tenant_id = ?  AND status = ?  │ │
│  │  │  SQLiteDialect:   SELECT ... WHERE tenant_id = ?  AND status = ?  │ │
│  │  │                                                                     │ │
│  │  │  output: { text: string, params: any[] }                            │ │
│  │  │  100% parameterized. Zero interpolation. SQL injection = impossible. │ │
│  │                                                                        │ │
│  │                                                                        │ │
│  │  QueryLogger                                                         │ │
│  │  │                                                                     │ │
│  │  │  Mỗi query: ghi log với context                                     │ │
│  │  │  { tenantId, requestId, plugin, table, operation,                   │ │
│  │  │    duration_ms, rows_affected }                                     │ │
│  │  │                                                                     │ │
│  │  │  Slow query (> 100ms) → WARNING                                     │ │
│  │  │  Very slow (> 500ms) → ALERT to ops                                 │ │
│  │                                                                        │ │
│  └────────────────────────────────────────────────────────────────────────┘ │
│                                                                              │
│  LUỒNG QUA PIPELINE (7 bước):                                              │
│                                                                              │
│  Plugin gọi ctx.db.query('sales').where({ status: 'new' }).execute()        │
│       │                                                                      │
│       ▼                                                                      │
│  ① CrmQueryBuilder.toAST() → QueryAST (DB-agnostic)                        │
│       │                                                                      │
│       ▼                                                                      │
│  ② ITenantStrategy.injectTenant(ast, tenantId) → AST with tenant_id        │
│       │                                                                      │
│       ▼                                                                      │
│  ③ QueryValidator.validate(ast) → 6 checks → PASS                          │
│       │                                                                      │
│       ▼                                                                      │
│  ④ QueryCounter.increment() → [3/50]                                        │
│       │                                                                      │
│       ▼                                                                      │
│  ⑤ ISQLDialect.generateSelect(ast) → { text, params }                      │
│       │                                                                      │
│       ▼                                                                      │
│  ⑥ Forward tới DB connection (từ Pool Manager)                               │
│       │                                                                      │
│       ▼                                                                      │
│  ⑦ QueryLogger.log({ table: sales, op: SELECT, duration: 5ms })            │
│       │                                                                      │
│       ▼                                                                      │
│  Return result set cho plugin                                                │
│                                                                              │
│  Pipeline overhead (①→⑤⑦): ~0.07ms (negligible vs ⑥ DB ~3-15ms)           │
└──────────────────────────────────────────────────────────────────────────────┘

3 tầng bảo vệ — Builder + Validator + RLS

Defense in Depth: Nếu 1 tầng có bug, 2 tầng còn lại vẫn chặn (PostgreSQL). MySQL chỉ 2 tầng (no RLS) → stricter validation.
3-layer security
    Plugin: ctx.db.query('sales').where({ status: 'new' }).execute()
         │
    ═════╪════════════════════════════════════════════════════════
    TẦNG 1: CrmQueryBuilder + ITenantStrategy (Application Level)
         │
         ├── Plugin KHÔNG viết SQL → không thể inject
         │   Builder tạo AST typed (not string) → structural safety
         │
         ├── ITenantStrategy.injectTenant(ast, tenantId):
         │   Duyệt AST → thêm tenant_id node. tenant_id lấy từ
         │   ExecutionContext (set bởi Gateway), không phải plugin.
         │
         ├── Plugin cố set tenant_id = 'other-tenant' trong where?
         │   → ITenantStrategy LUÔN prepend tenant_id = ctx.tenantId
         │   → QueryValidator verify: mismatch → BLOCK
         │
         ├── Plugin cố truy cập table không thuộc manifest?
         │   → QueryValidator check allowedTables → BLOCK
         │
    ═════╪════════════════════════════════════════════════════════
    TẦNG 2: QueryValidator (Validation Level)
         │
         │  6 rules, kiểm tra SAU inject, TRƯỚC generate SQL:
         │
         │  ┌──────────────────────────────────────────────────┐
         │  │ 1. tenant_id present in WHERE?          REQUIRED │
         │  │ 2. Table in allowedTables?              REQUIRED │
         │  │ 3. No DDL (ALTER/DROP)?                 REQUIRED │
         │  │ 4. No system tables (pg_catalog)?       REQUIRED │
         │  │ 5. All values parameterized?            REQUIRED │
         │  │ 6. JOIN tables have tenant filter?      REQUIRED │
         │  └──────────────────────────────────────────────────┘
         │  ANY FAIL → SecurityError, query BLOCKED, log alert.
         │
    ═════╪════════════════════════════════════════════════════════
    TẦNG 3: PostgreSQL RLS (Database Level) — nếu dialect hỗ trợ
         │
         │  CREATE POLICY tenant_iso ON sales
         │    USING (tenant_id = current_setting('app.current_tenant'));
         │
         │  ITenantStrategy.configureConnection():
         │    PostgresRLSStrategy: SET app.current_tenant = 'acme-123'
         │    MySQLTenantStrategy: SET @app_tenant = 'acme-123' (no RLS)
         │    SQLiteTenantStrategy: N/A (single-process)
         │
         │  PostgreSQL: RLS evaluated by DB engine. Kể cả app bug
         │  → RLS vẫn filter rows. 3 tầng bảo vệ.
         │  MySQL: chỉ 2 tầng → enforce stricter QueryValidator rules.
         │
    ═════╪════════════════════════════════════════════════════════
         ▼
    Query execution → Pool Manager → DB

Pool Manager — Cấu trúc nội tại

pool manager anatomy
┌─ POOL MANAGER ─── Singleton, tạo khi bootstrap ──────────────────────────┐
│                                                                            │
│  Trách nhiệm:                                                          │
│  1. Quản lý nhiều connection pool (metadata, shared, VIP)                 │
│  2. Route request tới đúng pool dựa trên tenant context                   │
│  3. Connection lifecycle: acquire → configure → release → health check    │
│  4. Tạo / hủy VIP pool khi tenant upgrade / downgrade                     │
│                                                                            │
│                                                                            │
│  ┌─ Thành phần bên trong ──────────────────────────────────────────────┐  │
│  │                                                                      │  │

│  │  │                                                                   │  │

│  │  │                                                                   │  │

│  │  │  │ Pool Name         │ Pool Instance                         │  │  │

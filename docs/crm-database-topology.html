Metadata Database — Schema chi tiếtMetadata Database — Detailed Schema

Metadata DB là "bộ não" của hệ thống. Chứa thông tin về tenants, plugins, configs — nhưng KHÔNG chứa business data (customers, sales...). Mọi app instance, mọi worker đều đọc DB này để biết "tenant nào dùng gì".
Metadata DB is the system's "brain." Contains information about tenants, plugins, configs — but does NOT contain business data (customers, sales...). Every app instance, every worker reads this DB to know "which tenant uses what."

Ai đọc/ghi Metadata DB?Who Reads/Writes Metadata DB?

metadata access patterns
THÀNH PHẦN                     TABLE              ĐỌC/GHI   KHI NÀO

━━━ Gateway Layer ━━━

Tenant Resolver               tenants            READ      Mỗi request (cache miss)
                                                           subdomain → tenant_id

━━━ Business Layer ━━━

Tenant Context Manager        tenants            READ      Build tenant context (miss)
                              tenant_plugins     READ      Lấy enabled plugins + config

Plugin Manager (bootstrap)    plugin_registry    READ      Load all active plugins
Plugin Manager (deploy)       plugin_registry    WRITE     Deploy new plugin version

━━━ Admin Operations ━━━

System Admin Console          tenants            R/W       CRUD tenants
                              plugin_registry    R/W       Deploy/undeploy plugins
                              system_config      R/W       System settings

Tenant Admin Console          tenant_plugins     R/W       Enable/disable plugins
                              webhooks           R/W       Manage webhooks

━━━ Cross-cutting ━━━

Audit Logger                  audit_log          WRITE     Mỗi write operation
                                                           (append-only, heavy writes)

Config Manager                system_config      READ      Hot reload configs


TRAFFIC PATTERN:

  ┌────────────────────────────────────────────────────────────┐
  │  tenants:          LOW read (cached in Redis + LRU)         │
  │  tenant_plugins:   LOW read (cached in Tenant Context LRU)  │
  │  plugin_registry:  VERY LOW (only bootstrap + deploy)       │
  │  audit_log:        HIGH write (every mutation = 1 insert)   │
  │  webhooks:         LOW read (by webhook worker per event)   │
  │  system_config:    VERY LOW (cached, hot reload interval)   │
  │                                                            │
  │  Metadata DB chủ yếu chịu write load từ audit_log         │
  │  → Partition + archive giải quyết vấn đề này               │
  └────────────────────────────────────────────────────────────┘

Luồng Query — Từ plugin code đến diskQuery Flow — From Plugin Code to Disk

query flow through layers
Plugin code gọi:  ctx.db.query('sales').where({ status: 'new' }).limit(10).execute()


  L3 BUSINESS LOGIC ── Plugin Core
  │
  │  ctx.db là TenantScopedDB (implements IDbContext)
  │  Plugin không biết DB nào, dialect nào, hay tenant isolation bên dưới
  │
  ▼
  L4 DATA ACCESS ── Query Builder Pipeline (7 bước)
  │
  │  ① CrmQueryBuilder.toAST()
  │  │  Fluent chain → QueryAST (DB-agnostic value object):
  │  │  { table:'sales', op:'SELECT',
  │  │    where:[{field:'status', op:'=', value:'new'}], limit:10 }
  │  │
  │  ② ITenantStrategy.injectTenant(ast, 'acme-123')
  │  │  Duyệt AST → prepend tenant_id node:
  │  │  where: [{field:'tenant_id', op:'=', value:'acme-123'},
  │  │          {field:'status', op:'=', value:'new'}]
  │  │
  │  ③ QueryValidator.validate(ast)
  │  │  6 checks: tenant_id? allowedTables? no DDL? parameterized? → PASS
  │  │
  │  ④ QueryCounter.increment()
  │  │  queryCount++ → [4/50]
  │  │
  │  ⑤ ISQLDialect.generateSelect(ast)
  │  │  PostgresDialect:
  │  │  { text: "SELECT * FROM sales WHERE tenant_id = $1 AND status = $2 LIMIT $3",
  │  │    params: ['acme-123', 'new', 10] }
  │  │  MySQLDialect (nếu tenant dùng MySQL):
  │  │  { text: "SELECT * FROM sales WHERE tenant_id = ? AND status = ? LIMIT ?",
  │  │    params: ['acme-123', 'new', 10] }
  │  │
  │  └── Forward { text, params } to pool connection
  │
  ├── Cache Manager (optional — if caching enabled for this query)
  │   │
  │   │  Cache key: t:acme-123:sales:hash(AST)
  │   │  L1 (in-memory): check → MISS
  │   │  L2 (Redis):     check → MISS
  │   │  → proceed to DB
  │   │  (after DB response: store in L2 Redis, TTL 5min → store in L1, TTL 60s)
  │
  ▼
  L4 DATA ACCESS ── Pool Manager
  │
  │  tenantCtx.dbPool = "shared"
  │  → Use existing connection #14 (already acquired for this request)
  │  → Connection configured by ITenantStrategy.configureConnection():
  │    PostgresRLSStrategy: SET app.tenant_id = 'acme-123'
  │    MySQLTenantStrategy: SET @app_tenant = 'acme-123'
  │
  ▼
  L5 INFRASTRUCTURE ── PostgreSQL (Shared Data DB)
  │
  │  ┌── PostgreSQL execution ──────────────────────────────────────┐
  │  │                                                              │
  │  │  Receive: SELECT * FROM sales                                │
  │  │           WHERE tenant_id = $1 AND status = $2 LIMIT $3     │
  │  │           params: ['acme-123', 'new', 10]                   │
  │  │                                                              │
  │  │  1. RLS check (tầng 3 — backup enforcement, PG only):       │
  │  │     Policy: tenant_id = current_setting('app.tenant_id')    │
  │  │     'acme-123' = 'acme-123' → PASS                         │
  │  │                                                              │
  │  │  2. Query plan:                                              │
  │  │     Use index: idx_sales_tenant_status                      │
  │  │     → Index scan, very efficient                            │
  │  │                                                              │
  │  │  3. Execute: read from buffer cache → 4 rows match           │
  │  │                                                              │
  │  │  4. Return result set → connection #14                      │
  │  └──────────────────────────────────────────────────────────────┘
  │
  │  ⑥ DB execute complete
  │  ⑦ QueryLogger.log({ table:sales, op:SELECT, duration:5ms })
  │
  ▼
  Result bubbles back up:

  PostgreSQL → Pool conn → Cache (store) → TenantScopedDB → Plugin Core

  Plugin receives: [{ id: "sale-42", title: "Enterprise deal", ... }, ...]
  Plugin does not know about QueryAST, ISQLDialect, ITenantStrategy, RLS, or caching.

  ┌─────────────────────────────────────────────────────────────────┐
  │                                                                 │
  │  customers table → PARTITION BY HASH(tenant_id)                │
  │                                                                 │
  │  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐            │
  │  │ customers_p0 │ │ customers_p1 │ │ customers_p2 │ ...        │
  │  │ hash mod 0   │ │ hash mod 1   │ │ hash mod 2   │            │
  │  │              │ │              │ │              │            │
  │  │ tenants A-G  │ │ tenants H-N  │ │ tenants O-Z  │            │
  │  └──────────────┘ └──────────────┘ └──────────────┘            │
  │                                                                 │
  │  Query: WHERE tenant_id = 'acme-123'                           │
  │  → PostgreSQL calculates hash → routes to correct partition    │
  │  → Scans ONLY that partition (partition pruning)               │
  │  → Much faster for large datasets                               │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘


  Phase 4: Shard (khi write throughput > 10K QPS)

  ┌─────────────────────────────────────────────────────────────────┐
  │                                                                 │
  │  Shard by tenant groups:                                        │
  │                                                                 │
  │  ┌─────────────────┐  ┌─────────────────┐  ┌────────────────┐ │
  │  │  Shard 1         │  │  Shard 2         │  │  Shard 3        │ │
  │  │  DB Instance     │  │  DB Instance     │  │  DB Instance    │ │
  │  │                  │  │                  │  │                 │ │
  │  │  Tenants A-G     │  │  Tenants H-N     │  │  Tenants O-Z    │ │
  │  │  (full schema)   │  │  (full schema)   │  │  (full schema)  │ │
  │  │  + own replicas  │  │  + own replicas  │  │  + own replicas │ │
  │  └─────────────────┘  └─────────────────┘  └────────────────┘ │
  │                                                                 │
  │  Pool Manager routing:                                          │
  │    tenant_id → hash → shard number → correct DB instance       │
  │                                                                 │
  │  Trade-off: cross-shard queries not possible                   │
  │  Benefit: near-linear write scalability                        │
  │                                                                 │
  │  Thường không cần đến Phase 4 cho đa số SaaS.                 │
  │  VIP tenants đã tách DB riêng → giảm load Shared DB đáng kể.   │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘

Context System Deep Dive

Execution Context

Hai loại context, quá trình build, vòng đời, cơ chế cache, và cách context điều khiển toàn bộ event/hook filtering.

01

Hai loại Context trong hệ thống

Tại sao cần 2 loại? Tenant Context giữ thông tin "tenant này là ai, dùng gì" — tồn tại lâu dài, dùng chung cho mọi request. Execution Context giữ thông tin "request này cần gì để chạy" — chỉ sống trong 1 request rồi bị hủy.

03

Execution Context — Cấu trúc chi tiết

execution context anatomy
┌─ EXECUTION CONTEXT ─── tồn tại DUY NHẤT trong 1 request ──────────────────┐
│                                                                            │
│  ┌─ Request Identity ────────────────────────────────────────────────────┐  │
│  │                                                                      │  │
│  │  requestId:   "req-uuid-789"    ← unique, dùng cho tracing         │  │
│  │  tenantId:    "acme-123"        ← copy từ Tenant Context           │  │
│  │  userId:      "user-456"        ← từ JWT token                     │  │
│  │  roles:       ["sales_manager"] ← từ JWT token                     │  │
│  │  ip:          "203.0.113.42"                                         │  │
│  │                                                                      │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                                                                            │
│  ┌─ Scoped Resources ─── acquire từ pool, auto-release sau request ────┐  │
│  │                                                                      │  │
│  │  ┌──────────────────────────────────────────────────────────────┐    │  │
│  │  │  db: TenantScopedDB                                        │    │  │
│  │  │                                                              │    │  │
│  │  │  Bọc 1 connection từ pool (shared hoặc VIP)                  │    │  │
│  │  │  Query Interceptor đã gắn sẵn:                               │    │  │
│  │  │    → Mọi query tự động thêm WHERE tenant_id = 'acme-123'    │    │  │
│  │  │    → SET app.current_tenant trên connection                  │    │  │
│  │  │  Plugin gọi ctx.db.query('customers') → an toàn             │    │  │
│  │  │                                                              │    │  │
│  │  │  Plugin KHÔNG THỂ:                                          │    │  │
│  │  │    ✗ Bỏ qua tenant_id filter                                │    │  │
│  │  │    ✗ Query raw SQL không qua interceptor                     │    │  │
│  │  │    ✗ Thay đổi tenant_id sang tenant khác                    │    │  │
│  │  └──────────────────────────────────────────────────────────────┘    │  │
│  │                                                                      │  │
│  │  ┌──────────────────────────────────────────────────────────────┐    │  │
│  │  │  cache: TenantScopedCache                                    │    │  │
│  │  │                                                              │    │  │
│  │  │  Bọc Redis client với auto-prefix:                           │    │  │
│  │  │    ctx.cache.get('customers:list')                           │    │  │
│  │  │    → thực tế get('t:acme-123:customers:list')                │    │  │
│  │  │                                                              │    │  │
│  │  │  Plugin không thể đọc cache của tenant khác                  │    │  │
│  │  └──────────────────────────────────────────────────────────────┘    │  │
│  │                                                                      │  │
│  │  ┌──────────────────────────────────────────────────────────────┐    │  │
│  │  │  logger: ScopedLogger                                        │    │  │
│  │  │                                                              │    │  │
│  │  │  Mọi log tự động kèm:                                       │    │  │
│  │  │    { tenantId, requestId, userId, plugin }                   │    │  │
│  │  │  PII auto-redaction trước khi ship                           │    │  │
│  │  └──────────────────────────────────────────────────────────────┘    │  │
│  │                                                                      │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                                                                            │
│  ┌─ Scoped Event Bus ────────────────────────────────────────────────────┐  │
│  │                                                                      │  │
│  │  Bọc system Event Bus với 2 ràng buộc:                               │  │
│  │                                                                      │  │
│  │  1. Emit filter:                                                    │  │
│  │     Plugin chỉ emit events đã khai báo trong manifest                │  │
│  │     customer-care manifest: emits: ["sales.created", "call.logged"]  │  │
│  │     → ctx.eventBus.emit("sales.created", data)  OK                  │  │
│  │     → ctx.eventBus.emit("user.deleted", data)   BLOCKED             │  │
│  │                                                                      │  │
│  │  2. Deliver filter:                                                  │  │
│  │     Khi deliver event đến handlers,                                  │  │
│  │     chỉ gọi handlers của plugins trong ctx.enabledPlugins            │  │
│  │     (chi tiết ở tab "Cơ chế Filtering")                              │  │
│  │                                                                      │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                                                                            │
│  ┌─ Shared Services ─── singleton refs, nhưng gọi với tenant scope ─────┐  │
│  │                                                                      │  │
│  │  email:         EmailService.send(ctx, ...)                          │  │
│  │                 → auto thêm tenant branding, from address            │  │
│  │                                                                      │  │
│  │  storage:       StorageService.upload(ctx, ...)                      │  │
│  │                 → auto prefix path: /tenants/acme-123/uploads/       │  │
│  │                                                                      │  │
│  │  notification:  NotificationService.push(ctx, ...)                   │  │
│  │                 → route đến đúng tenant's WebSocket channel          │  │
│  │                                                                      │  │
│  │  mq:            MQPublisher.publish(ctx, ...)                        │  │
│  │                 → auto gắn tenantId vào message metadata             │  │
│  │                                                                      │  │
│  │  Services là singleton, nhưng mọi lời gọi đều tenant-scoped         │  │
│  │  nhờ ctx chứa tenantId                                              │  │
│  │                                                                      │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                                                                            │
│  ┌─ Plugin Config Snapshot ─────────────────────────────────────────────┐  │
│  │                                                                      │  │
│  │  Copy từ Tenant Context → plugin config cho plugin đang xử lý:      │  │
│  │                                                                      │  │
│  │  Route /api/v1/sales → plugin: customer-care                        │  │
│  │  → ctx.pluginConfig = { twilio_sid: "xxx", sla: 24h }              │  │
│  │                                                                      │  │
│  │  Read-only snapshot — plugin không thể modify config lâu dài        │  │
│  │                                                                      │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                                                                            │
│  Tổng kích thước: ~15 KB                                                 │
│  (phần lớn là wrapper objects quanh pool resources)                       │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘

Cost summary — Build context cho request

timing breakdown
Request đầu tiên từ tenant (cold — no cached context):

  Phase 1: Resolve Tenant
    ├── Redis domain lookup:    MISS   → ~1ms
    └── DB query tenants table:         → ~5ms
                                          ────── ~6ms

  Phase 2: Build Tenant Context
    ├── DB query tenant_plugins:        → ~8ms
    ├── Map to singleton refs:          → ~0.1ms
    ├── Aggregate permissions:          → ~0.1ms
    ├── Resolve routing:                → ~0.1ms
    └── Store in LRU:                   → ~0.1ms
                                          ────── ~9ms

  Phase 3: Build Execution Context
    ├── Acquire DB connection:          → ~2ms
    ├── Create scoped wrappers:         → ~1ms
    └── Resolve plugin config:          → ~0.1ms
                                          ────── ~3ms

  Total (cold): ~18ms


Request tiếp theo từ cùng tenant (warm — cached context):

  Phase 1: Resolve Tenant
    └── Redis domain lookup:    HIT    → ~0.5ms
                                          ────── ~0.5ms

  Phase 2: Get Tenant Context
    └── LRU Cache lookup:       HIT    → ~0.1ms
                                          ────── ~0.1ms

  Phase 3: Build Execution Context
    ├── Acquire DB connection:          → ~2ms
    ├── Create scoped wrappers:         → ~1ms
    └── Resolve plugin config:          → ~0.1ms
                                          ────── ~3ms

  Total (warm): ~3.6ms


Chênh lệch: ~14ms cho request đầu tiên
Từ request thứ 2 trở đi: ổn định ~3-4ms

Vòng đời & Cache Strategy

context lifecycle
TENANT CONTEXT LIFECYCLE

  Create ─────── Cache ────── Serve ────── Invalidate ────── Rebuild
     │              │            │               │                │
     │              │            │               │                │
  1st request   LRU Cache    Mọi request     Trigger:         Next request
  from tenant   max 1000     tiếp theo       • plugin          auto rebuild
  → build       entries      dùng cached       enable/disable  + re-cache
  from DB                    context          • config change
                                              • tier change
                                              • manual flush


CHI TIẾT TỪNG GIAI ĐOẠN:

① CREATE — Khi tenant gửi request đầu tiên (hoặc sau invalidate)
│
│  Trigger: LRU cache MISS cho tenant_id
│  Action:  Query DB → build TenantContext → store in LRU
│  Cost:    ~15-30ms (1 DB round-trip)
│
② CACHE — Context nằm trong LRU, sẵn sàng phục vụ
│
│  Storage:  In-memory LRU cache (per app instance)
│  Capacity: max 1000 tenant contexts
│  Size:     ~10KB × 1000 = ~10MB total
│  Eviction: LRU — tenant ít active nhất bị evict trước
│  TTL:      evict sau 2h inactive (configurable)
│
│  Mỗi app instance có LRU RIÊNG
│  Instance 1 cache ACME ≠ Instance 2 cache ACME
│  Nhưng cùng data vì build từ cùng DB
│
③ SERVE — Request đến, dùng cached context
│
│  Action:  LRU lookup → HIT → return context
│           Update lastAccessed timestamp
│  Cost:    ~0.1ms (memory lookup)
│
④ INVALIDATE — Context trở nên stale, cần xóa
│
│  Triggers:
│  ┌────────────────────────────────────────────────────────────┐
│  │                                                            │
│  │  Tenant Admin enable/disable plugin                        │
│  │       → invalidate context của tenant đó                   │
│  │                                                            │
│  │  Tenant Admin thay đổi plugin config                       │
│  │       → invalidate context của tenant đó                   │
│  │                                                            │
│  │  Tenant upgrade/downgrade tier                              │
│  │       → invalidate context (quotas, routing thay đổi)      │
│  │                                                            │
│  │  System Admin deploy/undeploy plugin                        │
│  │       → invalidate TẤT CẢ tenant contexts                  │
│  │       (vì plugin cores singleton đã thay đổi)              │
│  │                                                            │
│  └────────────────────────────────────────────────────────────┘
│
│  Invalidation mechanism:
│  ┌────────────────────────────────────────────────────────────┐
│  │                                                            │
│  │  1. Delete from local LRU cache                            │
│  │  2. Publish via Redis Pub/Sub:                              │
│  │       channel: "context:invalidate"                        │
│  │       payload: { tenantId: "acme-123" }                    │
│  │       (hoặc { all: true } cho system-wide)                 │
│  │  3. ALL app instances receive message                       │
│  │  4. Each instance deletes from their local LRU               │
│  │                                                            │
│  └────────────────────────────────────────────────────────────┘
│
⑤ REBUILD — Next request triggers fresh build
│
│  Trigger: LRU cache MISS (vì vừa invalidate)
│  Action:  Query DB → build new TenantContext → store in LRU
│  Result:  Context phản ánh trạng thái mới nhất
│           (new plugins, new config, new tier)

Mọi thứ chạy SONG SONG, AN TOÀN, vì:
• Plugin cores stateless → thread-safe
• Mỗi request có Execution Context riêng → no shared mutable state
• DB connections riêng → no cross-tenant query
• Cache prefix riêng → no cross-tenant cache read
• Event filtering riêng → handlers khác nhau per tenant

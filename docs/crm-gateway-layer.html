Bản đồ L2 — Pipeline tuần tựL2 Map — Sequential Pipeline

Nguyên tắc cốt lõi: L2 là "cổng duy nhất" vào hệ thống. MỌI request (từ Web, Mobile, API) đều đi qua 5 middleware theo đúng thứ tự. Mỗi middleware có thể REJECT request ngay — nếu pass, chuyển tiếp cho middleware sau. Qua hết 5 bước → request đạt đủ điều kiện để vào L3.
Core principle: L2 is the "single gateway" into the system. ALL requests (from Web, Mobile, API) pass through 5 middleware in exact order. Each middleware can REJECT the request immediately — if it passes, it forwards to the next middleware. Through all 5 steps → the request qualifies to enter L3.
L2 pipeline
  Client (Web/Mobile/API) → HTTPS → Ingress
                                      │
  ┌─ L2 API GATEWAY ─────────────────────▼──────────────────────────────────┐
  │                                                                          │
  │  ❶ LOAD BALANCER          TLS terminate, chọn instance           ~0ms  │
  │  │  Nginx/K8s Ingress      VIP → dedicated pool                         │
  │  │                         Standard → shared pool (least-conn)          │
  │  │                         Health-check → loại instance chết            │
  │  ▼                                                                       │
  │  ❷ TENANT RESOLVER        URL → tenant_id + tier + status        ~1ms  │
  │  │  Redis cache → Meta DB   subdomain: acme.crm.com → "acme-123"       │
  │  │                         custom domain: SNI routing                    │
  │  │  REJECT: 404 Not Found / 403 Suspended / 410 Gone                   │
  │  ▼                                                                       │
  │  ❸ AUTH GUARD             JWT validate + RBAC extract            ~1.5ms│
  │  │  JWT decode + verify    tenant claim match? blacklist check?         │
  │  │                         → extract userId, roles, permissions         │
  │  │  REJECT: 401 Unauthorized / 401 Expired / 403 Forbidden             │
  │  ▼                                                                       │
  │  ❹ RATE LIMITER           Sliding window counter (Redis)          ~1ms  │
  │  │  Redis ZADD              Basic:100, Prem:500, Ent:2000, VIP:∞        │
  │  │                         Burst: 2× trong 5s                           │
  │  │  REJECT: 429 Too Many Requests + Retry-After                       │
  │  ▼                                                                       │
  │  ❺ ROUTE MATCHER          URL → plugin + method + permission      ~0.3ms│
  │  │  Plugin Registry         POST /api/v1/sales → care.createSale       │
  │  │  routeTable lookup      + check plugin enabled for tenant           │
  │  │                         + check user has required permission         │
  │  │  REJECT: 404 No Route / 403 Plugin Not Available / 403 Forbidden    │
  │  ▼                                                                       │
  │  OUTPUT:                                                                 │
  │  { tenantId, tier, userId, roles, plugin, method, permission }          │
  │  → Chuyển cho L3 Context Building                                       │
  │                                                                          │
  └──────────────────────────────────────────────────────────────────────────┘

  TỔNG THỜI GIAN L2: ~3.9ms (best case, mọi cache hit)
  Worst case (cache miss): ~8-12ms (Metadata DB query cho tenant/domain)

  REJECT RATE TRUNG BÌNH:
  ├── ~0.1% bị 401 (token expired, invalid)
  ├── ~0.01% bị 429 (rate limit, chủ yếu Basic tier)
  ├── ~0.001% bị 403 (permission, plugin not enabled)
  └── 99.8% requests pass qua Gateway thành công

Load Balancer + TLS TerminationLoad Balancer + TLS Termination

Vai trò: Điểm tiếp xúc đầu tiên. Terminate HTTPS → chuyển HTTP nội bộ (within cluster). Phân phối traffic tới đúng instance pool dựa trên tenant tier. Loại instance chết khỏi rotation.
Role: First point of contact. Terminate HTTPS → forward as internal HTTP (within cluster). Distribute traffic to the correct instance pool based on tenant tier. Remove dead instances from rotation.
load balancer
THÀNH PHẦN: Nginx / K8s Ingress Controller / AWS ALB

TLS TERMINATION:
  HTTPS → HTTP (cluster internal, plain HTTP giữa services)
  Certificate: wildcard *.crm.com hoặc per-tenant
  Custom domain: acme.example.com → SNI routing → crm backend

ROUTING LOGIC:

  Request từ acme.crm.com
    │
    ├── Tenant ACME tier = VIP?
    │   YES → route to VIP instance pool (crm-api-vip, 2-4 replicas)
    │         Sticky routing: cùng VIP tenant → cùng pool
    │
    │   NO  → route to shared instance pool (crm-api, 3-10 replicas)
    │         Algorithm: least-connections
    │         → chọn instance đang ít request nhất
    │
    └── ACME tier = premium → shared pool, Instance #3 selected

  Lưu ý: LB chưa biết tenant tier chính xác.
  VIP routing dựa trên domain/IP range configured tĩnh.
  Standard domains (*.crm.com) → shared pool.
  VIP domains (vip-initech.crm.com) → VIP pool.

HEALTH CHECK:
  ├── Probe: GET /health/live mỗi 10s
  ├── Instance fail 3 lần liên tiếp → loại khỏi rotation
  ├── Instance recover → thêm lại vào rotation
  └── K8s readiness probe: GET /health/ready mỗi 5s
      → chưa ready = không nhận traffic mới

HPA (Horizontal Pod Autoscaler):
  ├── crm-api (shared):  scale 3→10 khi CPU > 70%
  ├── crm-api-vip:       scale 2→4 khi CPU > 60%
  └── crm-workers:       scale 2→6 khi queue depth > 500

Tenant Resolver — URL → Tenant IdentityTenant Resolver — URL → Tenant Identity

Vai trò: Xác định request thuộc về tenant nào. Dựa vào subdomain, custom domain, hoặc header. Cache kết quả trong Redis để tránh query DB mỗi request. Output: tenantId + tier + status.
Role: Determine which tenant the request belongs to. Based on subdomain, custom domain, or header. Cache results in Redis to avoid querying DB on every request. Output: tenantId + tier + status.
tenant resolver
INPUT: HTTP Host header = "acme.crm.com"

STRATEGY (thử lần lượt, dừng khi match):

  ① Subdomain resolution (phổ biến nhất):
     Extract "acme" from "acme.crm.com"
     → Redis: GET domain-map:acme
     → HIT:  { tenantId: "acme-123", tier: "premium", status: "active" }
     → MISS: Query Metadata DB:
             SELECT id, tier, status, config
             FROM tenants WHERE subdomain = 'acme'
             → Cache Redis (TTL 5min)

  ② Custom domain resolution (Enterprise/VIP):
     Host = "crm.initech.io" (không phải *.crm.com)
     → Redis: GET domain-map:crm.initech.io
     → MISS: SELECT ... FROM tenants WHERE custom_domain = 'crm.initech.io'
     → Yêu cầu: tenant setup CNAME/A record → crm platform IP

  ③ Header-based resolution (M2M / API calls):
     Header X-Tenant-ID: acme-123
     → Validate tenantId exists in Redis/DB
     → Dùng cho API keys, server-to-server calls
     → Yêu cầu: JWT tenant claim phải khớp (double check ở Auth Guard)


OUTPUT:
  { tenantId: "acme-123", tier: "premium", status: "active" }
  → Gắn vào request context, dùng suốt phần còn lại của pipeline

REDIS CACHE:
  Key: domain-map:{subdomain|domain}
  Value: { tenantId, tier, status }
  TTL: 5 phút
  Invalidation: tenant update → DELETE domain-map:* cho tenant đó

REJECT CASES:
  ├── Subdomain/domain không tồn tại     → 404 Tenant Not Found
  ├── Tenant status = "suspended"         → 403 Account Suspended
  │   (chưa thanh toán, vi phạm TOS)       Body: { reason, contactEmail }
  ├── Tenant status = "deleted"           → 410 Gone
  └── Tenant status = "pending_setup"     → 503 Not Ready (đang setup)

PERFORMANCE:
  Redis HIT: ~0.5ms
  Redis MISS + DB: ~3-5ms (hiếm, ~5% requests)
  Trung bình: ~1ms

Auth Guard — JWT Validation + RBAC ExtractAuth Guard — JWT Validation + RBAC Extract

Vai trò: Xác thực người dùng (authentication) và trích xuất quyền hạn (authorization data). Validate JWT signature + expiry, kiểm tra token chưa bị revoke, đảm bảo tenant claim khớp với Tenant Resolver output. Output: userId, roles, permissions.
Role: Authenticate users and extract authorization data. Validate JWT signature + expiry, check token has not been revoked, ensure tenant claim matches Tenant Resolver output. Output: userId, roles, permissions.
auth guard
INPUT: Header "Authorization: Bearer eyJhbG..."

6 BƯỚC VALIDATION:

  ① Decode JWT header → get signing algorithm (RS256)
     Nếu algorithm không nằm trong whitelist → reject ngay

  ② Validate signature using public key
     Public key cached in-memory, rotated periodically
     Invalid signature → token giả mạo hoặc bị sửa

  ③ Check expiration: exp > now()?
     Access token TTL: 15 phút
     Expired → client phải dùng refresh token (7 ngày) để lấy token mới
     Refresh token: rotation per use (1 lần dùng → issue new pair)

  ④ Check tenant claim: jwt.tenantId == resolved tenantId?
     CỰC KỲ QUAN TRỌNG: Ngăn user Tenant A dùng token trên URL Tenant B
     Token ACME không thể dùng trên globex.crm.com
     → Cross-tenant access attack blocked

  ⑤ Check blacklist: Redis GET blacklist:{jti}
     jti = JWT ID (unique per token)
     Token bị blacklist khi: user logout, admin revoke, password changed
     TTL blacklist entry = thời gian còn lại của token
     (sau khi token expire → blacklist entry tự xóa)

  ⑥ Extract claims:
     userId:      "user-5"
     roles:       ["sales_manager"]
     permissions: ["sales:read", "sales:write", "calls:read", ...]


CÁC LOẠI AUTH:

  ┌──────────────┬────────────────────────────────────────────────────────┐
  │ Loại          │ Cơ chế                                                │
  ├──────────────┼────────────────────────────────────────────────────────┤
  │ User (Web)   │ JWT Bearer token (access 15m + refresh 7d)            │
  │ User (Mobile)│ JWT + device fingerprint + biometric optional         │
  │ M2M (API)    │ API Key + HMAC signature (no JWT, tenant in header)   │
  │ SSO (Ent.)   │ SAML 2.0 / OIDC → exchange for JWT pair             │
  │ Webhook (in) │ Signature validation (HMAC-SHA256 with shared secret) │
  └──────────────┴────────────────────────────────────────────────────────┘


REJECT CASES:
  ├── Invalid/missing token        → 401 Unauthorized
  ├── Signature verify failed      → 401 Invalid Token
  ├── Token expired                → 401 Token Expired + hint: refresh
  ├── Tenant claim mismatch        → 403 Forbidden (cross-tenant attack)
  └── Token blacklisted (revoked)  → 401 Token Revoked

PERFORMANCE:
  JWT decode + verify (cached key):  ~0.5ms
  Redis blacklist check:             ~0.5ms
  Tenant claim compare:              ~0ms (in-memory)
  Total:                             ~1-1.5ms

Rate Limiter — Per-Tier ThrottlingRate Limiter — Per-Tier Throttling

Vai trò: Bảo vệ hệ thống khỏi overload — cả cố ý (DDoS) lẫn vô ý (script bug gọi API lặp). Giới hạn PER-TENANT, không per-user — tenant Basic 100 req/min, Premium 500, Enterprise 2000, VIP unlimited.
Role: Protect the system from overload — both intentional (DDoS) and unintentional (script bug calling API repeatedly). Limit is PER-TENANT, not per-user — Basic tenant 100 req/min, Premium 500, Enterprise 2000, VIP unlimited.
rate limiter
ALGORITHM: Sliding Window Counter (Redis Sorted Set)

  Triển khai dùng Redis ZADD + ZREMRANGEBYSCORE:

  ┌────────────────────────────────────────────────────────────────────────┐
  │                                                                        │
  │  Key:    rl:{tenantId}  (Sorted Set)                                  │
  │  Member: requestId (unique per request)                               │
  │  Score:  timestamp (Unix milliseconds)                                │
  │                                                                        │
  │  Mỗi request:                                                         │
  │  1. ZREMRANGEBYSCORE rl:acme-123 0 (now - 60000)                     │
  │     → Xóa entries cũ hơn 60 giây (clean expired)                     │
  │                                                                        │
  │  2. ZCARD rl:acme-123                                                │
  │     → Đếm entries còn lại = số request trong window                   │
  │     → current = 147                                                    │
  │                                                                        │
  │  3. 147 < 500 (premium limit)?                                       │
  │     YES → ZADD rl:acme-123 {now} {requestId}                         │
  │           EXPIRE rl:acme-123 61  (auto cleanup)                       │
  │           → PASS                                                       │
  │                                                                        │
  │     NO  → REJECT 429 Too Many Requests                                │
  │           Retry-After: {seconds until oldest entry expires}           │
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘


TIER LIMITS:

  ┌───────────────┬───────────┬───────────────┬────────────────────────────┐
  │ Tier           │ Limit     │ Burst         │ Note                       │
  ├───────────────┼───────────┼───────────────┼────────────────────────────┤
  │ Basic         │ 100/min   │ 200 trong 5s  │ Solo users, small teams    │
  │ Premium       │ 500/min   │ 1000 trong 5s │ Growing businesses         │
  │ Enterprise    │ 2000/min  │ 4000 trong 5s │ Large teams                │
  │ VIP           │ Unlimited │ N/A           │ Dedicated infrastructure   │
  └───────────────┴───────────┴───────────────┴────────────────────────────┘

  Burst: cho phép gấp đôi limit trong cửa sổ 5 giây.
  Ví dụ: Premium = 500/min, nhưng 1000 req trong 5s đầu OK.
  Giúp handle spikes tự nhiên (batch import, page load).


RESPONSE HEADERS (mọi response, kể cả khi pass):
  X-RateLimit-Limit:     500
  X-RateLimit-Remaining: 353
  X-RateLimit-Reset:     1707567600  (Unix timestamp window reset)

METRICS:
  rate_limit_hits_total{tenant_id} — Counter (đếm số lần bị 429)
  → Alert: rate_limit_hits > 0 for 5min → WARNING Slack #alerts

Route Matcher — URL → Plugin + MethodRoute Matcher — URL → Plugin + Method

Vai trò: Map URL + HTTP method vào đúng plugin + handler method. Dùng routeTable từ Plugin Registry (populated khi plugin deploy). Output là tín hiệu cho L3 biết chính xác plugin nào xử lý, method nào gọi, permission nào cần.
Role: Map URL + HTTP method to the correct plugin + handler method. Uses routeTable from Plugin Registry (populated when plugin is deployed). Output signals L3 exactly which plugin handles the request, which method to call, and which permission is required.
route matcher
INPUT: POST /api/v1/sales

ROUTE TABLE (in-memory, from Plugin Registry):

  ┌──────────────────────────────────────────────────────────────────────┐
  │  HTTP    URL Pattern              Plugin          Method             │
  │  ──────  ────────────────────    ─────────────   ────────────────   │
  │  GET     /api/v1/customers       customer-data   listCustomers      │
  │  POST    /api/v1/customers       customer-data   createCustomer     │
  │  GET     /api/v1/customers/:id   customer-data   getCustomer        │
  │  PUT     /api/v1/customers/:id   customer-data   updateCustomer     │
  │  DELETE  /api/v1/customers/:id   customer-data   deleteCustomer     │
  │                                                                      │
  │  GET     /api/v1/sales           customer-care   listSales          │
  │  POST    /api/v1/sales           customer-care   createSale  ◀ match │
  │  GET     /api/v1/sales/:id       customer-care   getSale            │
  │  PUT     /api/v1/sales/:id       customer-care   updateSale         │
  │                                                                      │
  │  GET     /api/v1/calls           customer-care   listCalls          │
  │  POST    /api/v1/calls           customer-care   createCall         │
  │                                                                      │
  │  GET     /api/v1/reports         analytics       listReports        │
  │  POST    /api/v1/reports         analytics       createReport       │
  │  ...                                                                 │
  └──────────────────────────────────────────────────────────────────────┘

  Matching: trie-based prefix tree → O(path length), ~0.1ms
  Parametric: /api/v1/sales/:id → extract params { id: "sale-42" }

OUTPUT:
  {
    plugin:             "customer-care",
    method:             "createSale",
    requiredPermission: "sales:write",
    params:             {}
  }


ROUTE TABLE LIFECYCLE:
  ├── Xây dựng khi: app bootstrap (load tất cả plugin manifests)
  ├── Cập nhật khi: plugin deploy/upgrade/undeploy
  │   → MQ broadcast "plugin.deployed" → all instances rebuild
  ├── In-memory: không query DB/Redis per-request
  └── Collision detection: 2 plugins register cùng route → reject deploy

REJECT CASES:
  └── No route match → 404 Not Found

Permission Check — Plugin + User + TenantPermission Check — Plugin + User + Tenant

Vai trò: Kiểm tra 3 lớp quyền: plugin có enabled cho tenant? User có quyền? Tenant tier có bao gồm feature? Đây là bước cuối trước khi request vào L3. Dùng data đã extract từ Tenant Resolver + Auth Guard + Route Matcher.
Role: Check 3 permission layers: is the plugin enabled for the tenant? Does the user have permission? Does the tenant tier include the feature? This is the final step before the request enters L3. Uses data already extracted from Tenant Resolver + Auth Guard + Route Matcher.
permission check
3 LỚP KIỂM TRA:

  Route requires: "sales:write" (from Route Matcher)
  Tenant: acme-123, premium (from Tenant Resolver)
  User: user-5, roles: ["sales_manager"] (from Auth Guard)

  ┌────────────────────────────────────────────────────────────────────────┐
  │                                                                        │
  │  Check 1: Plugin enabled cho tenant?                                   │
  │                                                                        │
  │    Route target: plugin "customer-care"                                │
  │    Cần biết: ACME có enable customer-care?                            │
  │                                                                        │
  │    Ở giai đoạn này, Tenant Context chưa load đầy đủ.                 │
  │    → Quick check: Redis cache hoặc Metadata DB                        │
  │      SELECT 1 FROM tenant_plugins                                     │
  │      WHERE tenant_id='acme-123' AND plugin='customer-care'            │
  │      AND enabled=true                                                  │
  │    → EXISTS → PASS                                                     │
  │    → NOT EXISTS → 403 Plugin Not Available                             │
  │                                                                        │
  │                                                                        │
  │  Check 2: User có permission cần thiết?                                │
  │                                                                        │
  │    Required: "sales:write"                                            │
  │    User roles: ["sales_manager"]                                      │
  │    Role "sales_manager" grants: sales:read, sales:write,              │
  │                                  calls:read, calls:write              │
  │    "sales:write" ∈ role permissions?                                  │
  │    → YES → PASS                                                       │
  │    → NO  → 403 Forbidden ("You don't have sales:write permission")   │
  │                                                                        │
  │                                                                        │
  │  Check 3: Tenant-level permission?                                     │
  │                                                                        │
  │    Tenant aggregatedPermissions = union of all enabled plugins' perms │
  │    "sales:write" ∈ aggregatedPermissions?                             │
  │    → YES → PASS                                                       │
  │    → NO  → 403 Feature Not Included In Your Plan                      │
  │    (xảy ra khi plugin enabled nhưng cụ thể feature bị restrict       │
  │     dựa trên tier config — ví dụ: Premium có sales nhưng không có    │
  │     sales:export vì export chỉ Enterprise trở lên)                   │
  │                                                                        │
  │                                                                        │
  │  All 3 checks pass → PROCEED to L3                                   │
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘


RBAC MODEL:

  ┌─────────────────────────────────────────────────────────────────────┐
  │                                                                     │
  │  Tenant has Roles:                                                 │
  │  ├── admin:          * (all permissions)                           │
  │  ├── sales_manager:  sales:*, calls:*, customers:read             │
  │  ├── support:        tickets:*, calls:read, customers:read        │
  │  ├── viewer:         *.read (read-only across all)                │
  │  └── custom roles:   tenant-defined, any combination              │
  │                                                                     │
  │  User has Roles (assigned per-tenant):                             │
  │  user-5 → ["sales_manager"]                                       │
  │  user-8 → ["admin"]                                               │
  │  user-12 → ["support", "viewer"]                                  │
  │                                                                     │
  │  Permission format: {resource}:{action}                            │
  │  Actions: read, write, delete, export, admin                      │
  │  Wildcard: sales:* = all actions on sales                         │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘

  Cost: ~0.1ms (in-memory role lookup, no I/O)

🔬 Tại sao isolated-vm?
✓

V8 Isolate là hard boundary thực sự. Mỗi Isolate có heap riêng — plugin không thể truy cập hay corrupt memory của host process. Khác với Worker Threads (shared ArrayBuffer) hay child_process (overhead cao).

ApproachIsolationOverheadMemory limit
isolated-vm CHOSEN True V8 isolate, separate heap ~1ms setup Hard (V8 kills on breach)
Worker Threads Separate thread, shared memory possible ~5ms No built-in limit
child_process Full process isolation ~30ms via cgroups (external)
Promise.race (Phase 5) None — same heap as host ~0ms None
⚙ IsolatedSandboxService.runScript() — core flow

async runScript<T>( script: string, bridge: SandboxBridge, options: { timeoutMs?: number; memoryLimitMb?: number } = {} ): Promise<T> { // 1. Create fresh isolate with 50 MB limit — separate V8 heap const isolate = new ivm.Isolate({ memoryLimit: 50 }); try { const context = await isolate.createContext(); // 2. Inject bridge References (host fns → isolate refs) await context.global.set('__db_query', new ivm.Reference(async(table, whereJson) => { return JSON.stringify(await bridge.db.query(table, JSON.parse(whereJson))); })); // ... __db_count, __cache_get, __cache_set, __log ... // 3. Setup script: create ctx wrapper inside isolate await context.eval(BRIDGE_SETUP_SCRIPT); // 4. Compile + run plugin script with hard timeout const compiled = await isolate.compileScript( `(async function __plugin(ctx) {\n${script}\n})(ctx)` ); return await compiled.run(context, { timeout: 5000, promise: true }); } catch (err) { return this.handleError(err, startMs); // maps to typed exceptions + metrics } finally { isolate.dispose(); // always release isolate heap } }

🔢 Query Limit — QueryInterceptor.increment(true)
⚠

Phase 5 để lại khoảng trống: QueryInterceptor gọi QueryCounter.increment(false) — đếm queries nhưng không bao giờ throw. Phase 6 flip thành increment(true) để enforce giới hạn.

QueryInterceptor.ts — changeTypeScript diff
// Before (Phase 5 — counts but never throws): - QueryCounter.increment(false); // After (Phase 6 — throws QueryLimitExceededError on 51st query): + QueryCounter.increment(true);
✓

QueryCounter.increment(true) được gọi trong acquireConnection hook của Knex — mỗi DB connection acquire. Sau 50 queries, QueryLimitExceededError được throw từ bên trong Knex, propagate qua bridge.db.query(), qua applySyncPromise(), và được catch bởi IsolatedSandboxService.handleError().

QueryCounter.tsTypeScript
export const QUERY_LIMIT = 50; export const QueryCounter = { increment(throwOnLimit = false): number { const count = TenantContext.incrementQueryCount(); if (throwOnLimit && count > QUERY_LIMIT) { throw new QueryLimitExceededError(count, QUERY_LIMIT); } return count; }, };

Counter được lưu trong TenantContext (AsyncLocalStorage) — hoàn toàn isolated per request, không cần lock hay shared state. Ngay cả khi 1000 concurrent requests, mỗi request có counter riêng.

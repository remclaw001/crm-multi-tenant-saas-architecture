AmqpService
🔌 Connection + Channel Management
★

One channel per consumer là amqplib best practice. Shared channel trong AmqpService chỉ dùng cho publishing — không cho consuming. Mỗi consumer gọi amqp.createChannel() riêng để có flow control (prefetch) độc lập.

amqp.service.tsTypeScript
@Injectable() export class AmqpService implements OnModuleInit, OnModuleDestroy { private connection: amqp.ChannelModel | null = null; private sharedChannel: amqp.Channel | null = null; async onModuleInit(): Promise<void> { // connects to RABBITMQ_URL from config this.connection = await amqp.connect(config.RABBITMQ_URL); this.sharedChannel = await this.connection.createChannel(); } /** Shared channel — publishers only */ getChannel(): amqp.Channel { return this.sharedChannel!; } /** Dedicated channel per consumer (independent prefetch) */ async createChannel(): Promise<amqp.Channel> { return this.connection!.createChannel(); } }
AmqpPublisher

@Injectable() export class AmqpPublisher { publishAudit(message: AuditMessage): void { this.publish(EXCHANGES.AUDIT, QUEUES.AUDIT, message); } publishNotification(message: NotificationMessage): void { ... } publishSearchIndex(message: SearchIndexMessage): void { ... } publishWebhook(message: WebhookMessage): void { ... } private publish(exchange, routingKey, message): void { const ch = this.amqp.getChannel(); ch.publish(exchange, routingKey, Buffer.from(JSON.stringify(message)), { contentType: 'application/json', persistent: true, // survives broker restart (deliveryMode: 2) }, ); } }

AuditConsumer — Critical path
📝 Writes to audit_logs, nack → DLQ on failure
⚠

prefetch(1): AuditConsumer chỉ xử lý 1 message tại một thời điểm để đảm bảo back-pressure từ DB. Nếu DB chậm, consumer tự động giảm tốc độ consume — RabbitMQ buffer message trong queue thay vì flood consumer.

audit.consumer.tsTypeScript
async onApplicationBootstrap(): Promise<void> { this.channel = await this.amqp.createChannel(); await this.channel.prefetch(1); await this.channel.consume(QUEUES.AUDIT, (msg) => this.handleMessage(msg)); } private async handleMessage(msg): Promise<void> { let data: AuditMessage; try { data = JSON.parse(msg.content.toString()); } catch { this.channel.nack(msg, false, false); // malformed → DLQ return; } try { await this.knex('audit_logs').insert({ ... }); this.channel.ack(msg); } catch { this.channel.nack(msg, false, false); // DB error → DLQ (no hot-retry) } }
★

Tại sao nack(requeue=false) thay vì nack(requeue=true)? Requeue=true sẽ tạo hot-retry loop — message được re-process ngay lập tức mà không có delay, có thể làm crash consumer. Với requeue=false, message vào DLQ để inspect sau — không ảnh hưởng throughput của queue chính.

WebhookRetryProcessor
🔗 Outbound HTTP + HMAC-SHA256 signature
webhook-retry.processor.tsTypeScript
async process(job: Job<WebhookMessage>): Promise<void> { const body = JSON.stringify({ event, payload, tenantId, webhookId }); // HMAC-SHA256 signature (nếu x-webhook-secret có trong headers) const secret = headers['x-webhook-secret']; if (secret) { const sig = createHmac('sha256', secret).update(body).digest('hex'); reqHeaders['X-CRM-Signature'] = `sha256=${sig}`; delete reqHeaders['x-webhook-secret']; // không forward secret } const res = await fetch(url, { method: 'POST', headers: reqHeaders, body, signal: controller.signal, // 10s timeout }); if (!res.ok) throw new Error(`HTTP ${res.status} from ${url}`); // Non-2xx → job fails → BullMQ retries with exponential backoff }
★

Node.js 22+ built-in fetch: Không cần axios hay got — fetch có sẵn từ Node 18+. Project yêu cầu Node >=22 nên hoàn toàn safe. AbortController với 10s timeout đảm bảo job không block worker thread vô thời hạn.

⚠️ Vitest hoisting gotcha
⚠

vi.mock() factory chạy trước variable initialisation. Nếu factory reference một biến const ở outer scope, sẽ throw Cannot access 'x' before initialization vì vi.mock() được hoisted lên trên cùng file.

Pattern: vi.hoisted() để fix TDZ errorTypeScript
// ❌ Sai — const mockSendMail defined after vi.mock() hoisting const mockSendMail = vi.fn(); vi.mock('nodemailer', () => ({ createTransport: vi.fn().mockReturnValue({ sendMail: mockSendMail }), // TDZ! })); // ✓ Đúng — vi.hoisted() initialises TRƯỚC khi vi.mock() chạy const mockSendMail = vi.hoisted(() => vi.fn()); vi.mock('nodemailer', () => ({ createTransport: vi.fn().mockReturnValue({ sendMail: mockSendMail }), // ✓ }));
★

Mock path resolution: Path trong vi.mock() được resolve relative đến test file, không phải source file. Ví dụ: test tại src/workers/__tests__/ mock config bằng '../../config/env' (2 levels up), không phải '../../../config/env' (3 levels — sẽ resolve sai).

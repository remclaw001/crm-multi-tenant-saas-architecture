Vai trò từng thành phần

component roles
plugin.manifest.json
│
├── Identity          name, version, description
├── Requirements      minSystemVersion, required tier(s)
├── Dependencies      plugins needed, services needed
├── Permissions       db:read, db:write, email:send, storage:write
├── Resources         maxMemory, maxCPU, maxDBQueries per request
├── Events            emits: [...], listens: [...]
├── Routes            prefix, middleware requirements
└── Hooks             extension points provided & consumed


core.js  ─────── Business logic (stateless singleton)
                  │
                  ├── KHÔNG giữ state giữa các requests
                  ├── Nhận ExecutionContext → xử lý → trả kết quả
                  ├── Gọi DB, Cache qua context (tenant-scoped)
                  └── 1 instance phục vụ TẤT CẢ tenants


routes.js ────── Khai báo REST endpoints
                  │
                  ├── GET  /api/v1/sales          → core.getSales(ctx)
                  ├── POST /api/v1/sales          → core.createSale(ctx)
                  ├── GET  /api/v1/calls/:id      → core.getCall(ctx)
                  └── Routes tự động đăng ký khi plugin được load


events.js ────── Event handlers
                  │
                  ├── Sync handlers  (in-process, cùng request)
                  │   └── customer.created → tạo sales opportunity
                  │
                  └── Async handlers (distributed, qua MQ)
                      └── customer.created → gửi notification cho sales team


hooks.js ─────── Extension points
                  │
                  ├── Provides:  hook points mà plugins KHÁC có thể hook vào
                  │   └── before:saleCreate, after:saleCreate
                  │
                  └── Consumes:  hook vào plugins KHÁC
                      └── filter:customerList → thêm sales data vào customer list


migrations/ ──── DB schema cho plugin
                  │
                  ├── Chạy khi plugin được ENABLE cho 1 tenant
                  ├── Rollback khi plugin bị DISABLE
                  └── Mỗi file = 1 migration step (transactional)


ui/ ──────────── Frontend components
                  │
                  ├── remoteEntry.js → Module Federation entry point
                  ├── Shell app load UI này dynamically
                  └── Chỉ load khi tenant đã enable plugin


config.schema ── Tenant configuration validation
                  │
                  ├── Mỗi tenant có config RIÊNG cho plugin này
                  ├── Schema đảm bảo config hợp lệ
                  └── VD: Twilio SID, email templates, SLA thresholds

Tạo Plugin mới — Phase 5 thực tế

Không có CLI tool hay artifact registry. Tạo plugin = thêm NestJS module vào src/plugins/cores/, khai báo manifest, register trong PluginsModule.
create plugin — phase 5
Ví dụ: tạo plugin "loyalty-program"

① Khai báo Manifest   src/plugins/manifest/built-in-manifests.ts
│
│  export const LOYALTY_MANIFEST: PluginManifest = {
│    name:         'loyalty-program',
│    version:      '1.0.0',
│    description:  'Points and rewards for customers',
│    dependencies: ['customer-data'],             ← deps phải enabled trước
│    permissions:  ['customers:read', 'customers:write'],
│    limits:       DEFAULT_LIMITS,               ← { timeoutMs: 5000, memoryMb: 50, maxQueries: 50 }
│    hooks: [
│      { event: 'customer.create', type: 'after', priority: 20 },
│    ],
│  };
│
│  // Thêm vào array:
│  export const BUILT_IN_MANIFESTS = [
│    ...,
│    LOYALTY_MANIFEST,
│  ];
│
▼
② Tạo Core   src/plugins/cores/loyalty-program/loyalty-program.core.ts
│
│  @Injectable()
│  export class LoyaltyProgramCore implements IPluginCore, OnModuleInit {
│    readonly manifest = LOYALTY_MANIFEST;
│
│    constructor(
│      private readonly registry: PluginRegistryService,
│      private readonly hookRegistry: HookRegistryService,
│    ) {}
│
│    onModuleInit(): void {
│      this.registry.register(this);
│      this.hookRegistry.register(
│        'loyalty-program',
│        { event: 'customer.create', type: 'after', priority: 20 },
│        async (ctx, data) => { /* award signup points */ },
│      );
│    }
│
│    async getPoints(ctx: IExecutionContext, customerId: string) {
│      // Use ctx.db — QueryInterceptor scopes to tenant automatically
│      return ctx.db.db('loyalty_points').where({ customer_id: customerId });
│    }
│  }
│

③ Tạo Controller   src/plugins/cores/loyalty-program/loyalty-program.controller.ts

│    @Get('points/:customerId')
│    async getPoints(@Param('customerId') id: string,
│                    @CurrentTenant() tenant, @CurrentUser() user, @Req() req) {

│      if (!ctx.enabledPlugins.includes('loyalty-program'))
│        throw new ForbiddenException('loyalty-program plugin not enabled');

Cơ chế Scan & Load Plugin

Phase 6+ Roadmap. Phase 1 Discovery (query plugin_registry table) và Hot Deploy qua MQ chưa implement. Phase 5 thực tế: plugin cores là NestJS providers hardcode, load một lần khi app khởi động qua DI container. Self-register bằng cách gọi registry.register(this) trong OnModuleInit.
Khi nào scan xảy ra? (1) Application bootstrap — plugin cores tự đăng ký. (2) Hot deploy (Phase 6+).

PluginInitProcessor — First-Enable Init Job

plugin-init.processor.ts + tenant_plugins
Mục đích:
  Khi một plugin được enable lần đầu tiên cho một tenant, cần chạy
  seed / init data (Phase 6+). Phase 5: tất cả plugin đều no-op,
  nhưng cơ chế đã sẵn sàng để mở rộng.

DB column:  tenant_plugins.initialized_at  TIMESTAMPTZ NULL
  NULL         → chưa init (hoặc đang pending)
  <timestamp> → đã init thành công

QUEUE_PLUGIN_INIT  =  'plugin-init'  (BullMQ)
  concurrency : 5
  attempts    : 3  (retry với exponential backoff, delay 3 s)

Luồng:

  togglePlugin() enable path
    │
    │  SELECT initialized_at FROM tenant_plugins
    │    WHERE tenant_id = $1 AND plugin_name = $2
    │
    ├── initialized_at IS NULL  →  isFirstEnable = true
    │     → pluginInitQueue.add('init', { tenantId, pluginId })
    │     → Response includes  initializing: true
    │
    └── initialized_at NOT NULL → isFirstEnable = false
          → không enqueue
          → Response includes  initializing: false

  BullMQ worker (PluginInitProcessor)
    │
    │  1  Nhận job { tenantId, pluginId }
    │  2  Re-fetch initialized_at (idempotency guard):
    │       nếu đã set → return (job trùng / retry không cần làm gì)
    │  3  Nếu không có row → throw → BullMQ retry
    │       (race condition: INSERT chưa commit khi job bắt đầu)
    │  4  runInitFor(pluginId, tenantId)
    │       Phase 5: tất cả plugin đều no-op
    │       Phase 6+: seed dữ liệu mặc định theo từng plugin
    │  5  UPDATE tenant_plugins SET initialized_at = NOW()
    │       WHERE tenant_id = $1 AND plugin_name = $2
    │
    └── Done

Chú ý quan trọng:
  • Processor chạy ngoài TenantContext (BullMQ worker) →
    phải dùng raw SQL với explicit WHERE tenant_id = $1
    KHÔNG dùng Knex (QueryInterceptor không active)
  • poolRegistry.acquireMetadataConnection() trong try/finally
    để đảm bảo client.release() luôn được gọi

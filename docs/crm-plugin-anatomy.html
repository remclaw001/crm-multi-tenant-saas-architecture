02

Vai trò từng thành phần

component roles
<name>.core.ts ── Business logic (stateless singleton)
                  │
                  ├── implements IPluginCore, OnModuleInit
                  ├── KHÔNG giữ state giữa các requests
                  ├── Nhận ExecutionContext → xử lý → trả kết quả
                  ├── Gọi DB, Cache qua ctx.db / ctx.cache (tenant-scoped tự động)
                  └── 1 instance phục vụ TẤT CẢ tenants


<name>.controller.ts ── REST endpoints
                  │
                  ├── GET  /api/v1/plugins/<name>/...  → sandbox.execute(() => core.method(ctx))
                  ├── Validate enabledPlugins trước khi execute
                  └── Dùng @CurrentTenant(), @CurrentUser() decorators


manifest constant ── Khai báo trong built-in-manifests.ts
                  │
                  ├── name, version, description
                  ├── dependencies: plugins phải enabled trước
                  ├── permissions:  db:read, db:write, ...
                  ├── limits:       timeoutMs, maxQueries per request
                  └── hooks:        events plugin muốn hook vào


hooks (OnModuleInit) ── Extension points
                  │
                  ├── hookRegistry.register(pluginName, { event, type, priority }, handler)
                  ├── type: 'before' | 'after' | 'filter'
                  ├── Chạy in-process, đồng bộ, khi plugin khác emit event
                  └── Handler chỉ chạy nếu plugin đã ENABLED cho tenant đó


migrations ── DB schema
                  │
                  ├── Nằm trong src/db/migrations/ (dùng chung, chạy 1 lần)
                  ├── Mỗi file = 1 Knex migration (up/down)
                  └── Mọi table đều có tenant_id + ENABLE ROW LEVEL SECURITY


Frontend page ── UI layer
                  │
                  ├── app/(plugins)/<name>/page.tsx
                  ├── Bọc trong <PluginGate> — ẩn tự động nếu plugin chưa enabled
                  └── Gọi API qua api-client.ts (tự inject auth headers)

② Tạo Core   src/plugins/cores/loyalty-program/loyalty-program.core.ts
│
│  @Injectable()
│  export class LoyaltyProgramCore implements IPluginCore, OnModuleInit {
│    readonly manifest = LOYALTY_MANIFEST;
│
│    constructor(
│      private readonly registry: PluginRegistryService,
│      private readonly hookRegistry: HookRegistryService,
│    ) {}
│
│    onModuleInit(): void {
│      this.registry.register(this);
│      this.hookRegistry.register(
│        'loyalty-program',
│        { event: 'customer.create', type: 'after', priority: 20 },
│        async (ctx, data) => { /* award signup points */ },
│      );
│    }
│
│    async getPoints(ctx: IExecutionContext, customerId: string) {
│      // Use ctx.db — QueryInterceptor scopes to tenant automatically
│      return ctx.db.db('loyalty_points').where({ customer_id: customerId });
│    }
│  }
│
▼
③ Tạo Controller   src/plugins/cores/loyalty-program/loyalty-program.controller.ts
│
│  @Controller('api/v1/plugins/loyalty-program')
│  export class LoyaltyProgramController {
│    @Get('points/:customerId')
│    async getPoints(@Param('customerId') id: string,
│                    @CurrentTenant() tenant, @CurrentUser() user, @Req() req) {
│      const ctx = await this.contextBuilder.build(tenant, user, req.correlationId);
│      if (!ctx.enabledPlugins.includes('loyalty-program'))
│        throw new ForbiddenException('loyalty-program plugin not enabled');
│      return this.sandbox.execute(
│        () => this.core.getPoints(ctx, id),
│        this.core.manifest.limits.timeoutMs,
│      );
│    }
│  }
│

bootstrap scan flow
NestJS APP BOOTSTRAP  (main.ts → AppModule → ApiModule → PluginsModule)

Step 1: PluginInfraModule khởi tạo (@Global — MUST be first)
│  Singletons created: PluginRegistryService (empty Map), HookRegistryService (empty Map),
│  ExecutionContextBuilder, SandboxService,
│  PluginDependencyService (indexes BUILT_IN_MANIFESTS by name)

│
▼

│  CustomerDataModule, CustomerCareModule, AnalyticsModule, AutomationModule, MarketingModule
│  Each Core class instantiated — constructor injects infra services from @Global
│

Step 3: OnModuleInit() fires for each Core
│
│  Ví dụ — AutomationModule.onModuleInit():

│  │  // 1. Register plugin core                                    │
│  │  this.registry.register(this.core)                             │
│  │    → pluginCores.set('automation', coreInstance)               │
│  │                                                                │
│  │  // 2. Register action handlers                                │
│  │  this.actionRegistry.register(this.webhookHandler)             │
│  │  this.actionRegistry.register(this.customerUpdateHandler)      │
│  │  this.actionRegistry.register(this.caseCreateHandler)          │
│  └────────────────────────────────────────────────────────────────┘

│  Ví dụ — AutomationCore.onModuleInit():
│  ┌────────────────────────────────────────────────────────────────┐
│  │  this.hookRegistry.register(                                   │

│  │  )                                                             │
│  └────────────────────────────────────────────────────────────────┘

Mối quan hệ giữa các Plugin

relationship types
Plugins tương tác qua 4 cơ chế, KHÔNG bao giờ gọi trực tiếp nhau:


① DEPENDENCY (khai báo trong manifest)
─────────────────────────────────────────────
   customer-care  ──depends on──▶  customer-data

   • care KHÔNG THỂ hoạt động nếu customer-data chưa load
   • Hệ thống enforce: load customer-data TRƯỚC care
   • Nếu customer-data bị disable → care cũng bị disable


② ASYNC WORKERS (BullMQ — automation action execution)
─────────────────────────────────────────────
   customer.create event fires
        │
        ▼ (after hook, in-process)
   AutomationCore.fireTriggerEvents()
        │  → query matching triggers
        │  → INSERT automation_action_events (status=pending)
        ▼
   AutomationActionPoller (cron */5s)
        │  → SELECT pending events FOR UPDATE SKIP LOCKED
        │  → UPDATE status=queued
        │  → queue.add(QUEUE_AUTOMATION_ACTIONS, jobData)
        ▼
   AutomationActionProcessor (@Processor BullMQ)
        │  → TenantContext.run({ tenantId, tenantTier })
        │  → actionRegistry.getHandler(action.type).execute(ctx, params)
        ▼
   Handler: WebhookCallHandler | CustomerUpdateFieldHandler | CaseCreateHandler


③ HOOKS (in-process, synchronous)
─────────────────────────────────────────────
   customer-data  ──fires hook──▶  "after:customer.create"
                                            │
                               ┌────────────┴──────────────┐
                               ▼                           ▼
                          customer-care            automation
                       after hook (priority 10)   after hook (priority 20)
                   (handler no-op)             (calls fireTriggerEvents)

   • customer-data.createCustomer() gọi:
     → INSERT customers
     hookRegistry.runAfter('customer.create', ctx, customer)
   • Hook handlers chạy tuần tự theo priority (số nhỏ = chạy trước)
   • Hooks in-process, đồng bộ, không qua MQ


④ SHARED DB TABLE ACCESS (cross-plugin, via ctx.db)
─────────────────────────────────────────────
   automation  ──writes to──▶  support_cases table  ◀──owned by──  customer-care

   • CaseCreateHandler (automation) ghi trực tiếp vào support_cases qua ctx.db
   • Không gọi customer-care core — truy cập thẳng vào DB table dùng chung
   • QueryInterceptor tự scope theo tenant_id — không cần WHERE thủ công
   • ctx.cache (CacheManager) sẵn cho mọi plugin qua ExecutionContext

PluginInitProcessor — First-Enable Init Job

plugin-init.processor.ts + tenant_plugins
Mục đích:
  Khi một plugin được enable lần đầu tiên cho một tenant, cơ chế này
  cho phép chạy seed / init data tuỳ theo plugin. Hiện tại tất cả
  plugin đều no-op nhưng cơ chế đã sẵn sàng để mở rộng.

DB column:  tenant_plugins.initialized_at  TIMESTAMPTZ NULL
  NULL         → chưa init (hoặc đang pending)
  <timestamp> → đã init thành công

QUEUE_PLUGIN_INIT  =  'plugin-init'  (BullMQ)
  concurrency : 5
  attempts    : 3  (retry với exponential backoff, delay 3 s)

Luồng:

  togglePlugin() enable path
    │
    │  SELECT initialized_at FROM tenant_plugins
    │    WHERE tenant_id = $1 AND plugin_name = $2
    │
    ├── initialized_at IS NULL  →  isFirstEnable = true
    │     → pluginInitQueue.add('init', { tenantId, pluginId })
    │     → Response includes  initializing: true
    │
    └── initialized_at NOT NULL → isFirstEnable = false
          → không enqueue
          → Response includes  initializing: false

  BullMQ worker (PluginInitProcessor)
    │
    │  1  Nhận job { tenantId, pluginId }
    │  2  Re-fetch initialized_at (idempotency guard):
    │       nếu đã set → return (job trùng / retry không cần làm gì)
    │  3  Nếu không có row → throw → BullMQ retry
    │       (race condition: INSERT chưa commit khi job bắt đầu)
    │  4  runInitFor(pluginId, tenantId)
    │       Hiện tại: tất cả plugin đều no-op (hook sẵn sàng cho seed data)
    │  5  UPDATE tenant_plugins SET initialized_at = NOW()
    │       WHERE tenant_id = $1 AND plugin_name = $2
    │
    └── Done

Chú ý quan trọng:
  • Processor chạy ngoài TenantContext (BullMQ worker) →
    phải dùng raw SQL với explicit WHERE tenant_id = $1
    KHÔNG dùng Knex (QueryInterceptor không active)
  • poolRegistry.acquireMetadataConnection() trong try/finally
    để đảm bảo client.release() luôn được gọi

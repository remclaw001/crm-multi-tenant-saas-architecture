Vai trò từng thành phầnRole of Each Component

component roles
plugin.manifest.json
│
├── Identity          name, version, description
├── Requirements      minSystemVersion, required tier(s)
├── Dependencies      plugins needed, services needed
├── Permissions       db:read, db:write, email:send, storage:write
├── Resources         maxMemory, maxCPU, maxDBQueries per request
├── Events            emits: [...], listens: [...]
├── Routes            prefix, middleware requirements
└── Hooks             extension points provided & consumed


core.js  ─────── Business logic (stateless singleton)
                  │
                  ├── KHÔNG giữ state giữa các requests
                  ├── Nhận ExecutionContext → xử lý → trả kết quả
                  ├── Gọi DB, Cache qua context (tenant-scoped)
                  └── 1 instance phục vụ TẤT CẢ tenants


routes.js ────── Khai báo REST endpoints
                  │
                  ├── GET  /api/v1/sales          → core.getSales(ctx)
                  ├── POST /api/v1/sales          → core.createSale(ctx)
                  ├── GET  /api/v1/calls/:id      → core.getCall(ctx)
                  └── Routes tự động đăng ký khi plugin được load


events.js ────── Event handlers
                  │
                  ├── Sync handlers  (in-process, cùng request)
                  │   └── customer.created → tạo sales opportunity
                  │
                  └── Async handlers (distributed, qua MQ)
                      └── customer.created → gửi notification cho sales team


hooks.js ─────── Extension points
                  │
                  ├── Provides:  hook points mà plugins KHÁC có thể hook vào
                  │   └── before:saleCreate, after:saleCreate
                  │
                  └── Consumes:  hook vào plugins KHÁC
                      └── filter:customerList → thêm sales data vào customer list


migrations/ ──── DB schema cho plugin
                  │
                  ├── Chạy khi plugin được ENABLE cho 1 tenant
                  ├── Rollback khi plugin bị DISABLE
                  └── Mỗi file = 1 migration step (transactional)


ui/ ──────────── Frontend components
                  │
                  ├── remoteEntry.js → Module Federation entry point
                  ├── Shell app load UI này dynamically
                  └── Chỉ load khi tenant đã enable plugin


config.schema ── Tenant configuration validation
                  │
                  ├── Mỗi tenant có config RIÊNG cho plugin này
                  ├── Schema đảm bảo config hợp lệ
                  └── VD: Twilio SID, email templates, SLA thresholds

Phase 6+ Roadmap. Phase 1 Discovery (query plugin_registry table) và Hot Deploy qua MQ chưa implement. Phase 5 thực tế: plugin cores là NestJS providers hardcode, load một lần khi app khởi động qua DI container. Self-register bằng cách gọi registry.register(this) trong OnModuleInit.
Phase 6+ Roadmap. Phase 1 Discovery (query plugin_registry table) and hot deploy via MQ are not yet implemented. Current Phase 5: plugin cores are hardcoded NestJS providers, loaded once at app startup via DI container. Self-register by calling registry.register(this) in OnModuleInit.
Khi nào scan xảy ra? (1) Application bootstrap — plugin cores tự đăng ký. (2) Hot deploy (Phase 6+).
When does scanning occur? (1) Application bootstrap — plugin cores self-register. (2) Hot deploy (Phase 6+).

Mối quan hệ giữa các PluginRelationships Between Plugins

relationship types
Plugins tương tác qua 4 cơ chế, KHÔNG bao giờ gọi trực tiếp nhau:


① DEPENDENCY (khai báo trong manifest)
─────────────────────────────────────────────
   customer-care  ──depends on──▶  customer-data

   • care KHÔNG THỂ hoạt động nếu customer-data chưa load
   • Hệ thống enforce: load customer-data TRƯỚC care
   • Nếu customer-data bị disable → care cũng bị disable


② EVENTS (Phase 6+ roadmap — async pub/sub qua MQ)
─────────────────────────────────────────────
   Phase 5: plugin-to-plugin communication dùng HOOKS (đồng bộ).
   Phase 6+: MQ-based async events cho cross-plugin notifications.

   customer-data  ──emits──▶  "customer.created" (MQ — Phase 6+)
                                     │
                        ┌────────────┼──────────────┐
                        ▼            ▼              ▼
                   customer-care  automation    analytics
                   (create case)  (run rules)  (update count)

   • Publisher không biết có bao nhiêu subscribers
   • Subscriber có thể thêm/bớt mà không sửa publisher
   • Phase 5: chỉ có Hooks (sync, in-process)


③ HOOKS (in-process, synchronous — Phase 5 thực tế)
─────────────────────────────────────────────
   customer-data  ──fires hook──▶  "before:customer.create"
                                    "after:customer.create"
                                            │
                               ┌────────────┴──────────────┐
                               ▼                           ▼
                          automation               customer-care
                       before hook (priority 5)   after hook (priority 10)
                   (runs TRƯỚC khi insert)     (runs SAU khi insert)
                   Phase 5: no-op              Phase 5: no-op
                   Phase 7+: apply triggers    Phase 7+: create onboarding case

   • customer-data.createCustomer() gọi:
     hookRegistry.runBefore('customer.create', ctx, input)
     → INSERT customers
     hookRegistry.runAfter('customer.create', ctx, customer)
   • Hook handlers chạy tuần tự theo priority (số nhỏ = chạy trước)
   • Khác event: hooks in-process, đồng bộ, không qua MQ


④ SHARED SERVICES (indirect interaction)
─────────────────────────────────────────────
   customer-care  ──uses──▶  EmailService  ◀──uses──  automation
   analytics      ──uses──▶  CacheManager  ◀──uses──  marketing

   • Plugins không biết về nhau
   • Nhưng share cùng infrastructure services
   • Services đảm bảo tenant isolation

    └── automation handler  (priority 5  — runs FIRST)
         Phase 5: no-op. Phase 7+: inspect trigger conditions.

  → INSERT INTO customers

  hookRegistry.runAfter('customer.create')
    └── customer-care handler (priority 10 — runs AFTER insert)
         Phase 5: no-op. Phase 7+: auto-create onboarding support case.


Load Order (topological sort từ built-in-manifests.ts):
┌──────────────────────────────┐   ┌────────────────────────────────────────┐
│           Round 1            │──▶│                  Round 2               │
│  customer-data   analytics   │   │  customer-care  automation  marketing  │
└──────────────────────────────┘   └────────────────────────────────────────┘
customer-care, automation, marketing đều có thể load song song — tất cả deps có sẵn sau Round 1.


Disable Cascade (dựa trên dependency graph thực tế):
  Disable customer-data → cascade disable: customer-care, automation, marketing
  Disable customer-care → no cascade (không plugin nào phụ thuộc care)
  Disable analytics     → cascade disable: automation (automation phụ thuộc analytics)
  Disable automation    → no cascade (leaf node)
  Disable marketing     → no cascade (leaf node)

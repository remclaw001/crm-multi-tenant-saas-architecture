02

Vai trò từng thành phầnRole of Each Component

component roles
plugin.manifest.json
│
├── Identity          name, version, description
├── Requirements      minSystemVersion, required tier(s)
├── Dependencies      plugins needed, services needed
├── Permissions       db:read, db:write, email:send, storage:write
├── Resources         maxMemory, maxCPU, maxDBQueries per request
├── Events            emits: [...], listens: [...]
├── Routes            prefix, middleware requirements
└── Hooks             extension points provided & consumed


core.js  ─────── Business logic (stateless singleton)
                  │
                  ├── KHÔNG giữ state giữa các requests
                  ├── Nhận ExecutionContext → xử lý → trả kết quả
                  ├── Gọi DB, Cache qua context (tenant-scoped)
                  └── 1 instance phục vụ TẤT CẢ tenants


routes.js ────── Khai báo REST endpoints
                  │
                  ├── GET  /api/v1/sales          → core.getSales(ctx)
                  ├── POST /api/v1/sales          → core.createSale(ctx)
                  ├── GET  /api/v1/calls/:id      → core.getCall(ctx)
                  └── Routes tự động đăng ký khi plugin được load


events.js ────── Event handlers
                  │
                  ├── Sync handlers  (in-process, cùng request)
                  │   └── customer.created → tạo sales opportunity
                  │
                  └── Async handlers (distributed, qua MQ)
                      └── customer.created → gửi notification cho sales team


hooks.js ─────── Extension points
                  │
                  ├── Provides:  hook points mà plugins KHÁC có thể hook vào
                  │   └── before:saleCreate, after:saleCreate
                  │
                  └── Consumes:  hook vào plugins KHÁC
                      └── filter:customerList → thêm sales data vào customer list


migrations/ ──── DB schema cho plugin
                  │
                  ├── Chạy khi plugin được ENABLE cho 1 tenant
                  ├── Rollback khi plugin bị DISABLE
                  └── Mỗi file = 1 migration step (transactional)


ui/ ──────────── Frontend components
                  │
                  ├── remoteEntry.js → Module Federation entry point
                  ├── Shell app load UI này dynamically
                  └── Chỉ load khi tenant đã enable plugin


config.schema ── Tenant configuration validation
                  │
                  ├── Mỗi tenant có config RIÊNG cho plugin này
                  ├── Schema đảm bảo config hợp lệ
                  └── VD: Twilio SID, email templates, SLA thresholds

When does scanning occur? (1) Application bootstrap, (2) Hot deploy of a new plugin, (3) Plugin update broadcast from another instance.

Mối quan hệ giữa các PluginRelationships Between Plugins

relationship types
Plugins tương tác qua 4 cơ chế, KHÔNG bao giờ gọi trực tiếp nhau:


① DEPENDENCY (khai báo trong manifest)
─────────────────────────────────────────────
   customer-care  ──depends on──▶  customer-data

   • care KHÔNG THỂ hoạt động nếu customer-data chưa load
   • Hệ thống enforce: load customer-data TRƯỚC care
   • Nếu customer-data bị disable → care cũng bị disable


② EVENTS (loose coupling, pub/sub)
─────────────────────────────────────────────
   customer-data  ──emits──▶  "customer.created"
                                     │
                        ┌────────────┼──────────────┐
                        ▼            ▼              ▼
                   customer-care  automation    analytics
                   (tạo sales)   (send email) (update count)

   • Publisher không biết có bao nhiêu subscribers
   • Subscriber có thể thêm/bớt mà không sửa publisher
   • Sync events (in-process) + Async events (MQ)


③ HOOKS (extension points)
─────────────────────────────────────────────
   customer-data  ──provides hook──▶  "filter:customerList"
                                              │
                                     ┌────────┴────────┐
                                     ▼                 ▼
                              customer-care       analytics
                              (add sales data    (add score
                               to each customer)  to each customer)

   • Plugin A cung cấp hook point
   • Plugin B, C đăng ký handler cho hook đó
   • Khi A chạy đến hook point → gọi tất cả handlers
   • Khác event: hooks MODIFY data (filter/transform)


④ SHARED SERVICES (indirect interaction)
─────────────────────────────────────────────
   customer-care  ──uses──▶  EmailService  ◀──uses──  automation
   analytics      ──uses──▶  CacheManager  ◀──uses──  marketing

   • Plugins không biết về nhau
   • Nhưng share cùng infrastructure services
   • Services đảm bảo tenant isolation

Disable Cascade:
  Disable customer-data → cascade disable: care, analytics, automation, marketing
  Disable customer-care → cascade disable: automation, marketing
  Disable analytics     → no cascade (nothing depends on it)
  Disable marketing     → no cascade (leaf node)

Module Federation cho phép HostShell (React SPA) lazy-load UI của từng plugin như một remote JavaScript module độc lập — không cần bundle chung, không reload trang.
Module Federation allows the HostShell (React SPA) to lazy-load each plugin's UI as an independent remote JavaScript module — no shared bundle required, no page reload.

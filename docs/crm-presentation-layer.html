BOOTSTRAPPING SEQUENCE (thực tế):

  1. User truy cập /login (Next.js SSR)
  2. Zustand persist hydrates from localStorage 'crm-web-auth'
     Token found? → redirect /contacts (đã đăng nhập)
     No token? → render login form
  3. User nhập tenantSlug + email + password
     POST /auth/login → { token, user: { id, name, email, roles },
                          tenant: { id, subdomain, name, tier } }
     setAuth() → Zustand persist → localStorage → router.push('/contacts')
  4. CRM Layout: onFinishHydration() guard → auth check
     GET /api/v1/plugins  (Headers: Authorization + X-Tenant-ID)
     → { enabledPlugins: ["customer-data", "customer-care"] }
  5. Sidebar filter PLUGIN_PAGES theo enabledPlugins:
     customer-data → Contacts, customer-care → Cases
  6. User click "Cases" → Next.js router → /cases
  7. CasesPage renders → <PluginGate plugin="customer-care">
     Check enabledPlugins (TanStack Query cache, stale 5min)
     enabled → render <CasesList />
     disabled → ShieldOff icon + thông báo liên hệ admin
  8. CasesList: GET /api/v1/plugins/customer-care/cases
     queryKey: ['cases', tenantId]
     → L2 Gateway → L3 Plugin → L4 DB → response
  9. Navigation tiếp theo: TanStack Query cache → giảm API calls

OFFLINE-FIRST + SYNC:

  ┌────────────────────────────────────────────────────────────────────────┐
  │  Local SQLite DB ← cache recent data + pending mutations              │
  │                                                                        │
  │  Online:  API calls → response cache to SQLite → render              │
  │  Offline: render from SQLite → mutations queue in pending table       │
  │  Reconnect: sync engine → batch POST pending mutations → API         │
  │                                                                        │
  │  Conflict resolution:                                                 │
  │  ├── Last-write-wins (default, simple)                               │
  │  ├── Server-wins (cho shared resources)                              │
  │  └── User-prompt (cho critical conflicts)                            │
  │                                                                        │
  │  Pending queue:                                                       │
  │  ├── Stored in SQLite: { id, method, url, body, createdAt }         │
  │  ├── Max queue: 1000 operations                                      │
  │  ├── Retry on reconnect: sequential, respect order                   │
  │  └── Fail → flag for user review (không tự động discard)             │
  └────────────────────────────────────────────────────────────────────────┘

Security tại L1 — Client-side Defense

L1 là tuyến phòng thủ đầu tiên. Mặc dù backend (L2-L7) có defense in depth, L1 vẫn cần security riêng để ngăn chặn tấn công phía client: XSS, clickjacking, data leak qua browser, token theft.
L1 security
CSP (Content Security Policy) HEADERS:

  Served by CDN / Nginx cùng HTML:
  Content-Security-Policy:
    default-src 'self';
    script-src  'self' cdn.crm.com;          ← chỉ load JS từ CDN
    style-src   'self' 'unsafe-inline' fonts.googleapis.com;
    img-src     'self' data: *.crm.com s3.amazonaws.com;
    connect-src 'self' api.crm.com wss://ws.crm.com;
    frame-src   'none';                       ← no iframe embed
    object-src  'none';                       ← no Flash/plugins

  → Ngăn inline script injection (XSS), unauthorized external scripts,
    data exfiltration qua connect-src whitelist.


XSS PREVENTION:

  ├── React auto-escapes JSX output (default safe)
  ├── dangerouslySetInnerHTML: FORBIDDEN trong plugin UI code
  │   → ESLint rule: no-dangerously-set-inner-html → build fail
  ├── User input sanitization: DOMPurify cho rich text fields
  ├── URL sanitization: chỉ http/https (block javascript: protocol)
  └── Plugin UI sandboxed: chạy trong Host Shell Error Boundary
      → plugin XSS bug → chỉ ảnh hưởng plugin area, không host


CORS (Cross-Origin Resource Sharing):

  API server response headers:
  Access-Control-Allow-Origin: https://acme.crm.com   ← specific, not *
  Access-Control-Allow-Methods: GET, POST, PUT, DELETE
  Access-Control-Allow-Headers: Authorization, Content-Type, X-Tenant-ID
  Access-Control-Max-Age: 3600                         ← preflight cache 1h

  ├── CORS per-tenant: acme.crm.com chỉ call API với origin acme.crm.com
  ├── Admin console: admin.acme.crm.com → separate CORS config
  ├── Custom domains: crm.initech.io → added to CORS whitelist
  └── * (wildcard) NEVER used → prevent cross-tenant request


JWT STORAGE (thực tế):

  DÙNG: localStorage via Zustand persist middleware
  ├── Web App:   localStorage key = 'crm-web-auth'   (token + tenantId + user)
  ├── Admin:     localStorage key = 'crm-admin-auth' (token + user)
  ├── Zustand onFinishHydration() guard: chờ hydrate xong trước redirect
  ├── Tab close → token còn trong localStorage → không cần re-auth
  └── XSS risk: localStorage có thể bị đọc bởi injected script
      Giảm thiểu: CSP header ngăn inline script injection.

  Roadmap: chuyển sang httpOnly cookie + refresh token để tăng security.


CLICKJACKING PROTECTION:
  X-Frame-Options: DENY
  frame-ancestors 'none' (CSP)
  → App KHÔNG THỂ bị embed trong iframe của site khác


ADDITIONAL:
  ├── Referrer-Policy: strict-origin-when-cross-origin
  ├── X-Content-Type-Options: nosniff
  ├── Strict-Transport-Security: max-age=31536000 (HSTS 1 year)
  ├── Permissions-Policy: camera=(), microphone=(), geolocation=()
  │   (disable browser APIs không cần → giảm attack surface)
  └── Subresource Integrity (SRI): plugin remoteEntry.js có integrity hash
      → CDN bị compromise → hash mismatch → browser refuse to load
      (Roadmap: áp dụng khi MF được enable)

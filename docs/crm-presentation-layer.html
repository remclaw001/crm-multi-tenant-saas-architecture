Nguyên tắc cốt lõi: L1 là lớp DUY NHẤT user nhìn thấy. L1 hoàn toàn client-side (chạy trên browser/mobile device). Mọi tương tác đều gọi API tới L2 Gateway — L1 KHÔNG truy cập database, cache, hay MQ trực tiếp. Plugin UI được load ĐỘNG dựa trên tenant đã enable plugins nào.
Core principle: L1 is the ONLY layer users see. L1 is entirely client-side (runs in browser/mobile device). Every interaction calls the API to L2 Gateway — L1 does NOT directly access database, cache, or MQ. Plugin UI is loaded DYNAMICALLY based on which plugins the tenant has enabled.

Web App — Host Shell + Plugin Remotes

Vai trò: SPA chính mà user truy cập qua browser. Gồm Host Shell (khung ứng dụng) và Plugin Remotes (UI modules load động). Host Shell luôn giống nhau cho mọi tenant — sự khác biệt nằm ở REMOTES nào được load, dựa trên enabledPlugins.
Role: The primary SPA that users access via browser. Consists of Host Shell (application frame) and Plugin Remotes (dynamically loaded UI modules). Host Shell is always the same for every tenant — the difference lies in which REMOTES are loaded, based on enabledPlugins.

BOOTSTRAPPING SEQUENCE:

  1. User navigates to acme.crm.com
  2. CDN serves Host Shell (index.html + main.js, ~200KB gz)
  3. Host Shell checks auth:
     JWT in memory? → skip login
     No JWT? → redirect to /login → user authenticates → JWT pair
  4. Host Shell calls API: GET /api/v1/me/context
     → Backend returns: enabledPlugins, permissions, theme, config
  5. Host Shell builds sidebar nav from enabledPlugins
  6. User clicks "Sales" → URL changes to /sales
  7. Host Shell: /sales maps to customer-care remote
     → dynamic import() → fetch remoteEntry.js from CDN
     → ~50-150KB per plugin remote (gzipped)
     → render SalesPipeline component inside <PluginArea />
  8. SalesPipeline calls API: GET /api/v1/sales → L2 Gateway
     → normal request flow (Gateway → Context → Plugin → DB → response)
  9. Subsequent navigation: remotes cached in browser → instant load

Module Federation — Plugin UI System

Vai trò: Webpack Module Federation cho phép load plugin UI code TẠI RUNTIME mà không cần rebuild Host Shell. Plugin team build UI riêng → publish remoteEntry.js → Host Shell load động. Shared dependencies (React, UI Kit) chỉ load 1 lần.
Role: Webpack Module Federation allows loading plugin UI code AT RUNTIME without rebuilding the Host Shell. Plugin teams build their own UI → publish remoteEntry.js → Host Shell loads dynamically. Shared dependencies (React, UI Kit) are loaded only once.
module federation

DYNAMIC REMOTE LOADING:

  Host Shell không hardcode remotes. Load at runtime:Host Shell does not hardcode remotes. Loaded at runtime:

  async function loadPluginRemote(pluginName, version) {
    const url = `cdn.crm.com/plugins/${pluginName}/v${version}/remoteEntry.js`;

    // 1. Inject script tag
    await loadScript(url);

    // 2. Initialize container
    const container = window[pluginName];
    await container.init(__webpack_share_scopes__.default);

    // 3. Get module factory
    const factory = await container.get('./SalesPipeline');
    return factory();
  }

  → Plugin team deploy UI mới → CDN update → user refresh → load new UIPlugin team deploys new UI → CDN updates → user refreshes → loads new UI

OFFLINE-FIRST + SYNC:

  ┌────────────────────────────────────────────────────────────────────────┐
  │  Local SQLite DB ← cache recent data + pending mutations              │
  │                                                                        │
  │  Online:  API calls → response cache to SQLite → render              │
  │  Offline: render from SQLite → mutations queue in pending table       │
  │  Reconnect: sync engine → batch POST pending mutations → API         │
  │                                                                        │
  │  Conflict resolution:                                                 │
  │  ├── Last-write-wins (default, simple)                               │
  │  ├── Server-wins (cho shared resources)                              │
  │  └── User-prompt (cho critical conflicts)                            │
  │                                                                        │
  │  Pending queue:                                                       │
  │  ├── Stored in SQLite: { id, method, url, body, createdAt }         │
  │  ├── Max queue: 1000 operations                                      │
  │  ├── Retry on reconnect: sequential, respect order                   │
  │  └── Fail → flag for user review (không tự động discardnot auto-discarded)             │

Security tại L1 — Client-side DefenseSecurity at L1 — Client-side Defense

L1 là tuyến phòng thủ đầu tiên. Mặc dù backend (L2-L7) có defense in depth, L1 vẫn cần security riêng để ngăn chặn tấn công phía client: XSS, clickjacking, data leak qua browser, token theft.
L1 is the first line of defense. Although the backend (L2-L7) has defense in depth, L1 still needs its own security to prevent client-side attacks: XSS, clickjacking, data leak via browser, token theft.

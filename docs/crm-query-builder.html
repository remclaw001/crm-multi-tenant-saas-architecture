Kiến trúc DIP — 3-layer Query ArchitectureDIP Architecture — 3-layer Query Architecture

Nguyên tắc: Plugin KHÔNG BAO GIỜ viết raw SQL. Mọi truy vấn đi qua CrmQueryBuilder (fluent API) → QueryAST (DB-agnostic) → ISQLDialect (DB-specific). Plugin chỉ biết IDbContext — không biết đang dùng PostgreSQL, MySQL, hay SQLite.
Principle: Plugins NEVER write raw SQL. Every query goes through CrmQueryBuilder (fluent API) → QueryAST (DB-agnostic) → ISQLDialect (DB-specific). Plugins only know IDbContext — they have no knowledge of whether PostgreSQL, MySQL, or SQLite is in use.
3-layer architecture
3 LAYER TÁCH BIỆT:

  ┌─────────────────────────────────────────────────────────────────┐
  │                                                                 │
  │  Layer 1: CrmQueryBuilder «per-query» (plugin-facing)          │
  │  Fluent API: .where(), .join(), .orderBy(), .limit()...        │
  │  Build → QueryAST. KHÔNG biết DB nào. KHÔNG generate SQL.       │
  │                                                                 │
  │  Layer 2: QueryAST «value object» (shared data model)           │
  │  Pure data: { table, operation, where[], join[],               │
  │               orderBy[], groupBy[], limit, offset,             │
  │               insertData, updateData, returning }              │
  │  Không behavior. Serializable. Testable.                       │
  │                                                                 │
  │  Layer 3: ISQLDialect + ITenantStrategy (DB-specific)           │
  │  AST → DB-specific parameterized SQL + tenant isolation        │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘

  Plugin → CrmQueryBuilder → QueryAST → ISQLDialect ◁── PostgresDialect
                                                       ◁── MySQLDialect
                                                       ◁── SQLiteDialect
                                  │
                                  └──▶ ITenantStrategy ◁── PostgresRLSStrategy
                                                        ◁── MySQLTenantStrategy
                                                        ◁── SQLiteTenantStrategy


TẠI SAO TÁCH NHƯ VẬY:

  ① Security — Plugin không viết SQL = không thể inject.
     Builder tạo AST typed. TenantInjector duyệt AST → inject tenant_id.
     QueryValidator duyệt AST → validate structure. Structural safety.

  ② Multi-DB — Cùng AST, khác SQL output:
     PostgresDialect: $1, $2, RETURNING *, ON CONFLICT, ILIKE
     MySQLDialect:    ?, ?, LAST_INSERT_ID(), ON DUPLICATE KEY, LIKE
     SQLiteDialect:   ?, ?, last_insert_rowid(), limited feature set

  ③ DX (Developer Experience) — Fluent, composable, testable:
     ctx.db.query('customers').where({ status: 'active' }).limit(20)
     IDE autocomplete. Type-safe. Mock ctx.db cho unit test.


CLASS DIAGRAM TỔNG THỂ:

  ┌───────────────────────────────────────────────────────────────────┐

  │  │    updateData, returning }                                     │
  │  │                                                                │
  │  ├──▶ consumed by ITenantStrategy.injectTenant(ast, tenantId)      │

DEPENDENCY INJECTION — bootstrap:

CrmQueryBuilder — Plugin-facing fluent API

  ┌──────────────────────────────────────────────────────────────────┐
  │ «per-query» CrmQueryBuilder                                     │
  │──────────────────────────────────────────────────────────────────│
  │ - _table: string                                                 │
  │ - _operation: 'SELECT' | 'INSERT' | 'UPDATE' | 'DELETE'         │

  │ + whereRaw(expr, params)     parameterized only                  │
  │ + orWhere(conditions)        OR group                             │
  │ + join(table, on, type?)     INNER/LEFT/RIGHT JOIN                │
  │ + leftJoin(table, on)        LEFT JOIN shorthand                  │
  │ + orderBy(field, dir)        ORDER BY                             │
  │ + groupBy(fields)            GROUP BY                             │
  │ + having(conditions)         HAVING (after GROUP BY)              │

  │ + offset(n)                  OFFSET n                             │

  │ INTERNAL (plugin không gọi trực tiếp)                            │
  │ - toAST()      → QueryAST        build abstract syntax tree        │
  └──────────────────────────────────────────────────────────────────┘


IDbContext — entry points trên TenantScopedDB:

  ctx.db.query(table)              → CrmQueryBuilder (SELECT mode)
  ctx.db.insert(table, data)       → auto INSERT, return record
  ctx.db.insertMany(table, rows)   → batch INSERT, return records
  ctx.db.update(table, where, data) → auto UPDATE, return count
  ctx.db.delete(table, where)      → auto DELETE, return count
  ctx.db.transaction(fn)           → BEGIN/COMMIT/ROLLBACK wrapper
  ctx.db.queryBuilder(table)       → CrmQueryBuilder (generic mode)

Implementations — PostgreSQL vs MySQL vs SQLite

dialect comparison
ISQLDialect — 3 implementations

  ┌──────────────────────┬───────────────────┬──────────────────┬──────────────────┐
  │ Feature              │ PostgresDialect   │ MySQLDialect      │ SQLiteDialect    │
  ├──────────────────────┼───────────────────┼──────────────────┼──────────────────┤
  │ Placeholder          │ $1, $2, $3        │ ?, ?, ?          │ ?, ?, ?          │
  │ Identifier escape    │ "column"          │ `column`         │ "column"         │
  │ RETURNING            │ RETURNING *       │ ✗ (emulated)     │ ✗ (emulated)     │
  │ INSERT → get id      │ RETURNING id      │ LAST_INSERT_ID() │ last_insert_rowid│
  │ Upsert               │ ON CONFLICT DO    │ ON DUPLICATE KEY │ ON CONFLICT DO   │
  │ JSON access          │ col->>'key'       │ JSON_EXTRACT()   │ json_extract()   │
  │ ILIKE (case-insens.) │ ILIKE             │ LIKE (default CI)│ LIKE + COLLATE   │
  │ Boolean              │ TRUE / FALSE      │ 1 / 0            │ 1 / 0            │
  │ Array                │ ANY(ARRAY[...])   │ ✗ (JSON array)   │ ✗ (JSON array)   │
  │ Full-text search     │ to_tsvector()     │ MATCH AGAINST    │ FTS5             │
  └──────────────────────┴───────────────────┴──────────────────┴──────────────────┘


ITenantStrategy — 3 implementations

  ┌──────────────────────┬─────────────────────┬───────────────────┬──────────────────┐
  │ Feature              │ PostgresRLSStrategy │ MySQLStrategy     │ SQLiteStrategy   │
  ├──────────────────────┼─────────────────────┼───────────────────┼──────────────────┤
  │ AST injection        │ ✓ WHERE/INSERT      │ ✓ WHERE/INSERT    │ ✓ WHERE/INSERT   │
  │ Connection config    │ SET app.tenant      │ SET @app_tenant   │ N/A (in-process) │
  │ DB-level isolation   │ ✓ RLS policies       │ ✗                 │ ✗                │
  │ Defense layers       │ 3 (AST+RLS+audit)   │ 2 (AST+audit)     │ 1 (AST only)     │
  ├──────────────────────┼─────────────────────┼───────────────────┼──────────────────┤
  │ Use case             │ Production primary  │ Enterprise custom │ Unit test        │
  └──────────────────────┴─────────────────────┴───────────────────┴──────────────────┘

  MySQL không có RLS → chỉ 2 tầng bảo vệ.
  ITenantStrategy.supportsDatabaseLevelIsolation() trả false
  → hệ thống enforce stricter QueryValidator rules để bù.


Ví dụ — cùng AST, khác output:

  AST: { table: 'sales', op: 'INSERT',
         insertData: { title: 'Deal', amount: 50000 },
         returning: ['*'] }

  PostgresDialect:
  INSERT INTO sales (tenant_id, title, amount) VALUES ($1, $2, $3) RETURNING *
  params: ['acme-123', 'Deal', 50000]

  MySQLDialect:
  INSERT INTO sales (tenant_id, title, amount) VALUES (?, ?, ?)
  params: ['acme-123', 'Deal', 50000]
  + follow-up: SELECT * FROM sales WHERE id = LAST_INSERT_ID()

  SQLiteDialect:
  INSERT INTO sales (tenant_id, title, amount) VALUES (?, ?, ?)
  params: ['acme-123', 'Deal', 50000]
  + follow-up: SELECT * FROM sales WHERE rowid = last_insert_rowid()

Query Pipeline — 7 bước tự độngQuery Pipeline — 7 Automatic Steps

Mọi query đi qua 7 bước. Plugin chỉ thấy bước 1 (gọi builder) và bước 7 (nhận result). Bước 2→6 hoàn toàn tự động, plugin KHÔNG CAN THIỆP được.
Every query passes through 7 steps. Plugins only see step 1 (calling the builder) and step 7 (receiving the result). Steps 2→6 are fully automatic — plugins CANNOT intervene.
full pipeline
PLUGIN CODE:

  const sales = await ctx.db.query('sales')
    .where({ status: 'new' })
    .select(['id', 'title', 'amount'])
    .orderBy('created_at', 'desc')
    .limit(20)
    .execute();                        ← terminal method triggers pipeline


7 BƯỚC TỰ ĐỘNG:

  ┌─────────────────────────────────────────────────────────────────────────┐
  │                                                                         │
  │  ① CrmQueryBuilder.toAST()                           DB-agnostic    │
  │  │  Chain methods → internal state → QueryAST                          │
  │  │  { table: 'sales', operation: 'SELECT',                             │
  │  │    select: ['id', 'title', 'amount'],                               │
  │  │    where: [{ field: 'status', op: '=', value: 'new' }],            │
  │  │    orderBy: [{ field: 'created_at', dir: 'desc' }],                │
  │  │    limit: 20 }                                                       │
  │  ▼                                                                      │
  │  ② ITenantStrategy.injectTenant(ast, tenantId)    Strategy Pattern  │
  │  │  PostgresRLS / MySQL / SQLite: inject tenant_id vào AST.where[]     │
  │  │  INSERT: thêm tenant_id vào data columns                            │
  │  │  JOIN: inject tenant_id cho MỌI joined table                        │
  │  ▼                                                                      │
  │  ③ QueryValidator.validate(ast)                      DB-agnostic    │
  │  │  6 safety checks trên AST:                                           │
  │  │  ✓ tenant_id present?  ✓ Table in allowedTables?                    │
  │  │  ✓ No DDL?  ✓ No system tables?  ✓ All parameterized?              │
  │  │  ✓ JOIN tables have tenant filter?                                   │
  │  │  ANY FAIL → SecurityError thrown, query BLOCKED.                     │
  │  ▼                                                                      │
  │  ④ QueryCounter.increment()                          DB-agnostic    │
  │  │  count++ → [n/50]. n ≥ 50 → QueryLimitError.                        │
  │  ▼                                                                      │
  │  ⑤ ISQLDialect.generateSelect(ast)                  Strategy Pattern  │
  │  │  PostgresDialect → $1, $2, RETURNING                                │
  │  │  MySQLDialect    → ?, ?, LAST_INSERT_ID                             │
  │  │  { text: "SELECT id, title, amount FROM sales                       │
  │  │           WHERE tenant_id = $1 AND status = $2                      │
  │  │           ORDER BY created_at DESC LIMIT $3",                       │
  │  │    params: ['acme-123', 'new', 20] }                                │
  │  │                                                                      │
  │  │  100% parameterized. Zero interpolation. SQL injection = impossible.│
  │  ▼                                                                      │
  │  ⑥ IConnectionPool.query(text, params)               abstracted     │
  │  │  pg driver / mysql2 driver / better-sqlite3 driver                   │
  │  │  Connection đã configured bởi ITenantStrategy.configureConnection() │
  │  ▼                                                                      │
  │  ⑦ QueryLogger.log()                                 DB-agnostic    │
  │  │  { table, op, duration, rows, tenant, plugin, requestId }           │
  │  │  duration > 100ms → WARN.  > 500ms → ALERT.                        │
  │  ▼                                                                      │
  │  Return Result → plugin nhận Row[]                                    │
  └─────────────────────────────────────────────────────────────────────────┘


TIMING (typical, PostgreSQL):

  ① toAST()                 ~0.01ms   in-memory object build
  ② ITenantStrategy.inject   ~0.01ms   AST node manipulation
  ③ QueryValidator            ~0.02ms   6 rule checks on AST
  ④ QueryCounter              ~0.001ms  increment integer
  ⑤ ISQLDialect.generate      ~0.02ms   AST → string generation
  ⑥ DB round-trip             ~3-15ms   network + query execution
  ⑦ QueryLogger               ~0.01ms   async log emit
  ─────────────────────────────────────
  Pipeline overhead (①→⑤⑦): ~0.07ms    negligible vs ⑥ DB round-trip


DB-AGNOSTIC vs DB-SPECIFIC:

  ┌─────────────────────────────────┬──────────────────────────────────┐
  │ DB-AGNOSTIC (không đổi khi swap)│ DB-SPECIFIC (swap via interface) │
  ├─────────────────────────────────┼──────────────────────────────────┤
  │ CrmQueryBuilder (fluent API)   │ ISQLDialect implementations      │
  │ QueryAST (data structure)      │ ITenantStrategy implementations  │
  │ QueryValidator (AST rules)     │ IMigrationDialect implementations│
  │ QueryCounter (budget)          │ IConnectionPool implementations  │
  │ QueryLogger (observability)    │ DB driver (pg, mysql2, sqlite3)  │
  │ Plugin code (100%)             │                                  │
  ├─────────────────────────────────┼──────────────────────────────────┤
  │ ~70% codebase                   │ ~30% codebase                   │
  └─────────────────────────────────┴──────────────────────────────────┘

SQL Generation — AST → Parameterized SQL

Nguyên tắc: SQLGenerator KHÔNG BAO GIỜ interpolate giá trị vào SQL string. Mọi value → $N placeholder. Params array truyền riêng cho pg driver. SQL injection = impossible ở tầng structural.
Principle: SQLGenerator NEVER interpolates values into the SQL string. Every value → $N placeholder. Params array is passed separately to the pg driver. SQL injection = impossible at the structural level.
SQL generation examples
① SELECT — cơ bản

  Plugin:
  ctx.db.query('customers')
    .where({ status: 'active', tier: 'premium' })
    .select(['id', 'name', 'email'])
    .orderBy('name', 'asc')
    .limit(50)
    .execute()

  AST sau TenantInjector:
  { table: 'customers', operation: 'SELECT',
    select: ['id', 'name', 'email'],
    where: [
      { field: 'tenant_id', op: '=', value: 'acme-123' },
      { field: 'status', op: '=', value: 'active' },
      { field: 'tier', op: '=', value: 'premium' }
    ],
    orderBy: [{ field: 'name', dir: 'ASC' }],
    limit: 50 }

  Generated SQL:
  SELECT id, name, email FROM customers
  WHERE tenant_id = $1 AND status = $2 AND tier = $3
  ORDER BY name ASC LIMIT $4
  params: ['acme-123', 'active', 'premium', 50]


② INSERT — single record

  Plugin:
  ctx.db.insert('sales', {
    title: 'Enterprise Deal',
    amount: 50000,
    customer_id: 'cust-42',
    stage: 'qualification'
  })

  Generated SQL:
  INSERT INTO sales (tenant_id, title, amount, customer_id, stage)
  VALUES ($1, $2, $3, $4, $5)
  RETURNING *
  params: ['acme-123', 'Enterprise Deal', 50000, 'cust-42', 'qualification']

  tenant_id auto-injected as first column. Plugin KHÔNG specify.


③ INSERT MANY — batch

  Plugin:
  ctx.db.insertMany('contacts', [
    { name: 'Alice', email: '[email protected]' },
    { name: 'Bob', email: '[email protected]' },
    { name: 'Carol', email: '[email protected]' }
  ])

  Generated SQL:
  INSERT INTO contacts (tenant_id, name, email)
  VALUES ($1, $2, $3), ($4, $5, $6), ($7, $8, $9)
  RETURNING *
  params: ['acme-123', 'Alice', '[email protected]',
           'acme-123', 'Bob', '[email protected]',
           'acme-123', 'Carol', '[email protected]']

  tenant_id injected cho MỖI row. 1 round-trip cho N rows.


④ UPDATE — with conditions

  Plugin:
  ctx.db.update('sales',
    { id: 'sale-42' },                          ← WHERE
    { stage: 'closed-won', closed_at: new Date() }  ← SET
  )

  Generated SQL:
  UPDATE sales SET stage = $1, closed_at = $2
  WHERE tenant_id = $3 AND id = $4
  RETURNING *
  params: ['closed-won', '2026-02-14T...', 'acme-123', 'sale-42']


⑤ DELETE — with conditions

  Plugin:
  ctx.db.delete('contacts', { id: 'contact-99' })

  Generated SQL:
  DELETE FROM contacts
  WHERE tenant_id = $1 AND id = $2
  params: ['acme-123', 'contact-99']

  Soft delete? Plugin quyết định: ctx.db.update(table, where, { deleted_at: now })
  Hoặc hard delete: ctx.db.delete(). Cả 2 đều bị tenant_id lock.


⑥ TRANSACTION — multi-statement atomic

  Plugin:
  const result = await ctx.db.transaction(async (trx) => {
    // trx là IDbContext scoped trong transaction
    const sale = await trx.insert('sales', { title: 'Big Deal', amount: 100000 });
    await trx.update('customers', { id: 'cust-42' }, { last_sale: sale.id });
    await trx.insert('activities', { type: 'sale_created', ref_id: sale.id });
    return sale;
  });

  Generated SQL sequence:
  BEGIN
  INSERT INTO sales (tenant_id, title, amount) VALUES ($1,$2,$3) RETURNING *
  UPDATE customers SET last_sale = $1 WHERE tenant_id = $2 AND id = $3 RETURNING *
  INSERT INTO activities (tenant_id, type, ref_id) VALUES ($1,$2,$3) RETURNING *
  COMMIT

  Nếu bất kỳ statement nào throw → auto ROLLBACK.
  Cùng 1 connection trong toàn bộ transaction (no pool release giữa chừng).
  Mỗi statement vẫn đi qua full pipeline: inject → validate → count → log.

advanced patterns
① JOIN — auto-inject tenant_id cho CẢ HAI bảng

  Plugin:
  ctx.db.query('sales')
    .select(['sales.title', 'sales.amount', 'customers.name'])
    .join('customers', { 'sales.customer_id': 'customers.id' })
    .where({ 'sales.stage': 'closed-won' })
    .orderBy('sales.amount', 'desc')
    .limit(10)
    .execute()

  Generated SQL:
  SELECT sales.title, sales.amount, customers.name
  FROM sales
  INNER JOIN customers ON sales.customer_id = customers.id
    AND customers.tenant_id = $1      ← auto-injected cho joined table
  WHERE sales.tenant_id = $2           ← auto-injected cho main table
    AND sales.stage = $3
  ORDER BY sales.amount DESC LIMIT $4
  params: ['acme-123', 'acme-123', 'closed-won', 10]

  CRITICAL: TenantInjector inject tenant_id cho MỌI table trong JOIN.
  Nếu không → cross-tenant data leak qua joined table.
  QueryValidator check: mọi JOIN table PHẢI có tenant_id filter. Thiếu → BLOCK.

⑥ whereRaw — parameterized escape hatch

  Plugin:
  ctx.db.query('customers')

Security Layer — 3 tầng bảo vệSecurity Layer — 3 Defence Layers

Defense in Depth: Nếu 1 tầng có bug, 2 tầng còn lại vẫn chặn. Xác suất cả 3 fail đồng thời ≈ 0.
Defense in Depth: If one layer has a bug, the other two still block. Probability of all 3 failing simultaneously ≈ 0.
3-layer security
TẦNG 1 — CrmQueryBuilder + ITenantStrategy (Application Level)

  Plugin code → CrmQueryBuilder → AST → ITenantStrategy.injectTenant().

  Bảo vệ:
  ├── Plugin KHÔNG viết SQL trực tiếp → không thể inject
  ├── Builder tạo AST typed → structural safety
  ├── ITenantStrategy.injectTenant() thêm tenant_id vào AST TRƯỚC KHI generate SQL
  ├── tenant_id lấy từ ExecutionContext (set bởi Gateway, not plugin)
  └── Plugin KHÔNG THỂ override tenant_id trong AST

  Attack vector bị chặn:
  ✗ Plugin cố set tenant_id = 'other-tenant' trong where:
    → ITenantStrategy LUÔN prepend tenant_id = ctx.tenantId
    → QueryValidator verify: tenant_id value = ctx.tenantId. Mismatch → BLOCK.

  ✗ Plugin cố dùng .whereRaw('tenant_id = $1', ['other-tenant']):
    → QueryValidator scan AST: whereRaw node chứa 'tenant_id' → BLOCK.
    → Lý do: plugin không bao giờ được tự specify tenant_id.

  ✗ Plugin cố truy cập table không thuộc manifest:
    → QueryValidator check table against plugin.manifest.allowedTables.
    → customer-care plugin query 'plugin_registry' → BLOCK.


TẦNG 2 — QueryValidator (Validation Level)

  6 rules, kiểm tra SAU inject, TRƯỚC execute:

  ┌──────────────────────────────────────────────────────────────────┐
  │ Rule                        │ Chặn gì                          │
  ├──────────────────────────────────────────────────────────────────┤
  │ 1. tenant_id in WHERE       │ Cross-tenant data access          │
  │ 2. Table in allowedTables   │ Unauthorized table access         │
  │ 3. No DDL (ALTER/DROP)      │ Schema destruction                │
  │ 4. No system tables         │ Metadata leak (pg_catalog)        │
  │ 5. All values parameterized │ SQL injection                     │
  │ 6. JOIN tables have tenant  │ Cross-tenant via JOIN             │
  ├──────────────────────────────────────────────────────────────────┤
  │ ANY rule FAIL → SecurityError thrown, query BLOCKED.            │
  │ Log: security_alert + full AST dump for forensic.              │
  └──────────────────────────────────────────────────────────────────┘


TẦNG 3 — PostgreSQL RLS (Database Level)

  Backup layer. Nếu Tầng 1 + Tầng 2 đều có bug → RLS vẫn chặn.

  ┌──────────────────────────────────────────────────────────────────┐
  │ CREATE POLICY tenant_isolation ON sales                        │
  │   USING (tenant_id = current_setting('app.current_tenant'));    │
  │                                                                  │
  │ ConnectionConfigurator: SET app.current_tenant = 'acme-123'    │
  │ trên mỗi connection TRƯỚC KHI query chạy.                     │
  │                                                                  │
  │ Kết quả:                                                         │
  │ ├── PostgreSQL ENGINE filter rows theo RLS policy                │
  │ ├── Kể cả query thiếu tenant_id trong WHERE                     │
  │ │   → RLS tự thêm filter ở engine level                         │
  │ ├── Kể cả ITenantStrategy inject SAI tenant_id                   │
  │ │   → RLS override: chỉ trả rows match current_setting          │
  │ └── RLS evaluated by DB engine, KHÔNG bypass từ application     │
  └──────────────────────────────────────────────────────────────────┘


BẢNG TÓM TẮT — Mỗi attack cần vượt 3 tầng:

  ┌────────────────────────────────────────────────────────────────┐
  │ Attack                │ Tầng 1     │ Tầng 2     │ Tầng 3     │
  │                       │ QB+Strategy│ Validator  │ RLS        │

Performance — Cache, N+1, Prepared Statements

performance patterns
① QUERY RESULT CACHING — 2-layer

  Plugin gọi ctx.db.query() → pipeline chạy → SQL thực thi.
  Nhưng NẾU kết quả đã có trong cache:

  ┌──────────────────────────────────────────────────────────────────┐
  │  ctx.db.query('customers').where({ id: 'cust-42' }).first()    │
  │       │                                                         │
  │       ▼                                                         │
  │  TenantScopedDB check cache TRƯỚC pipeline:                    │
  │       │                                                         │
  │       ├──▶ L1 Memory (0.01ms) → HIT? → return cached           │
  │       │                                                         │
  │       ├──▶ L2 Redis (0.5ms) → HIT? → populate L1, return      │
  │       │                                                         │
  │       └──▶ MISS → run full pipeline → cache result             │
  │            cache key: t:acme-123:sales:SELECT:hash(AST)        │
  └──────────────────────────────────────────────────────────────────┘

  Cache invalidation:
  - INSERT/UPDATE/DELETE trên table → invalidate all cached SELECTs cho table đó
  - Invalidation qua Redis Pub/Sub → tất cả instances clear L1
  - TTL per-tier: Basic 5min, Premium 10min, Enterprise 15min

  Cache scope:
  - Cache key LUÔN chứa tenant_id prefix → zero cross-tenant cache leak
  - Key format: t:{tenantId}:{plugin}:{table}:{hash(AST)}
  - Plugin A cache ≠ Plugin B cache (khác plugin prefix)


② N+1 DETECTION + PREVENTION

  QueryCounter không chỉ đếm — nó phát hiện PATTERN:

  ┌──────────────────────────────────────────────────────────────────┐
  │  N+1 Pattern detected:                                        │
  │                                                                  │
  │  Request req-789:                                                │
  │    query #1: SELECT * FROM sales WHERE tenant_id=$1 LIMIT 20   │
  │    query #2: SELECT * FROM customers WHERE id=$1 AND tenant..  │
  │    query #3: SELECT * FROM customers WHERE id=$1 AND tenant..  │
  │    query #4: SELECT * FROM customers WHERE id=$1 AND tenant..  │
  │    ...                                                           │
  │    query #21: SELECT * FROM customers WHERE id=$1 AND tenant.. │
  │                                                                  │
  │  QueryLogger detects: 20 queries same table+structure, diff id │
  │  → WARNING: N+1 detected on 'customers' (20 queries)           │
  │  → Log: suggestion "Use .whereIn() or JOIN instead"            │
  └──────────────────────────────────────────────────────────────────┘

  Giải pháp plugin nên dùng:

  Thay vì (N+1):
  const sales = await ctx.db.query('sales').limit(20).execute();
  for (const sale of sales) {
    sale.customer = await ctx.db.query('customers')
      .where({ id: sale.customer_id }).first();   ← 20 queries!
  }

  Dùng whereIn (2 queries):
  const sales = await ctx.db.query('sales').limit(20).execute();
  const customerIds = sales.map(s => s.customer_id);
  const customers = await ctx.db.query('customers')
    .whereIn('id', customerIds).execute();         ← 1 query!

  Hoặc JOIN (1 query):
  const result = await ctx.db.query('sales')
    .select(['sales.*', 'customers.name as customer_name'])
    .join('customers', { 'sales.customer_id': 'customers.id' })
    .limit(20).execute();                          ← 1 query!


③ PREPARED STATEMENT CACHING

  PostgreSQL (via pg driver) tự cache prepared statements:

  First call:
    PREPARE stmt_abc AS SELECT * FROM sales WHERE tenant_id=$1 AND status=$2
    EXECUTE stmt_abc('acme-123', 'new')

  Subsequent calls (same query shape):
    EXECUTE stmt_abc('acme-123', 'active')      ← skip PREPARE, reuse plan

  Benefit: ~30% faster cho repeated query shapes.
  CrmQueryBuilder giúp maximize prepared stmt reuse vì:
  - Cùng builder chain → cùng AST shape → cùng SQL template
  - Chỉ khác params → PostgreSQL reuse execution plan

Migration Paradox — Shared DB

Khi enable/disable plugin
trong Shared Database

Enabling/Disabling Plugins
in a Shared Database

Tables dùng chung nhưng plugins enable riêng — Migration Engine giải quyết nghịch lý này như thế nào?

Tables are shared, yet plugins are enabled per-tenant — how does the Migration Engine resolve this paradox?

00

Nghịch lý cốt lõiThe Core Paradox

the paradox
BÀI TOÁN:

  Shared Data DB chứa data CỦA MỌI standard tenant trong CÙNG tables.

  ┌────────────────────────────────────────────────────────────────────┐
  │  sales table (1 bảng duy nhất, shared)                              │
  │                                                                    │
  │  id       │ tenant_id    │ title            │ amount              │
  │  ─────────┼──────────────┼──────────────────┼──────────────────   │
  │  sale-42  │ acme-123     │ Enterprise deal  │ 50000               │
  │  sale-43  │ acme-123     │ Startup package  │ 5000                │
  │  sale-88  │ globex-456   │ Widget order     │ 12000               │
  │  sale-91  │ wayne-789    │ Tech upgrade     │ 80000               │
  └────────────────────────────────────────────────────────────────────┘

  Bảng sales thuộc plugin customer-care.
  3 tenants (ACME, GLOBEX, WAYNE) đều đã enable customer-care.
  Data của cả 3 nằm TRONG CÙNG BẢNG.


CÂU HỎI:

  ① Khi ACME enable customer-care (tenant đầu tiên):
     → Migration tạo table sales. OK rõ ràng.

  ② Khi GLOBEX enable customer-care (tenant thứ hai):
     → Table sales ĐÃ TỒN TẠI. Migration làm gì?
     → Có chạy CREATE TABLE lại không? Lỗi không?

  ③ Khi ACME disable customer-care:
     → ACME không dùng nữa. DROP TABLE sales?
     → Nhưng GLOBEX và WAYNE vẫn dùng! DROP = mất data họ!

  ④ Khi upgrade customer-care v1.5 → v2.0 (thêm cột mới):
     → ALTER TABLE sales ADD COLUMN pipeline_stage TEXT;
     → Chỉ ACME upgrade, GLOBEX còn v1.5 → GLOBEX thấy cột mới?


ĐÁP ÁN TỔNG QUÁT:

  ┌────────────────────────────────────────────────────────────────────┐
  │                                                                    │
  │  Migration Engine TÁCH migration thành 2 loại:                    │
  │                                                                    │
  │  Schema Migrations (DDL) — CREATE TABLE, ALTER TABLE, ADD INDEX    │
  │  → Chạy 1 LẦN cho toàn hệ thống.                                 │
  │  → Idempotent: CREATE TABLE IF NOT EXISTS.                       │
  │  → Không bao giờ DROP TABLE khi disable.                          │
  │  → Track ở level: (plugin, version) — không có tenant.            │
  │                                                                    │
  │  Tenant Migrations (DML) — seed data, default configs, RLS        │
  │  → Chạy PER TENANT mỗi lần enable.                               │
  │  → Track ở level: (tenant, plugin, version).                     │
  │  → Rollback khi disable = xóa DATA (rows), không xóa TABLE.     │
  │                                                                    │
  └────────────────────────────────────────────────────────────────────┘

MỖI PLUGIN CÓ 2 THƯ MỤC MIGRATION:

  │   │   ├── 001_create_sales.sql
  │   │   ├── 002_create_call_logs.sql
  │   │   ├── 003_create_tickets.sql

  │       └── 003_create_default_pipeline.sql
  │

CHI TIẾT TỪNG LOẠI:


  ┌─ SCHEMA MIGRATIONS (DDL) ──────────────────────────────────────────────┐
  │                                                                        │
  │  Purpose:  Tạo/sửa cấu trúc bảng (tables, indexes, constraints)      │
  │  Scope:    Toàn Shared Data DB — không gắn tenant nào                  │
  │  Runs:     1 lần duy nhất cho mỗi version                             │
  │  Track:    table schema_migrations trong Metadata DB                   │
  │                                                                        │
  │  ┌──────────────────────────────────────────────────────────────────┐ │
  │  │  schema_migrations                                               │ │
  │  │                                                                  │ │
  │  │  plugin_name  │ version │ file                   │ executed_at   │ │
  │  │  ─────────────┼─────────┼────────────────────────┼─────────────  │ │
  │  │  cust-care    │ 001     │ create_sales.sql       │ 2025-06-15   │ │
  │  │  cust-care    │ 002     │ create_call_logs.sql   │ 2025-06-15   │ │
  │  │  cust-care    │ 003     │ create_tickets.sql     │ 2025-06-15   │ │
  │  │  cust-care    │ 004     │ add_indexes.sql        │ 2025-06-15   │ │
  │  │                                                                  │ │
  │  │  Không có cột tenant_id. 1 record per migration file.           │ │
  │  └──────────────────────────────────────────────────────────────────┘ │
  │                                                                        │
  │  SQL example:                                                        │
  │  ┌──────────────────────────────────────────────────────────────────┐ │
  │  │  -- schema/001_create_sales.sql                                  │ │
  │  │                                                                  │ │
  │  │  CREATE TABLE IF NOT EXISTS sales (                             │ │
  │  │    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),       │ │
  │  │    tenant_id   UUID NOT NULL,                                    │ │
  │  │    customer_id UUID NOT NULL,                                    │ │
  │  │    title       TEXT,                                             │ │
  │  │    amount      DECIMAL,                                          │ │
  │  │    status      TEXT DEFAULT 'new',                               │ │
  │  │    created_at  TIMESTAMPTZ DEFAULT now()                        │ │
  │  │  );                                                              │ │
  │  │                                                                  │ │
  │  │  -- IF NOT EXISTS = idempotent                                  │ │
  │  │  -- Chạy lần 2 → no-op. Không lỗi.                             │ │
  │  └──────────────────────────────────────────────────────────────────┘ │
  │                                                                        │
  │  Khi nào chạy:                                                       │
  │  ├── Plugin DEPLOY lần đầu tiên vào hệ thống                         │
  │  ├── Plugin UPGRADE version (new migration files)                     │
  │  └── KHÔNG chạy khi tenant enable/disable                             │
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘


  ┌─ TENANT MIGRATIONS (DML) ──────────────────────────────────────────────┐
  │                                                                        │
  │  Purpose:  Setup per-tenant: RLS, seed data, default configs           │
  │  Scope:    Per tenant — mỗi tenant chạy riêng                          │
  │  Runs:     Mỗi lần tenant enable plugin                               │
  │  Track:    table tenant_plugin_migrations trong Metadata DB             │
  │                                                                        │
  │  ┌──────────────────────────────────────────────────────────────────┐ │
  │  │  tenant_plugin_migrations                                        │ │
  │  │                                                                  │ │
  │  │  tenant_id  │ plugin    │ version │ executed_at                  │ │
  │  │  ───────────┼───────────┼─────────┼──────────────                │ │
  │  │  acme-123   │ cust-care │ 001     │ 2025-07-01                  │ │
  │  │  acme-123   │ cust-care │ 002     │ 2025-07-01                  │ │
  │  │  acme-123   │ cust-care │ 003     │ 2025-07-01                  │ │
  │  │  globex-456 │ cust-care │ 001     │ 2025-08-10                  │ │
  │  │  globex-456 │ cust-care │ 002     │ 2025-08-10                  │ │
  │  │  globex-456 │ cust-care │ 003     │ 2025-08-10                  │ │
  │  │                                                                  │ │
  │  │  CÓ cột tenant_id. Per-tenant per-plugin per-version.           │ │
  │  └──────────────────────────────────────────────────────────────────┘ │
  │                                                                        │
  │  SQL example:                                                        │
  │  ┌──────────────────────────────────────────────────────────────────┐ │
  │  │  -- tenant/002_seed_default_config.sql                           │ │
  │  │  -- Biến {{tenant_id}} được thay thế bởi Migration Engine       │ │
  │  │                                                                  │ │
  │  │  INSERT INTO sales (tenant_id, title, status)                    │ │
  │  │  VALUES ('{{tenant_id}}', 'Sample opportunity', 'new')         │ │
  │  │  ON CONFLICT DO NOTHING;                                        │ │
  │  │                                                                  │ │
  │  │  -- Seed default sales pipeline stages cho tenant này           │ │
  │  │  -- Chỉ ảnh hưởng DATA (rows), không ảnh hưởng STRUCTURE       │ │
  │  └──────────────────────────────────────────────────────────────────┘ │
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘

Enable Plugin — 3 kịch bản cụ thểEnable Plugin — 3 Concrete Scenarios

scenario: first enable
SCENARIO A: ACME enable customer-care
(Tenant ĐẦU TIÊN enable plugin này trên toàn hệ thống)

  Trạng thái trước:
  ├── Plugin customer-care v1.5.0 đã DEPLOY vào hệ thống
  ├── schema_migrations: trống cho customer-care
  ├── Chưa tenant nào enable
  └── Bảng sales, call_logs, tickets CHƯA TỒN TẠI


  Migration Engine xử lý:

  ┌────────────────────────────────────────────────────────────────────────┐
  │                                                                        │
  │  Step 1: Check Schema Migrations                                     │
  │                                                                        │
  │    SELECT MAX(version) FROM schema_migrations                          │
  │    WHERE plugin_name = 'customer-care'                                 │
  │    → NULL (chưa chạy bao giờ)                                         │
  │                                                                        │
  │    Pending: 001, 002, 003, 004                                         │
  │                                                                        │
  │  Step 2: Run Schema Migrations (DDL — toàn DB)                          │
  │                                                                        │
  │    BEGIN;                                                               │
  │      pg_advisory_lock(hash('schema:customer-care'))                    │
  │                                                                        │
  │      CREATE TABLE IF NOT EXISTS sales (...)     ← table MỚI       │
  │      CREATE TABLE IF NOT EXISTS call_logs (...) ← table MỚI       │
  │      CREATE TABLE IF NOT EXISTS tickets (...)   ← table MỚI       │
  │      CREATE INDEX ...                                               │
  │                                                                        │
  │      INSERT INTO schema_migrations VALUES                              │
  │        ('customer-care', '001'), ('customer-care', '002'),             │
  │        ('customer-care', '003'), ('customer-care', '004');             │
  │    COMMIT;                                                              │
  │                                                                        │
  │    Tables tạo xong. Mọi tenant tương lai sẽ dùng chung.             │
  │                                                                        │
  │  Step 3: Run Tenant Migrations (DML — cho ACME)                       │
  │                                                                        │
  │    BEGIN;                                                               │
  │      pg_advisory_lock(hash('tenant:acme-123:customer-care'))           │
  │                                                                        │
  │      -- Enable RLS cho ACME trên tables mới                         │
  │      ALTER TABLE sales ENABLE ROW LEVEL SECURITY;                   │
  │      CREATE POLICY IF NOT EXISTS tenant_iso ON sales                │
  │        USING (tenant_id = current_setting('app.tenant_id')::UUID);  │
  │                                                                        │
  │      ↑ RLS policies cũng idempotent: IF NOT EXISTS.                │
  │        Chạy 1 lần, apply cho MỌI tenant dùng chung table.          │
  │                                                                        │
  │      -- Seed default data cho ACME                                     │
  │      INSERT INTO sales (tenant_id, title, status)                      │
  │      VALUES ('acme-123', 'Sample opportunity', 'new');                 │
  │                                                                        │
  │      INSERT INTO tenant_plugin_migrations VALUES                       │
  │        ('acme-123', 'customer-care', '001'),                           │
  │        ('acme-123', 'customer-care', '002'),                           │
  │        ('acme-123', 'customer-care', '003');                           │
  │    COMMIT;                                                              │
  │                                                                        │
  │  Step 4: Update tenant_plugins                                       │
  │    UPDATE tenant_plugins SET enabled = true                            │
  │    WHERE tenant_id = 'acme-123' AND plugin = 'customer-care';         │
  │                                                                        │
  │  Step 5: Invalidate ACME tenant context cache                        │
  │    → Next request rebuilds context with customer-care enabled.         │
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘

Disable Plugin — Bảng chung không bị ảnh hưởngDisable Plugin — Shared Tables Are Unaffected

disable: soft vs hard
ACME disable customer-care
(GLOBEX và WAYNE vẫn dùng)


OPTION A: SOFT DISABLE (mặc định — khuyến nghị)

  ┌────────────────────────────────────────────────────────────────────────┐
  │                                                                        │
  │  Step 1: Update tenant_plugins                                       │
  │    UPDATE tenant_plugins SET enabled = false                          │
  │    WHERE tenant_id = 'acme-123' AND plugin = 'customer-care';         │
  │                                                                        │
  │  Step 2: Invalidate ACME tenant context cache                        │
  │    → enabledPlugins không còn customer-care                            │
  │    → Routes /sales, /calls, /tickets → 404 cho ACME                   │
  │    → Event handlers customer-care: SKIP cho ACME                      │
  │    → Hook consumers customer-care: SKIP cho ACME                      │
  │                                                                        │
  │  Step 3: KHÔNG có gì xảy ra ở database level                        │
  │                                                                        │
  │    Tables:     GIỮA NGUYÊN (sales, call_logs, tickets vẫn ở đó)     │
  │    ACME data:  GIỮA NGUYÊN (rows với tenant_id=acme-123 vẫn ở đó)  │
  │    RLS:        GIỮA NGUYÊN (policies vẫn active)                    │
  │    Indexes:    GIỮA NGUYÊN                                          │
  │                                                                        │
  │  Hiệu ứng:                                                          │
  │  ├── ACME không thể truy cập sales API nữa (routes disabled)         │
  │  ├── ACME data vẫn nằm trong table (không xóa)                        │
  │  ├── Nếu ACME re-enable → data cũ vẫn còn → hoạt động ngay          │
  │  ├── GLOBEX, WAYNE: KHÔNG BỊ ẢNH HƯỞNG GÌ                          │
  │  └── Schema migrations: KHÔNG THAY ĐỔI (tables vẫn tồn tại)        │
  │                                                                        │
  │  Soft disable = vô hình hóa plugin cho tenant đó.                   │
  │  Như ẩn app trên điện thoại — app vẫn còn, data vẫn còn.            │
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘


OPTION B: HARD DISABLE + DATA PURGE (yêu cầu tường minh — hiếm)

  ┌────────────────────────────────────────────────────────────────────────┐
  │                                                                        │
  │  Chỉ chạy khi tenant YÊU CẦU xóa data (GDPR, account close...)     │
  │                                                                        │
  │  Step 1: Soft disable (như trên)                                     │
  │                                                                        │
  │  Step 2: Purge ACME data FROM shared tables                           │
  │                                                                        │
  │    BEGIN;                                                               │
  │      DELETE FROM tickets   WHERE tenant_id = 'acme-123';            │
  │      DELETE FROM call_logs WHERE tenant_id = 'acme-123';            │
  │      DELETE FROM sales     WHERE tenant_id = 'acme-123';            │
  │    COMMIT;                                                              │
  │                                                                        │
  │    Xóa ROWS, không xóa TABLE.                                      │
  │    Table vẫn tồn tại cho GLOBEX, WAYNE.                             │
  │                                                                        │
  │  Step 3: Remove tenant migration tracking                            │
  │    DELETE FROM tenant_plugin_migrations                                │
  │    WHERE tenant_id = 'acme-123' AND plugin = 'customer-care';         │
  │                                                                        │
  │  Step 4: Audit log ghi: "ACME data purged for customer-care"         │
  │                                                                        │
  │  Hiệu ứng:                                                          │
  │  ├── ACME data: XÓA VĨNH VIỄN                                       │
  │  ├── Tables: VẪN TỒN TẠI (shared, dùng bởi tenants khác)            │
  │  ├── Nếu ACME re-enable → bắt đầu từ 0 (data đã mất)               │
  │  ├── GLOBEX, WAYNE: KHÔNG BỊ ẢNH HƯỞNG GÌ                          │
  │  └── Schema: KHÔNG THAY ĐỔI                                          │
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘


TRƯỜNG HỢP ĐẶC BIỆT: Tenant CUỐI CÙNG disable

  ┌────────────────────────────────────────────────────────────────────────┐
  │                                                                        │
  │  ACME, GLOBEX, WAYNE đều disable customer-care.                       │
  │  Không còn tenant nào dùng.                                            │
  │                                                                        │
  │  Tables sales, call_logs, tickets VẪN TỒN TẠI.                      │
  │                                                                        │
  │  Tại sao không DROP?                                                   │
  │  ├── 1. Tenant mới có thể enable bất cứ lúc nào                     │
  │  ├── 2. Soft-disabled tenants vẫn có data trong tables               │
  │  ├── 3. DROP TABLE = DDL lock → ảnh hưởng toàn DB                   │
  │  ├── 4. Rỗng tables chiếm ~0 disk (chỉ metadata)                   │
  │  └── 5. Audit/compliance có thể cần data lịch sử                   │
  │                                                                        │
  │  Tables rỗng = harmless. Để yên.                                    │
  │  System Admin CÓ THỂ manual cleanup nếu plugin bị deprecated,        │
  │  nhưng đây là admin operation, không phải auto behavior.              │
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘

QUY TẮC MIGRATION KHI UPGRADE:

  ┌────────────────────────────────────────────────────────────────────────┐
  │                                                                        │
  │  1. Schema migrations PHẢI backward compatible:                      │
  │     ├── ADD COLUMN (OK — old code ignores new column)                 │
  │     ├── ADD INDEX  (OK — old code không bị ảnh hưởng)                │
  │     ├── DROP COLUMN (CẤM — old code sẽ break)                        │
  │     ├── RENAME COLUMN (CẤM — old code sẽ break)                     │
  │     └── ALTER TYPE (THẬN TRỌNG — cần compatible cast)                │
  │                                                                        │
  │  2. Nếu cần breaking schema change (DROP/RENAME column):              │
  │     → 2-phase migration:                                               │
  │     Phase 1 (v2.0): ADD new column, copy data, code uses both.       │
  │     Phase 2 (v3.0): Sau khi MỌI tenant upgrade → DROP old column.   │
  │                                                                        │
  │  3. Schema DDL = chạy 1 lần, ảnh hưởng mọi tenant NGAY.            │
  │     → PHẢI test kỹ. PHẢI backward compatible.                        │
  │     → Deploy OFF-PEAK hours.                                           │
  │                                                                        │
  │  4. Tenant DML = chạy per-tenant, tenant upgrade khi nào thì chạy. │
  │     → Safe. Isolated. Rollback dễ.                                    │
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘

Edge Cases & Risks

edge cases
EDGE CASE 1: Schema migration FAIL giữa chừng

  ┌────────────────────────────────────────────────────────────────────────┐
  │                                                                        │
  │  Deploy customer-care v2.0:                                            │
  │  schema/005: ALTER TABLE sales ADD COLUMN → OK                      │
  │  schema/006: CREATE INDEX CONCURRENTLY  → FAIL (disk full)           │
  │                                                                        │
  │  Vấn đề: 005 đã commit, 006 fail.                                   │
  │  Cột mới đã thêm nhưng index chưa tạo.                                │
  │                                                                        │
  │  Giải pháp:                                                          │
  │  ├── Mỗi schema migration file = 1 transaction riêng                  │
  │  ├── Tracking ghi: 005 done, 006 FAILED                               │
  │  ├── Next attempt: retry từ 006 (005 đã recorded, skip)              │
  │  ├── Alerting: "Schema migration partial failure" → ops investigate   │
  │  └── Rollback 005 nếu cần: schema cũng có DOWN migration             │
  │        005_down: ALTER TABLE sales DROP COLUMN pipeline_stage;        │
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘


EDGE CASE 2: Tenant enable VÀ disable cùng lúc (concurrent)

  ┌────────────────────────────────────────────────────────────────────────┐
  │                                                                        │
  │  ACME admin click "enable" → request 1                                │
  │  ACME admin click "disable" 1 giây sau → request 2                    │
  │                                                                        │
  │  Giải pháp: Advisory Lock                                          │
  │  pg_advisory_lock(hash('tenant:acme-123:customer-care'))              │
  │                                                                        │
  │  Request 1 acquire lock → chạy enable migration                        │
  │  Request 2 đợi lock → lock released → chạy disable                    │
  │  → Kết quả: enable xong, rồi disable → cuối cùng disabled.           │
  │  → Consistent. Không race condition.                                   │
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘


EDGE CASE 3: Orphaned data sau khi plugin UNDEPLOY

  ┌────────────────────────────────────────────────────────────────────────┐
  │                                                                        │
  │  System Admin undeploy customer-care khỏi toàn hệ thống.             │
  │  Plugin code bị remove. Không ai có thể enable nữa.                  │
  │                                                                        │
  │  Nhưng tables (sales, call_logs, tickets) VẪN TỒN TẠI.             │
  │  Data VẪN NẰM TRONG tables.                                          │
  │                                                                        │
  │  Giải pháp:                                                          │
  │  ├── Plugin status = "deprecated" (giai đoạn chuyển tiếp)            │
  │  │   → Tenants still read-only access for 90 days                     │
  │  │   → Export data window                                              │
  │  ├── After 90 days: status = "removed"                                │
  │  │   → Admin CAN run manual cleanup:                                  │
  │  │     DROP TABLE IF EXISTS sales;                                    │
  │  │     DROP TABLE IF EXISTS call_logs;                                │
  │  │     DROP TABLE IF EXISTS tickets;                                  │
  │  │   → Chỉ khi KHÔNG CÒN tenant nào enabled                         │
  │  └── Audit trail giữ lại record vĩnh viễn                            │
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘


EDGE CASE 4: ALTER TABLE trên bảng lớn (100M+ rows)

  ┌────────────────────────────────────────────────────────────────────────┐
  │                                                                        │
  │  Schema migration: ALTER TABLE sales ADD COLUMN pipeline_stage TEXT    │
  │  Bảng sales có 100 triệu rows (tất cả tenants cộng lại).            │
  │                                                                        │
  │  PostgreSQL 11+:                                                    │
  │  ADD COLUMN with DEFAULT = metadata-only operation.                   │
  │  Near-instant. Không rewrite table. Không lock.                     │
  │                                                                        │
  │  Nhưng: ADD COLUMN NOT NULL without DEFAULT = full table rewrite.   │
  │  → CẤM trong schema migrations.                                     │
  │  → Phải dùng: ADD COLUMN + DEFAULT + backfill async + ADD NOT NULL.  │
  │                                                                        │
  │  Safe migration pattern:                                                │
  │  v2.0: ADD COLUMN pipeline_stage TEXT DEFAULT 'new';   (instant)      │
  │  v2.1: Background job: backfill NULL rows;             (async)        │
  │  v2.2: ALTER COLUMN SET NOT NULL;                      (fast, no NULLs)│
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘


EDGE CASE 5: FK constraint cross-plugin khi disable

  ┌────────────────────────────────────────────────────────────────────────┐
  │                                                                        │
  │  sales.customer_id FK → customers.id                                  │
  │  (care plugin)           (data plugin)                                 │
  │                                                                        │
  │  ACME disable customer-data nhưng VẪN enable customer-care?           │
  │                                                                        │
  │  Giải pháp: Dependency enforcement                                  │
  │  customer-care depends on customer-data (manifest).                    │
  │  → System REFUSES disable data nếu care vẫn enabled.                  │
  │  → Error: "Cannot disable customer-data: customer-care depends on it" │
  │  → Must disable care FIRST, then data.                                 │
  │                                                                        │
  │  → FK constraints luôn valid vì dependency chain enforced.            │
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘

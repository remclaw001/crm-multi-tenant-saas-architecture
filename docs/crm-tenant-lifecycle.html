Tenant Management

Vòng đời Tenant

Provisioning → Active → Tier Change → Offboard. Bao gồm VIP migration và Billing & Quota enforcement.

00

Bản đồ vòng đời Tenant

Tenant lifecycle: Một tenant đi qua nhiều trạng thái trong suốt vòng đời. Mỗi chuyển trạng thái đều có flow cụ thể ảnh hưởng đến Metadata DB, connection pool, plugin registry, Redis cache, và billing.
lifecycle overview
  TRẠNG THÁI TENANT:

  ┌───────────────────────────────────────────────────────────────────────────┐
  │                                                                           │
  │   Admin tạo              Thanh toán             Hết hạn grace            │
  │       │                  thành công                  │                   │
  │       ▼                      ▲                        ▼                   │
  │  PROVISIONING  ──────►  ACTIVE  ──────►  GRACE PERIOD  ──────►  SUSPENDED  │
  │                              │                                    │       │
  │                              │ tier change                        │ 30d   │
  │                              ▼                                    ▼       │
  │                         TIER CHANGED                     OFFBOARDED    │
  │                              │                                           │
  │                              ▼                                           │
  │                     VIP: MIGRATING  ──────►  ACTIVE                      │
  │                     (nếu nâng lên VIP tier — cần Dedicated DB mới)     │
  │                                                                           │
  └───────────────────────────────────────────────────────────────────────────┘


  TRẠNG THÁI → Ý NGHĨA:

  ┌──────────────────────┬──────────────────────────────────────────────────────┐
  │ PROVISIONING         │ Đang tạo records, enable plugins, chạy migration.    │
  │                      │ ~30s. Gateway từ chối mọi request.                   │
  ├──────────────────────┼──────────────────────────────────────────────────────┤
  │ ACTIVE               │ Hoạt động bình thường. Full access theo tier.         │
  ├──────────────────────┼──────────────────────────────────────────────────────┤
  │ MIGRATING            │ Đang di chuyển sang VIP Dedicated DB (~5–30 phút).   │
  │                      │ Read-only. Write ops bị từ chối.                     │
  ├──────────────────────┼──────────────────────────────────────────────────────┤
  │ GRACE PERIOD         │ Hóa đơn quá hạn. Còn dùng đầy đủ. Cảnh báo 7 ngày.  │
  ├──────────────────────┼──────────────────────────────────────────────────────┤
  │ SUSPENDED            │ Read-only. Không tạo/sửa/xóa data. 30 ngày.          │
  ├──────────────────────┼──────────────────────────────────────────────────────┤
  │ OFFBOARDED           │ Đã export, đang cleanup. Không truy cập được.         │
  └──────────────────────┴──────────────────────────────────────────────────────┘


  CÁC THAO TÁC CHÍNH:

  ┌──────────────────────┬──────────────────┬──────────────────────────────────┐
  │ Thao tác             │ Ai trigger       │ Tác động hệ thống                │
  ├──────────────────────┼──────────────────┼──────────────────────────────────┤
  │ Tạo tenant           │ System Admin     │ DB records, pool slot, plugins   │
  │ Nâng tier            │ Tenant Admin     │ Pool cap, plugins, quotas        │
  │ Nâng lên VIP         │ Tenant Admin     │ Migrate sang Dedicated DB        │
  │ Hạ tier              │ Tenant Admin     │ Giảm pool cap, disable plugins   │
  │ Hạ từ VIP            │ Tenant Admin     │ Migrate về Shared DB             │
  │ Offboard tự nguyện   │ Tenant Admin     │ Export, cleanup, release         │
  │ Offboard tự động     │ Billing system   │ Sau grace + suspend period       │
  └──────────────────────┴──────────────────┴──────────────────────────────────┘
01

Tạo Tenant — Provisioning Flow

Mục tiêu: Trong ~30ms sync, một tenant mới phải có record trong Metadata DB, domain routing trong Redis, connection pool slot, và tất cả default plugins sẵn sàng với schema + seed data.
provisioning flow
POST /system/tenants  (System Admin)
Body: { name, domain, tier, adminEmail }


BƯỚC 1: Validate
│
│  SELECT FROM tenants WHERE domain = ?  → phải trả về 0 rows
│  Kiểm tra tier hợp lệ: ['basic','premium','enterprise','vip']
│  → PASS


BƯỚC 2: Tạo Tenant Record trong Metadata DB  (~1ms)
│
│  INSERT INTO tenants:
│  { tenantId:  "acme-123",          ← UUID generated
│    name:      "ACME Corp",
│    domain:    "acme.crm.com",
│    tier:      "premium",
│    status:    "provisioning",
│    createdAt: now(),
│    settings:  { ...tier_defaults } }
│
│  INSERT INTO tenant_admins:
│  { tenantId: "acme-123", userId: adminUserId, role: "admin" }


BƯỚC 3: Ghi Redis domain-map  (~0.5ms)
│
│  SET domain-map:acme.crm.com → "acme-123"
│  TenantResolver (L2) dùng key này để resolve tenantId từ Host header


BƯỚC 4: Cấp Connection Pool Slot  (~1ms)
│
│  Nếu tier ≠ vip:
│    TenantQuotaEnforcer.register("acme-123", cap=20)  ← premium
│  Nếu tier = vip:
│    Bỏ qua bước này — VIP Dedicated DB sẽ được provisioned riêng


BƯỚC 5: Enable Default Plugins theo Tier

  ┌────────────────┬──────────────────────────────────────────────────────┐
  │ Tier           │ Default plugins enabled khi tạo mới                  │
  ├────────────────┼──────────────────────────────────────────────────────┤
  │ Basic          │ customer-data                                        │
  │ Premium        │ customer-data, customer-care, analytics              │
  │ Enterprise     │ customer-data, customer-care, analytics, marketing   │
  │ VIP            │ customer-data, customer-care, analytics, marketing,  │
  │                │ automation  (+ plugins custom theo contract)          │
  └────────────────┴──────────────────────────────────────────────────────┘

  Với mỗi plugin cần enable:
  │
  ├── SchemaMigrationRunner.ensureSchema(pluginId, db)
  │       Chạy DDL nếu bảng chưa tồn tại (idempotent — safe to re-run)
  │       CREATE TABLE IF NOT EXISTS customers (...)
  │       CREATE INDEX IF NOT EXISTS idx_customers_tenant_id ...
  │       CREATE POLICY IF NOT EXISTS tenant_isolation ON customers ...
  │
  └── TenantMigrationRunner.seedTenant(pluginId, tenantId, tier, db)
          INSERT plugin_installations: { tenantId, pluginId, status:'active' }
          INSERT plugin_configs:       { tenantId, pluginId, config:{...defaults} }


BƯỚC 6: Kích hoạt  (~0.5ms)
│
│  UPDATE tenants SET status = 'active' WHERE tenantId = 'acme-123'
│  TenantContext LRU: warm cache (optional, lazy load cũng được)


BƯỚC 7: Gửi email chào mừng  (async, không block response)
│
│  PUBLISH crm.system exchange
│  routing_key: tenant.provisioned
│  body: { tenantId, adminEmail, tier, loginUrl }


TỔNG KẾT:

  ┌────────────────────────────────┬────────┬───────────────────────────────┐
  │ Bước                           │ Time   │ Target                        │
  ├────────────────────────────────┼────────┼───────────────────────────────┤
  │ Validate + Tenant record       │ ~2ms   │ Metadata DB                   │
  │ Redis domain-map               │ ~0.5ms │ Redis                         │
  │ Pool slot registration         │ ~1ms   │ In-memory + Metadata DB       │
  │ Plugin schema (DDL, idempotent)│ ~5ms   │ Shared Data DB                │
  │ Plugin seed (DML per plugin)   │ ~3ms   │ Shared Data DB                │
  │ Activate                       │ ~0.5ms │ Metadata DB                   │
  │ Welcome email (async)          │ ~2ms   │ RabbitMQ → Worker             │
  ├────────────────────────────────┼────────┼───────────────────────────────┤
  │ TOTAL sync                     │ ~14ms  │ Tenant sẵn sàng nhận request  │
  └────────────────────────────────┴────────┴───────────────────────────────┘
02

Nâng Tier — Upgrade Flow

Hai trường hợp: Nâng Basic/Premium/Enterprise lên nhau là thao tác đơn giản, không downtime. Nâng lên VIP phức tạp hơn nhiều vì phải provision Dedicated DB mới và migrate toàn bộ dữ liệu.
upgrade — shared tiers
TRƯỜNG HỢP 1: Basic / Premium / Enterprise → cao hơn (không phải VIP)
Ví dụ: acme-123 từ Premium → Enterprise. Không downtime.


BƯỚC 1: Update Metadata DB
│  UPDATE tenants SET tier='enterprise', updatedAt=now()
│  WHERE tenantId='acme-123'


BƯỚC 2: Tăng Connection Pool Cap
│  TenantQuotaEnforcer.updateCap("acme-123", newCap=30)
│  premium (20) → enterprise (30). Áp dụng ngay cho connections mới.


BƯỚC 3: Config Hot-reload
│  ConfigResolver.invalidate("acme-123")
│  Redis PUBLISH config:reload { tenantId: "acme-123" }
│  → Tất cả app instances nhận config mới:
│    Rate limit:      500 → 2000 rps
│    Storage quota:   50GB → 200GB
│    Sandbox timeout: 5s → 5s  (không thay đổi giữa các shared tiers)


BƯỚC 4: Enable Plugins mới (nếu có)
│  delta = enterprise_plugins - premium_plugins = { marketing }
│  Với mỗi plugin trong delta:
│  ├── SchemaMigrationRunner.ensureSchema('marketing', db)
│  └── TenantMigrationRunner.seedTenant('marketing', 'acme-123', 'enterprise', db)


BƯỚC 5: Invalidate TenantContext Cache
│  Redis PUBLISH cache:invalidate { tenantId: "acme-123", scope: "tenant-context" }
│  → Tất cả instances xóa LRU entry cho acme-123
│  → Request tiếp theo load lại TenantContext với tier mới


  Kết quả: Tenant nhận features mới ngay lập tức. Không có downtime.
upgrade — to VIP
TRƯỜNG HỢP 2: Bất kỳ tier → VIP
Cần Dedicated DB mới. Tenant ở read-only mode trong quá trình migration.


BƯỚC 1: Bắt đầu Migration
│  UPDATE tenants SET status='migrating', tier='vip'
│  Gateway kiểm tra status 'migrating':
│    GET requests  → PASS
│    Write requests → 503 "Đang nâng cấp, vui lòng thử lại sau ít phút"


BƯỚC 2: Provision Dedicated DB Instance
│  Tạo PostgreSQL instance mới (8–32 vCPU, 32–128GB RAM theo contract VIP)
│  Chạy full schema migration (tất cả plugins đã enable):
│    SchemaMigrationRunner với PostgresMigrationDialect
│    → Tạo tables, indices, RLS policies trên DB mới


BƯỚC 3: Migrate Data
│
│  Copy toàn bộ data của acme-123 từ Shared DB → Dedicated DB:
│
│  Shared DB (có tenant_id column):           Dedicated DB (không có tenant_id):
│  ┌──────────────────────────────────┐       ┌────────────────────────────────┐
│  │ customers WHERE tenant_id=acme  │  ──►  │ customers (all rows = acme)   │
│  │ sales WHERE tenant_id=acme      │       │ sales                          │
│  │ ...                              │       │ ...                            │
│  └──────────────────────────────────┘       └────────────────────────────────┘
│
│  Thời gian migration: ~5–30 phút tùy volume dữ liệu.
│  Dùng streaming COPY hoặc pg_dump per-tenant + pg_restore.


BƯỚC 4: Verify + Switch
│  Row count check: Shared DB == Dedicated DB cho mỗi table
│  Checksum verify trên primary keys (sample hoặc full)
│  PoolRouter.registerVipPool("acme-123", dedicatedConnString)
│  UPDATE tenants SET status='active'


BƯỚC 5: Cleanup Shared DB  (async, 24h sau verify OK)
│  DELETE FROM customers WHERE tenant_id='acme-123'
│  DELETE FROM sales WHERE tenant_id='acme-123'
│  ... (tất cả plugin tables)
│  Giữ 24h để có rollback window nếu cần.
03

Hạ Tier — Downgrade Flow

Cần cảnh báo trước: Hạ tier có thể disable một số plugins và tenant sẽ không truy cập được data của plugins đó (data vẫn còn trong DB, không bị xóa). Hệ thống phải hiện impact rõ ràng và yêu cầu confirm trước khi thực hiện.
downgrade — shared tiers
TRƯỜNG HỢP 1: Enterprise / Premium → thấp hơn (không phải VIP)
Ví dụ: acme-123 từ Enterprise → Premium.


BƯỚC 1: Tính Impact và Yêu cầu Confirm
│  delta_disable = enterprise_plugins - premium_plugins = { marketing }
│  Trả về cho Tenant Admin:
│  {
│    pluginsToDisable: ["marketing"],
│    dataNote: "Dữ liệu marketing vẫn còn trong DB nhưng không truy cập được.",
│    connectionCapChange: "30 → 20 connections",
│    rateLimit: "2000 → 500 rpm",
│    storageQuota: "200GB → 50GB (bạn đang dùng 80GB — cần cleanup trong 30 ngày)"
│  }
│  Tenant phải gọi: POST /tenant/downgrade/confirm  để xác nhận


BƯỚC 2: Disable Plugins dư thừa
│  Với mỗi plugin trong delta_disable (marketing):
│  │
│  ├── PluginRegistry.removeRoutes('marketing', 'acme-123')
│  ├── EventBus.removeHandlers('marketing', 'acme-123')
│  ├── HookRegistry.removeHooks('marketing', 'acme-123')
│  │
│  └── TenantMigrationRunner.disableTenant('marketing', 'acme-123', db)
│          UPDATE plugin_installations SET status='disabled'
│          Schema (DDL) KHÔNG xóa. Data vẫn còn. Chỉ đổi status.


BƯỚC 3: Update Tier + Giảm Pool Cap
│  UPDATE tenants SET tier='premium'
│  TenantQuotaEnforcer.updateCap("acme-123", newCap=20)
│  Connections đang active (tối đa 30) vẫn chạy đến khi release.
│  Sau khi release, cap mới (20) sẽ được áp dụng.


BƯỚC 4: Enforce Quota mới
│  Nếu đang dùng 80GB > 50GB (premium quota):
│  → Gửi notification: "Vui lòng xóa bớt data trong 30 ngày. Sau đó uploads mới sẽ bị block."
│  → Chưa hard-block ngay — grace period 30 ngày


BƯỚC 5: Invalidate Cache
│  Redis PUBLISH cache:invalidate { tenantId: "acme-123", scope: "tenant-context" }
downgrade — from VIP
TRƯỜNG HỢP 2: VIP → Enterprise
Cần migrate ngược: Dedicated DB → Shared DB.


BƯỚC 1–2:  Như trên (impact check + disable delta plugins nếu có)

BƯỚC 3: Bắt đầu Migration
│  UPDATE tenants SET status='migrating'
│  Tenant ở read-only mode trong quá trình migration


BƯỚC 4: Migrate Data về Shared DB
│
│  Dedicated DB (không có tenant_id):         Shared DB (thêm tenant_id):
│  ┌──────────────────────────────────┐       ┌────────────────────────────────┐
│  │ customers (all rows = acme)      │  ──►  │ customers WHERE tenant_id=acme │
│  │ sales                            │       │ sales WHERE tenant_id=acme     │
│  └──────────────────────────────────┘       └────────────────────────────────┘
│
│  Verify count + checksum sau khi copy xong


BƯỚC 5: Switch Pool Routing
│  PoolRouter.deregisterVipPool("acme-123")
│  TenantQuotaEnforcer.register("acme-123", cap=30)  ← enterprise cap
│  UPDATE tenants SET tier='enterprise', status='active'


BƯỚC 6: Decommission Dedicated DB  (7 ngày sau verify OK)
│  Backup cuối → s3://crm-backups/acme-123/vip-final-{timestamp}/
│  Terminate DB instance
│  7 ngày buffer để rollback nếu phát hiện vấn đề.
04

Offboard — Xóa Tenant

Không thể hoàn tác sau retention period: Offboard có 2 nguồn trigger — Tenant Admin tự nguyện, hoặc Billing system tự động sau khi hết suspension period. Cả hai đi qua cùng flow nhưng với origin khác nhau trong audit log.
offboard flow
TRIGGER:
  A) Tự nguyện:  POST /tenant/offboard  (Tenant Admin)
  B) Tự động:    Billing system trigger sau suspension_period = 30 ngày


BƯỚC 1: Khóa Tenant  (immediate)
│  UPDATE tenants SET status='offboarding'
│  Redis SET tenant-status:acme-123 → "offboarding"
│  Gateway: mọi request → 401 "Tài khoản đang được đóng"


BƯỚC 2: Export Data → S3  (async worker, ~5–60 phút)
│
│  Worker: data-export
│  ├── Dump toàn bộ tables của acme-123:
│  │     SELECT * FROM customers WHERE tenant_id='acme-123' → JSON
│  │     SELECT * FROM sales WHERE tenant_id='acme-123'     → JSON
│  │     ... (tất cả plugin tables)
│  ├── PUT s3://crm-exports/acme-123/{timestamp}/customers.json
│  ├── PUT s3://crm-exports/acme-123/{timestamp}/sales.json
│  ├── PUT s3://crm-exports/acme-123/{timestamp}/export_manifest.json
│  │     { tables, rowCounts, exportedAt, tenantId, tier }
│  └── PUT s3://crm-exports/acme-123/{timestamp}/attachments.zip
│
│  Gửi email cho Tenant Admin: pre-signed download URL (expire 90 ngày)


BƯỚC 3: Disable tất cả Plugins
│  Với mỗi enabled plugin của acme-123:
│  ├── Unregister routes, events, hooks (in-memory, immediate)
│  └── UPDATE plugin_installations SET status='disabled'


BƯỚC 4: Xóa Redis Entries
│  DEL  domain-map:acme.crm.com
│  DEL  tenant-status:acme-123
│  DEL  domain-map:custom.acme.com        ← nếu có custom domain
│  DEL  rl:acme-123                        ← rate limit bucket
│  SCAN + DEL t:acme-123:*                ← toàn bộ cache entries
│  Auth blacklist entries tự expire theo JWT TTL (max 7 ngày)


BƯỚC 5: Release Connection Pool
│  TenantQuotaEnforcer.deregister("acme-123")
│  Chờ active connections drain (timeout 30s)
│  Force close nếu còn sót


BƯỚC 6: Soft Delete
│  UPDATE tenants SET
│    status      = 'offboarded',
│    offboardedAt = now(),
│    domain      = null    ← giải phóng domain để có thể tái sử dụng
│  WHERE tenantId='acme-123'
│  KHÔNG xóa record khỏi DB. Giữ để audit + compliance.


BƯỚC 7: Ghi Audit Trail
│  INSERT INTO audit_log:
│  { action: 'tenant.offboarded', tenantId: 'acme-123',
│    triggeredBy, reason, exportPath, timestamp }
│  Audit log là immutable — không bị xóa theo bất kỳ retention policy nào.


BƯỚC 8: Hard Delete  (scheduled job, 90 ngày sau offboard)
│
│  Scheduler (cron) chạy:
│  ├── Shared Data DB:  DELETE FROM * WHERE tenant_id='acme-123'
│  ├── Metadata DB:     DELETE FROM tenants WHERE tenantId='acme-123'
│  │                    DELETE FROM tenant_admins WHERE tenantId='acme-123'
│  │                    GIỮ audit_log entries — không xóa bao giờ
│  ├── S3:              DELETE s3://tenants/acme-123/*  (files + attachments)
│  │                    Export files ở s3://crm-exports/ giữ theo policy riêng
│  └── VIP:             Decommission Dedicated DB nếu chưa làm trước đó


TIMELINE:

  Day 0   : Offboard triggered → tenant locked, export started
  Day 0   : Export hoàn tất → email với download link gửi cho admin
  Day 0–90: Tenant Admin có thể download export (pre-signed S3 URL)
  Day 90  : Hard delete tất cả data
  Forever : Audit log được giữ
05

Billing & Quota Enforcement

Nguyên tắc: Không hard-block ngay khi vượt giới hạn mà dùng grace period để tránh mất data không cần thiết. Chỉ suspend sau khi đã cảnh báo đủ số lần và hết thời gian grace.
subscription lifecycle
SUBSCRIPTION LIFECYCLE:

  ACTIVE
  │  Bình thường. Full access theo tier.
  │
  │  Ngày N: Hóa đơn đến hạn, thanh toán thất bại
  ▼
  GRACE PERIOD  (7 ngày)
  │  Vẫn dùng được đầy đủ.
  │  Email cảnh báo ngày 1, 3, 7.
  │  Banner cảnh báo trong Admin Console.
  │  Header X-Billing-Warning trong mọi API response.
  │
  │  Thanh toán thành công → quay lại ACTIVE ngay lập tức
  │  Ngày 7: Vẫn chưa thanh toán
  ▼
  SUSPENDED  (30 ngày)
  │  Read-only mode.
  │  GET requests: PASS
  │  POST / PUT / DELETE: 402 Payment Required
  │  EventBus filter out suspended tenants (không nhận events).
  │  Email nhắc nhở mỗi tuần.
  │
  │  Thanh toán thành công → quay lại ACTIVE ngay lập tức
  │  Ngày 30: Vẫn chưa thanh toán
  ▼
  OFFBOARDED  (automated offboard flow)


ENFORCEMENT TẠI GATEWAY (AuthGuard kiểm tra tenant.status):

  ┌─────────────────┬────────────────────────────────────────────────────────┐
  │ Status          │ Behavior                                               │
  ├─────────────────┼────────────────────────────────────────────────────────┤
  │ provisioning    │ 503 Service Unavailable "Đang khởi tạo"               │
  │ migrating       │ 503 "Đang nâng cấp hệ thống, vui lòng thử lại sau"   │
  │ grace_period    │ PASS + header X-Billing-Warning: "..."                 │
  │ suspended       │ GET → PASS;  POST/PUT/DELETE → 402 Payment Required   │
  │ offboarding     │ 401 "Tài khoản đang được đóng"                        │
  │ offboarded      │ 404 Tenant not found                                   │
  └─────────────────┴────────────────────────────────────────────────────────┘
quota limits
STORAGE QUOTA (per tenant):

  ┌──────────────┬───────────┬──────────────────────────────────────────────┐
  │ Tier         │ Quota     │ Enforcement                                  │
  ├──────────────┼───────────┼──────────────────────────────────────────────┤
  │ Basic        │ 5 GB      │ Block upload khi vượt quota ngay            │
  │ Premium      │ 50 GB     │ Warn ở 80%, block ở 100%                    │
  │ Enterprise   │ 200 GB    │ Warn ở 80%, block ở 100%                    │
  │ VIP          │ Unlimited │ Billing per GB — không block               │
  └──────────────┴───────────┴──────────────────────────────────────────────┘

  Kiểm tra tại StorageService.upload():
    current = SELECT SUM(size) FROM files WHERE tenant_id = ?
    if current + newFileSize > quota → 413 Payload Too Large


API RATE LIMIT (per minute, sliding window — Redis):

  ┌──────────────┬──────────┬──────────────────────────────────────────────┐
  │ Tier         │ Limit    │ Vượt giới hạn                                │
  ├──────────────┼──────────┼──────────────────────────────────────────────┤
  │ Basic        │ 100 rpm  │ 429 Too Many Requests                       │
  │ Premium      │ 500 rpm  │ 429 Too Many Requests                       │

Multi-Tenant SaaS — Khái niệm cốt lõiMulti-Tenant SaaS — Core Concepts

Định nghĩa: Multi-tenant = 1 application instance phục vụ nhiều tổ chức (tenants). Mỗi tenant thấy hệ thống như riêng của họ — dữ liệu cách ly, cấu hình riêng, billing riêng — nhưng chia sẻ cùng codebase và infrastructure.
Definition: Multi-tenant = 1 application instance serving multiple organizations (tenants). Each tenant sees the system as their own — isolated data, individual configuration, individual billing — but shares the same codebase and infrastructure.

  ├── Read replicas + query routing (heavy reads → replica)
  └── Background job queues per tenant (fair scheduling)

Shared Database — Deep Dive

shared db architecture
CẤU TRÚC:

  Mọi tenant dùng chung DB instance, chung tables.
  Mỗi table có cột tenant_id — discriminator column.

  ┌─ customers ──────────────────────────────────────────┐
  │ id       │ tenant_id  │ name        │ email            │
  ├──────────┼────────────┼─────────────┼──────────────────┤
  │ cust-01  │ acme-123   │ John Doe    │ john@acme.com    │
  │ cust-02  │ globex-456 │ Jane Smith  │ jane@globex.com  │
  │ cust-03  │ acme-123   │ Bob Lee     │ bob@acme.com     │
  │ cust-04  │ initech-789│ Alice Chen  │ alice@initech.com│
  └──────────┴────────────┴─────────────┴──────────────────┘

  Mọi query PHẢI có WHERE tenant_id = ? (hoặc tương đương).
  Thiếu tenant_id = data leak. Đây là lỗi nghiêm trọng nhất.


4 CƠ CHẾ BẢO VỆ (defense in depth):

  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                  │
  │  Tầng 1: Application-level injection                            │
  │  Middleware/interceptor tự động thêm tenant_id vào mọi query.   │
  │  Developer KHÔNG tự viết WHERE tenant_id — hệ thống inject.     │
  │                                                                  │
  │  Tầng 2: Query validation                                       │
  │  Trước khi execute: kiểm tra query CÓ tenant_id filter không.  │
  │  Không có → BLOCK query, log security alert.                    │
  │                                                                  │
  │  Tầng 3: Database-level enforcement (nếu DB hỗ trợ)             │
  │  PostgreSQL RLS: CREATE POLICY ... USING (tenant_id = ...)      │
  │  Oracle VPD: Virtual Private Database                           │
  │  DB engine tự filter rows — kể cả app có bug.                  │
  │  MySQL, SQLite: KHÔNG có tính năng này.                         │
  │                                                                  │
  │  Tầng 4: Audit & monitoring                                     │
  │  Log mọi query + tenant context. Alert cross-tenant anomaly.   │
  │  Periodic audit: scan query logs cho queries thiếu tenant_id.  │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘


INDEXING STRATEGY:

  Composite index bắt đầu bằng tenant_id:
  CREATE INDEX idx_customers_tenant_email ON customers(tenant_id, email);
  CREATE INDEX idx_sales_tenant_status   ON sales(tenant_id, status, created_at);

  Tại sao? Vì mọi query đều có tenant_id → index prefix match.
  Nếu không có tenant_id ở đầu index → full index scan → chậm.

  Partitioning (optional, khi data > 100GB):
  PARTITION BY LIST (tenant_id)     — mỗi tenant 1 partition
  PARTITION BY HASH (tenant_id)     — N partitions, hash distribute
  → Partition pruning: query chỉ scan đúng partition của tenant.

  ✓ Chi phí thấp nhất per tenant       ✗ Isolation phụ thuộc app code
  ✓ Schema migration 1 lần             ✗ Noisy neighbor risk
  ✓ Cross-tenant analytics dễ          ✗ Backup/restore per-tenant khó
  ✓ Connection pool hiệu quả           ✗ Data residency compliance khó
  ✓ Scale tới hàng nghìn tenants       ✗ Nếu DB down → TẤT CẢ tenants down
shared db architecture
STRUCTURE:

  All tenants share the same DB instance and tables.
  Each table has a tenant_id column — the discriminator column.

  ┌─ customers ──────────────────────────────────────────┐
  │ id       │ tenant_id  │ name        │ email            │
  ├──────────┼────────────┼─────────────┼──────────────────┤
  │ cust-01  │ acme-123   │ John Doe    │ john@acme.com    │
  │ cust-02  │ globex-456 │ Jane Smith  │ jane@globex.com  │
  │ cust-03  │ acme-123   │ Bob Lee     │ bob@acme.com     │
  │ cust-04  │ initech-789│ Alice Chen  │ alice@initech.com│
  └──────────┴────────────┴─────────────┴──────────────────┘

  Every query MUST include WHERE tenant_id = ? (or equivalent).
  Missing tenant_id = data leak. This is the most critical bug.


4 PROTECTION LAYERS (defense in depth):

  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                  │
  │  Layer 1: Application-level injection                           │
  │  Middleware/interceptor automatically injects tenant_id into    │
  │  every query. Developers do NOT write WHERE tenant_id manually  │
  │  — the system injects it.                                        │
  │                                                                  │
  │  Layer 2: Query validation                                      │
  │  Before execution: verify the query HAS a tenant_id filter.    │
  │  If not → BLOCK query, log security alert.                      │
  │                                                                  │
  │  Layer 3: Database-level enforcement (if DB supports it)         │
  │  PostgreSQL RLS: CREATE POLICY ... USING (tenant_id = ...)      │
  │  Oracle VPD: Virtual Private Database                           │
  │  DB engine filters rows automatically — even if app has a bug. │
  │  MySQL, SQLite: do NOT have this feature.                        │
  │                                                                  │
  │  Layer 4: Audit & monitoring                                    │
  │  Log all queries + tenant context. Alert on cross-tenant        │
  │  anomalies. Periodic audit: scan query logs for queries         │
  │  missing tenant_id.                                              │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘


INDEXING STRATEGY:

  Composite index starting with tenant_id:
  CREATE INDEX idx_customers_tenant_email ON customers(tenant_id, email);
  CREATE INDEX idx_sales_tenant_status   ON sales(tenant_id, status, created_at);

  Why? Because every query has tenant_id → index prefix match.
  Without tenant_id at the index prefix → full index scan → slow.

  Partitioning (optional, when data > 100GB):
  PARTITION BY LIST (tenant_id)     — 1 partition per tenant
  PARTITION BY HASH (tenant_id)     — N partitions, hash distribute
  → Partition pruning: query scans only the tenant's partition.


ADVANTAGES:                          DISADVANTAGES:

  ✓ Lowest cost per tenant            ✗ Isolation depends on app code
  ✓ Schema migration once             ✗ Noisy neighbor risk
  ✓ Cross-tenant analytics easy       ✗ Per-tenant backup/restore difficult
  ✓ Efficient connection pool         ✗ Data residency compliance difficult
  ✓ Scale to thousands of tenants     ✗ If DB goes down → ALL tenants down

ANTI-PATTERNS — AVOID THESE MISTAKES:

  ┌────────────────────────────────────────────────────────────────────────┐
  │                                                                        │
  │  ① Trust client-supplied tenantId                                     │
  │  Wrong: X-Tenant-ID header from client → used directly.              │
  │  Right: tenantId from JWT (signed, server-issued) or subdomain.      │
  │                                                                        │
  │  ② Developer manually writes WHERE tenant_id = ...                    │
  │  Wrong: developer manually adds tenant filter to every query.         │
  │  Right: framework auto-injects → developer CANNOT forget.             │
  │                                                                        │
  │  ③ Shared DB without backup enforcement                               │
  │  Wrong: only app-level injection, no DB-level safety net.             │
  │  Right: RLS (if PG) or query validator + audit + monitoring.          │
  │                                                                        │
  │  ④ God role that crosses tenant boundary                              │
  │  Wrong: "super admin" role can query any tenant data.                 │
  │  Right: platform admin access must go through a separate audit        │
  │         pipeline, time-limited, logged, NOT the same auth flow.       │
  │                                                                        │
  │  ⑤ Over-engineer isolation too early                                   │
  │  Wrong: DB-per-tenant for 10 free-tier tenants.                       │
  │  Right: Shared DB + good abstraction layer → migrate when needed.     │
  │  Good abstraction layer = swap strategy without changing business code.  │
  │                                                                        │
  │  ⑥ No cross-tenant isolation testing                                  │
  │  Wrong: only testing happy path within 1 tenant.                      │
  │  Right: integration tests: login as tenant A, try tenant B data.     │
  │         Automated: every API endpoint tested with 2 tenants.          │
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘

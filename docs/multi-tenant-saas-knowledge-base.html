Knowledge Base — Multi-Tenant SaaS

Kiến trúc Multi-Tenant SaaS

Tổng quan kiến trúc, patterns và trade-offs cho hệ thống multi-tenant SaaS. Nền tảng lý thuyết chung — không gắn với implementation cụ thể nào.

00

Multi-Tenant SaaS — Khái niệm cốt lõi

Định nghĩa: Multi-tenant = 1 application instance phục vụ nhiều tổ chức (tenants). Mỗi tenant thấy hệ thống như riêng của họ — dữ liệu cách ly, cấu hình riêng, billing riêng — nhưng chia sẻ cùng codebase và infrastructure.
core concepts
SINGLE-TENANT vs MULTI-TENANT:

  Single-tenant:                       Multi-tenant:
  ┌─────────┐ ┌─────────┐ ┌─────────┐  ┌──────────────────────────────┐
  │ App     │ │ App     │ │ App     │  │ App Instance                 │
  │ ACME    │ │ GLOBEX  │ │ INITECH │  │                              │
  │         │ │         │ │         │  │  ┌─────┐ ┌──────┐ ┌───────┐ │
  │ DB      │ │ DB      │ │ DB      │  │  │ACME │ │GLOBEX│ │INITECH│ │
  │ ACME    │ │ GLOBEX  │ │ INITECH │  │  └─────┘ └──────┘ └───────┘ │
  └─────────┘ └─────────┘ └─────────┘  │  DB (shared or partitioned) │
                                        └──────────────────────────────┘
  3 deployments, 3 DBs,                1 deployment, shared resources,
  3x ops cost, simple isolation         lower cost, complex isolation


5 TRỤ CỘT CỦA MULTI-TENANT:

  ┌────────────────────┬──────────────────────────────────────────────────┐
  │ ① Tenant Isolation │ Dữ liệu tenant A không bao giờ lộ cho tenant B. │
  │                    │ Đây là yêu cầu TUYỆT ĐỐI, không có ngoại lệ.  │
  ├────────────────────┼──────────────────────────────────────────────────┤
  │ ② Identity & Auth  │ Mỗi tenant có user pool riêng. User thuộc đúng   │
  │                    │ 1 tenant. Roles/permissions scoped by tenant.    │
  ├────────────────────┼──────────────────────────────────────────────────┤
  │ ③ Configuration    │ Mỗi tenant có config riêng: theme, features,     │
  │                    │ limits, integrations. Không ảnh hưởng tenant khác│
  ├────────────────────┼──────────────────────────────────────────────────┤
  │ ④ Resource Mgmt    │ Fair usage: tenant A dùng nhiều không làm chậm   │
  │                    │ tenant B. Rate limiting, quotas, noisy neighbor. │
  ├────────────────────┼──────────────────────────────────────────────────┤
  │ ⑤ Billing          │ Mỗi tenant có plan, usage tracking, invoice riêng│
  │                    │ Tiered pricing: Free → Pro → Enterprise.         │
  └────────────────────┴──────────────────────────────────────────────────┘

NOISY NEIGHBOR — BÀI TOÁN KINH ĐIỂN:

  Tenant A chạy report nặng → DB CPU 100% → Tenant B, C, D bị chậm.

  Giải pháp phổ biến:
  ├── Rate limiting per tenant (request/min, query/request)
  ├── Connection pool quota per tenant (max connections)
  ├── Query timeout per tier (Basic: 5s, Premium: 15s)
  ├── Dedicated resources cho VIP tenants (riêng DB, riêng cache)
  ├── Read replicas + query routing (heavy reads → replica)
  └── Background job queues per tenant (fair scheduling)

Tenant Isolation — 3 chiến lược chính

3 strategies overview
3 CHIẾN LƯỢC ISOLATION:

  ┌─────────────────────────────────────────────────────────────────────────┐
  │                                                                         │
  │  Strategy A: SHARED DATABASE                                          │
  │  Tất cả tenants cùng 1 DB, cùng tables. Cách ly bằng tenant_id column.│
  │                                                                         │
  │  ┌──────────────────────────────────────────┐                          │
  │  │ DB Instance                               │                          │
  │  │ ┌─────────────────────────────────────┐  │                          │
  │  │ │ customers table                      │  │                          │
  │  │ │  ACME row | ACME row | GLOBEX row   │  │                          │
  │  │ │  INITECH row | ACME row | ...       │  │                          │
  │  │ └─────────────────────────────────────┘  │                          │
  │  └──────────────────────────────────────────┘                          │
  │  Chi phí thấp nhất. Ops đơn giản nhất.                                 │
  │  Isolation phức tạp nhất. Noisy neighbor risk cao nhất.                │
  │                                                                         │
  ├─────────────────────────────────────────────────────────────────────────┤
  │                                                                         │
  │  Strategy B: DATABASE-PER-TENANT                                      │
  │  Mỗi tenant có DB instance riêng. Isolation hoàn toàn ở infra level.  │
  │                                                                         │
  │  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐                 │
  │  │ DB ACME  │ │ DB GLOBEX│ │DB INITECH│ │ DB ...   │                 │
  │  │ customers│ │ customers│ │ customers│ │ customers│                 │
  │  │ sales    │ │ sales    │ │ sales    │ │ sales    │                 │
  │  └──────────┘ └──────────┘ └──────────┘ └──────────┘                 │
  │  Isolation đơn giản nhất. Performance predictable.                     │
  │  Chi phí cao nhất. Ops phức tạp nhất (N DBs to manage).               │
  │                                                                         │
  ├─────────────────────────────────────────────────────────────────────────┤
  │                                                                         │
  │  Strategy C: HYBRID                                                   │
  │  Đa số tenants dùng shared DB. VIP/enterprise tenants có DB riêng.    │
  │                                                                         │
  │  ┌──────────────────────────┐ ┌──────────┐ ┌──────────┐              │
  │  │ Shared DB                │ │ DB VIP-1 │ │ DB VIP-2 │              │
  │  │ ACME | GLOBEX | 500 more│ │ INITECH  │ │ STARK    │              │
  │  └──────────────────────────┘ └──────────┘ └──────────┘              │
  │  Best of both worlds: cost-effective cho đa số, premium cho VIP.      │
  │  Phức tạp routing logic. Cần abstraction layer tốt.                  │
  │                                                                         │
  └─────────────────────────────────────────────────────────────────────────┘


KHÔNG CÓ "ĐÚNG SAI" — CHỈ CÓ TRADE-OFFS:

  ┌──────────────┬───────────┬───────────┬───────────┐
  │ Tiêu chí     │ Shared DB │ Per-Tenant│ Hybrid    │
  ├──────────────┼───────────┼───────────┼───────────┤
  │ Cost/tenant  │ Thấp     │ Cao      │ Trung bình│
  │ Isolation    │ App-level │ Infra    │ Cả hai   │
  │ Ops phức tạp │ Thấp     │ Cao      │ Trung bình│
  │ Scale tenants│ 10K+     │ 100s     │ 10K+     │
  │ Compliance   │ Cần audit │ Dễ chứng │ Per-tier │
  │ Backup       │ Phức tạp │ Đơn giản │ Per-tier │
  │ Migration    │ 1 lần    │ N lần    │ 1+N lần  │
  └──────────────┴───────────┴───────────┴───────────┘


THỨ TỰ PHÁT TRIỂN THƯỜNG GẶP:

  Startup MVP → Shared DB (nhanh, rẻ)
       │
       ▼  Product-market fit, khách VIP xuất hiện
  Growth  → Hybrid (shared default + dedicated cho VIP)
       │
       ▼  Enterprise compliance, data residency requirements
  Scale   → Hybrid + Sharding (geo-distributed, regulation)

  Hầu hết SaaS không bao giờ cần per-tenant DB cho TẤT CẢ tenants.
  Hybrid là chiến lược phổ biến nhất trong thực tế.

Shared Database — Deep Dive

shared db architecture
CẤU TRÚC:

  Mọi tenant dùng chung DB instance, chung tables.
  Mỗi table có cột tenant_id — discriminator column.

  ┌─ customers ──────────────────────────────────────────┐
  │ id       │ tenant_id  │ name        │ email            │
  ├──────────┼────────────┼─────────────┼──────────────────┤
  │ cust-01  │ acme-123   │ John Doe    │ john@acme.com    │
  │ cust-02  │ globex-456 │ Jane Smith  │ jane@globex.com  │
  │ cust-03  │ acme-123   │ Bob Lee     │ bob@acme.com     │
  │ cust-04  │ initech-789│ Alice Chen  │ alice@initech.com│
  └──────────┴────────────┴─────────────┴──────────────────┘

  Mọi query PHẢI có WHERE tenant_id = ? (hoặc tương đương).
  Thiếu tenant_id = data leak. Đây là lỗi nghiêm trọng nhất.


4 CƠ CHẾ BẢO VỆ (defense in depth):

  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                  │
  │  Tầng 1: Application-level injection                            │
  │  Middleware/interceptor tự động thêm tenant_id vào mọi query.   │
  │  Developer KHÔNG tự viết WHERE tenant_id — hệ thống inject.     │
  │                                                                  │
  │  Tầng 2: Query validation                                       │
  │  Trước khi execute: kiểm tra query CÓ tenant_id filter không.  │
  │  Không có → BLOCK query, log security alert.                    │
  │                                                                  │
  │  Tầng 3: Database-level enforcement (nếu DB hỗ trợ)             │
  │  PostgreSQL RLS: CREATE POLICY ... USING (tenant_id = ...)      │
  │  Oracle VPD: Virtual Private Database                           │
  │  DB engine tự filter rows — kể cả app có bug.                  │
  │  MySQL, SQLite: KHÔNG có tính năng này.                         │
  │                                                                  │
  │  Tầng 4: Audit & monitoring                                     │
  │  Log mọi query + tenant context. Alert cross-tenant anomaly.   │
  │  Periodic audit: scan query logs cho queries thiếu tenant_id.  │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘


INDEXING STRATEGY:

  Composite index bắt đầu bằng tenant_id:
  CREATE INDEX idx_customers_tenant_email ON customers(tenant_id, email);
  CREATE INDEX idx_sales_tenant_status   ON sales(tenant_id, status, created_at);

  Tại sao? Vì mọi query đều có tenant_id → index prefix match.
  Nếu không có tenant_id ở đầu index → full index scan → chậm.

  Partitioning (optional, khi data > 100GB):
  PARTITION BY LIST (tenant_id)     — mỗi tenant 1 partition
  PARTITION BY HASH (tenant_id)     — N partitions, hash distribute
  → Partition pruning: query chỉ scan đúng partition của tenant.


ƯU ĐIỂM:                              NHƯỢC ĐIỂM:

  ✓ Chi phí thấp nhất per tenant       ✗ Isolation phụ thuộc app code
  ✓ Schema migration 1 lần             ✗ Noisy neighbor risk
  ✓ Cross-tenant analytics dễ          ✗ Backup/restore per-tenant khó
  ✓ Connection pool hiệu quả           ✗ Data residency compliance khó
  ✓ Scale tới hàng nghìn tenants       ✗ Nếu DB down → TẤT CẢ tenants down

Database-per-Tenant — Deep Dive

db-per-tenant architecture
CẤU TRÚC:

  Mỗi tenant có DB instance (hoặc schema) riêng biệt.
  Isolation ở infrastructure level — không cần tenant_id column.

  Variant A: Instance-per-tenant (strongest isolation)

  ┌──────────┐ ┌──────────┐ ┌──────────┐
  │ PG Host 1│ │ PG Host 2│ │ PG Host 3│
  │ DB: acme │ │ DB:globex│ │DB:initech│
  │ customers│ │ customers│ │ customers│
  │ sales    │ │ sales    │ │ sales    │
  └──────────┘ └──────────┘ └──────────┘
  Hoàn toàn tách biệt. Khác host, khác process.

  Variant B: Schema-per-tenant (moderate isolation)

  ┌────────────────────────────────────────┐
  │ PG Host 1 (shared instance)            │
  │ ┌──────────┐┌──────────┐┌──────────┐ │
  │ │ schema:  ││ schema:  ││ schema:  │ │
  │ │ acme     ││ globex   ││ initech  │ │
  │ │ customers││ customers││ customers│ │
  │ │ sales    ││ sales    ││ sales    │ │
  │ └──────────┘└──────────┘└──────────┘ │
  └────────────────────────────────────────┘
  Cùng instance, khác schema. Dễ quản lý hơn variant A.


KHI NÀO CHỌN DB-PER-TENANT:

  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                  │
  │  ✓ Compliance yêu cầu data isolation (HIPAA, SOC2, GDPR)       │
  │  ✓ Tenant yêu cầu data residency (EU data ở EU, US data ở US) │
  │  ✓ Enterprise tenant trả tiền cho dedicated resources           │
  │  ✓ Workload rất khác nhau giữa tenants (1 tenant dominate)     │
  │  ✓ Tenant muốn custom DB tuning (index, extension, config)     │
  │  ✓ Tenant cần backup/restore độc lập                           │
  │                                                                  │
  │  ✗ KHÔNG phù hợp khi:                                           │
  │  ✗ Hàng nghìn tenants (ops overhead quá lớn)                   │
  │  ✗ Free/low-tier tenants (cost per tenant quá cao)             │
  │  ✗ Cross-tenant analytics needed (query N databases)           │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘


OPS CHALLENGES (N databases):

  ├── Schema migration: phải chạy trên N DBs, rollback phức tạp
  │   → cần migration orchestrator (parallel execute, track status)
  │
  ├── Monitoring: N connection pools, N sets of metrics
  │   → cần centralized monitoring (Prometheus + per-tenant labels)
  │
  ├── Backup: N backup schedules, N restore procedures
  │   → cần automated backup pipeline (snapshot → S3 → verify)
  │
  ├── Connection management: N pools → connection explosion
  │   → cần connection pooler (PgBouncer per-instance or shared)
  │
  └── Provisioning: tenant signup → create DB → migrate schema → seed
      → cần automated provisioning pipeline (< 30 seconds target)


ƯU ĐIỂM:                              NHƯỢC ĐIỂM:

  ✓ Isolation mạnh nhất                ✗ Chi phí cao nhất per tenant
  ✓ No noisy neighbor                  ✗ Ops complexity O(N)
  ✓ Custom tuning per tenant           ✗ Migration chạy N lần
  ✓ Backup/restore per-tenant dễ       ✗ Cross-tenant queries khó
  ✓ Compliance chứng minh dễ           ✗ Connection pool explosion
  ✓ Data residency đơn giản            ✗ Provisioning pipeline cần

Hybrid — Chiến lược phổ biến nhất

Thực tế: Hầu hết SaaS thành công đều dùng Hybrid. Shared DB cho 95% tenants (cost-effective), dedicated DB cho 5% VIP (premium revenue). Đây là sweet spot giữa cost và isolation.
hybrid architecture
CẤU TRÚC HYBRID:

  ┌──────────────────────────────────────────────────────────────────────┐
  │  Tenant Context Manager — routing brain                              │
  │                                                                      │
  │  tenantId → lookup tier → route to correct DB:                      │
  │                                                                      │
  │  ┌───────────────────────────────────────┐                          │
  │  │ Tier: Free, Basic, Premium            │                          │
  │  │ → Route to Shared Data DB              │                          │
  │  │   (RLS isolation, connection pool)    │                          │
  │  └───────────────────────────────────────┘                          │
  │                                                                      │
  │  ┌───────────────────────────────────────┐                          │
  │  │ Tier: Enterprise                      │                          │
  │  │ → Route to Shared Data DB              │                          │
  │  │   (RLS isolation, conn cap nâng cao)  │                          │
  │  └───────────────────────────────────────┘                          │
  │                                                                      │
  │  ┌───────────────────────────────────────┐                          │
  │  │ Tier: VIP                             │                          │
  │  │ → Route to Dedicated DB                │                          │
  │  │   (own instance, custom config)       │                          │
  │  └───────────────────────────────────────┘                          │
  └──────────────────────────────────────────────────────────────────────┘

       Shared Data DB                Dedicated DBs
  ┌──────────────────────┐   ┌─────────┐  ┌─────────┐
  │ 500 tenants          │   │ INITECH │  │ STARK   │
  │ (Free/Basic/Premium/ │   │ (VIP)   │  │ (VIP)   │
  │  Enterprise)         │   │         │  │         │
  │                      │   │         │  │         │
  │ tenant_id isolation  │   │ no RLS  │  │ no RLS  │
  │ RLS + app-level      │   │ needed  │  │ needed  │
  │ shared resources     │   │ custom  │  │ custom  │
  └──────────────────────┘   └─────────┘  └─────────┘


YÊU CẦU KIẾN TRÚC CHO HYBRID:

  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                  │
  │  ① Abstraction Layer                                            │
  │  Application code KHÔNG BIẾT đang dùng shared hay dedicated.    │
  │  Cùng API: ctx.db.query('sales').where({...}).execute()         │
  │  Routing transparent — chỉ infra layer biết.                    │
  │                                                                  │
  │  ② Pool Router                                                  │
  │  tenantId → tier lookup → pool selection → connection           │
  │  Shared tenants → SharedPool (PgBouncer → Shared DB)           │
  │  VIP tenants → VipPool-{name} (direct → Dedicated DB)          │
  │                                                                  │
  │  ③ Schema Consistency                                           │
  │  Shared DB và Dedicated DBs PHẢI có cùng schema version.       │
  │  Migration runner: chạy trên Shared DB + MỌI Dedicated DBs.    │
  │  Version mismatch → tenant bị block cho đến khi migrate xong.  │
  │                                                                  │
  │  ④ Tenant Promotion Path                                       │
  │  Tenant upgrade Free → VIP:                                     │
  │  1. Provision dedicated DB                                      │
  │  2. Migrate schema + seed                                       │
  │  3. Export tenant data from Shared DB                           │
  │  4. Import data to Dedicated DB                                 │
  │  5. Update routing config                                       │
  │  6. Verify + switch (ideally zero-downtime)                     │
  │                                                                  │
  │  ⑤ Unified Observability                                       │
  │  Metrics, logs, alerts phải cover cả Shared + Dedicated DBs.   │
  │  Dashboard nhìn thấy ALL tenants dù họ ở DB nào.              │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘


DECISION FLOW — tenant mới vào pool nào:

  Tenant signup
       │
       ├── Plan = Free/Basic/Premium? → Shared DB
       │
       ├── Plan = Enterprise?
       │   ├── Contract yêu cầu dedicated? → Dedicated DB
       │   └── Không yêu cầu?  → Shared DB (upgrade later nếu cần)
       │
       └── Plan = VIP? → Dedicated DB (auto-provision)

  │                        │                  │                  │                  │
  └────────────────────────┴──────────────────┴──────────────────┴──────────────────┘


REAL-WORLD EXAMPLES:

  Shared DB:
  ├── Slack (early days) — millions of workspaces, shared PostgreSQL
  ├── Shopify (shared tables) — millions of stores, tenant_id per row
  ├── Notion — shared database, application-level isolation
  └── Hầu hết SaaS startup với < 1000 tenants

  DB-per-Tenant:
  ├── Salesforce — per-org database pods
  ├── Healthcare SaaS (HIPAA) — patient data isolation required
  ├── Government SaaS — data residency mandated
  └── Enterprise B2B với < 100 khách hàng lớn

  Hybrid:
  ├── AWS (internal) — shared resources + dedicated for large accounts
  ├── Atlassian — shared for small teams, dedicated for enterprise
  ├── Most mature B2B SaaS — default shared, opt-in dedicated
  └── Hầu hết SaaS scale-up sau Series B

Authentication — Identity trong Multi-Tenant

authentication patterns
NGUYÊN TẮC: USER LUÔN THUỘC VỀ 1 TENANT

  User = { userId, tenantId, email, passwordHash, roles[] }

  Đăng nhập = xác minh identity + resolve tenant context.
  Mọi action sau đó đều scoped bởi tenantId từ token.


JWT — PATTERN PHỔ BIẾN NHẤT:

  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                  │
  │  Login Flow:                                                    │
  │                                                                  │
  │  POST /auth/login { email, password, tenant: "acme" }           │
  │       │                                                          │
  │       ▼                                                          │
  │  Auth Service:                                                   │
  │  1. Resolve tenant (subdomain / body / email domain)            │
  │  2. Verify credentials against tenant's user store              │
  │  3. Issue JWT:                                                   │
  │                                                                  │
  │  {                                                               │
  │    "sub": "user-42",                                             │
  │    "tenantId": "acme-123",                                      │
  │    "tenantTier": "premium",                                     │
  │    "roles": ["sales-manager"],                                   │
  │    "iat": 1708000000,                                            │
  │    "exp": 1708000900     ← 15 minutes                           │
  │  }                                                               │
  │       │                                                          │
  │       ▼                                                          │
  │  Client stores JWT → sends in Authorization header              │
  │  Server validates → extract tenantId → scope everything         │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘

  CRITICAL: tenantId trong JWT = source of truth.
  Server KHÔNG BAO GIỜ tin client-supplied tenantId ngoài JWT.
  JWT signed bởi server → client không thể tamper tenantId.


TENANT RESOLUTION STRATEGIES:

  ┌────────────────────┬──────────────────────────────────────────────┐
  │ Strategy           │ Cách hoạt động                              │
  ├────────────────────┼──────────────────────────────────────────────┤
  │ Subdomain          │ acme.app.com → extract "acme" → lookup      │
  │                    │ Phổ biến nhất. UX tốt. DNS wildcard *.app.com│
  ├────────────────────┼──────────────────────────────────────────────┤
  │ Custom domain      │ crm.acme.com → CNAME → lookup domain map    │
  │                    │ Enterprise feature. Cần SSL cert per domain │
  ├────────────────────┼──────────────────────────────────────────────┤
  │ Email domain       │ john@acme.com → extract "acme.com" → lookup  │
  │                    │ Auto-join. Dùng khi login, không runtime    │
  ├────────────────────┼──────────────────────────────────────────────┤
  │ JWT claim          │ Token chứa tenantId → trusted source         │
  │                    │ Dùng cho API calls sau khi đã authenticate  │
  ├────────────────────┼──────────────────────────────────────────────┤
  │ Path prefix       │ /api/tenants/acme-123/... → extract from URL │
  │                    │ Dùng cho internal APIs, admin panels        │
  └────────────────────┴──────────────────────────────────────────────┘


TOKEN LIFECYCLE:

  Access Token (JWT, 15min-1hr)
  ├── Short-lived. Chứa tenantId, roles.
  ├── Stateless verification (chỉ cần public key).
  └── Revocation: khó (phải blacklist hoặc chờ expire).

  Refresh Token (opaque, 7-30 days)
  ├── Long-lived. Stored in DB (revocable).
  ├── Dùng để lấy Access Token mới.
  └── Revoke: DELETE from DB → effective immediately.

  Token Revocation Pattern:
  ├── Approach 1: Blacklist JWT ID trong Redis (TTL = token lifetime)
  ├── Approach 2: Short access token (15min) + refresh token rotation
  └── Approach 3: Token version per-user (bump version = revoke all)


SSO / ENTERPRISE AUTH:

  Enterprise tenants thường yêu cầu:
  ├── SAML 2.0 integration (Okta, Azure AD, OneLogin)
  ├── OIDC / OAuth2 (Google Workspace, Microsoft Entra)
  ├── SCIM provisioning (auto-create/deactivate users)
  └── Mandatory MFA per tenant policy

  Flow: User → Tenant's IdP → SAML assertion → App validates
       → Map SAML attributes to local user + tenant → Issue JWT
       → Từ đây trở đi giống flow thường (JWT-based)

Authorization — RBAC trong Multi-Tenant

authorization patterns
NGUYÊN TẮC: ROLES + PERMISSIONS SCOPED BY TENANT

  Mỗi tenant có BỘ ROLES RIÊNG. "Admin" của tenant A ≠ "Admin" của tenant B.
  Cùng user role "sales-manager" → permissions có thể khác nhau giữa tenants.


RBAC MODEL:

  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                  │
  │  User ──belongs to──▶ Tenant                                    │
  │    │                                                             │
  │    └──has──▶ Role(s) ──within──▶ Tenant                         │
  │               │                                                  │
  │               └──grants──▶ Permission(s)                         │
  │                                                                  │
  │  Ví dụ:                                                        │
  │  User "john" @ Tenant "acme"                                    │
  │    ├── Role: "sales-manager"                                    │
  │    │   ├── sales:read, sales:create, sales:update               │
  │    │   ├── customers:read                                       │
  │    │   └── reports:read                                         │
  │    └── Role: "team-lead"                                        │
  │        └── users:read, users:invite                             │
  │                                                                  │
  │  Effective permissions = UNION of all roles' permissions        │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘


3 MÔ HÌNH PHỔ BIẾN:

  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                  │
  │  ① Fixed Roles (đơn giản nhất)                                  │
  │                                                                  │
  │  Hệ thống định nghĩa sẵn N roles. Tenant chỉ assign user.     │
  │  VD: Owner, Admin, Manager, Member, Viewer                     │
  │                                                                  │
  │  ✓ Đơn giản implement. Dễ hiểu.                                │
  │  ✗ Không flexible. Tenant muốn custom → không được.            │
  │                                                                  │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  ② Configurable Roles (phổ biến nhất)                           │
  │                                                                  │
  │  Hệ thống cung cấp default roles + tenant có thể tạo thêm.    │
  │  Mỗi role = set of permissions. Tenant tự chọn permissions.    │
  │                                                                  │
  │  Default: Owner, Admin, Member                                  │
  │  Tenant "ACME" tạo thêm: "Sales Lead", "Read-only Auditor"    │
  │                                                                  │
  │  ✓ Flexible nhưng manageable.                                   │
  │  ✓ Permissions do hệ thống định nghĩa (bounded).               │
  │  ✗ UI phức tạp hơn.                                             │
  │                                                                  │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  ③ ABAC — Attribute-Based (phức tạp nhất)                       │
  │                                                                  │
  │  Policy engine evaluate: user attributes + resource attributes  │
  │  + environment (time, IP, device) → allow/deny.                 │
  │                                                                  │
  │  VD: "Sales reps can only edit deals < $50K. Deals ≥ $50K      │
  │       require manager role. After 6pm, read-only for all."      │
  │                                                                  │
  │  ✓ Extremely flexible. Handles complex business rules.          │
  │  ✗ Complex to implement, debug, and explain to users.           │
  │  ✗ Performance overhead (policy evaluation per request).        │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘


PERMISSION CHECK — Ở ĐÂU TRONG PIPELINE:

  Request → Auth (who?) → Tenant (which org?) → Authz (allowed?) → Execute

  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                  │
  │  Route-level: check trước khi gọi handler                      │
  │  Middleware: if (!user.can('sales:create')) → 403 Forbidden     │
  │  Nhanh. Block sớm. Phù hợp CRUD operations.                    │
  │                                                                  │
  │  Field-level: check per field trong response                    │
  │  User "viewer" thấy sales.title, nhưng KHÔNG thấy sales.amount │
  │  Phức tạp hơn. Cần field-level access control list.            │
  │                                                                  │
  │  Row-level: check per record                                    │
  │  User "sales-rep" chỉ thấy deals assigned cho mình.            │
  │  Cần filter query: WHERE assigned_to = currentUserId            │
  │  Hoặc DB-level: RLS policy per-user (nặng, ít dùng cho RBAC). │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘


SYSTEM-LEVEL vs TENANT-LEVEL ROLES:

  System roles (platform operator):
  ├── Platform Admin: quản lý tenants, billing, system config
  ├── Support Agent: read-only access vào tenant data (with audit)
  └── DevOps: infrastructure management, no business data access

  Tenant roles (within a tenant):
  ├── Tenant Owner: full control trong org của họ
  ├── Tenant Admin: user management, config
  ├── Custom roles: per-tenant defined
  └── KHÔNG BAO GIỜ cross-tenant access

  Ranh giới tuyệt đối: Tenant role KHÔNG BAO GIỜ grant quyền
  truy cập data tenant khác. Kể cả Tenant Owner.

Decision Matrix — Chọn gì cho dự án của bạn

decision framework
ISOLATION STRATEGY — CHỌN THEO CONTEXT:

  Trả lời 5 câu hỏi → biết nên chọn gì:

  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                  │
  │  Q1: Bao nhiêu tenants dự kiến trong 2 năm tới?                │
  │  ├── < 50 tenants, mỗi tenant lớn   → DB-per-Tenant            │
  │  ├── 50 - 500 tenants               → Hybrid                   │
  │  └── > 500 tenants                  → Shared DB hoặc Hybrid    │
  │                                                                  │
  │  Q2: Có yêu cầu compliance đặc biệt?                           │
  │  ├── HIPAA / FedRAMP / data residency → DB-per-Tenant cho tier đó│
  │  └── SOC2 / ISO 27001               → Shared DB + RLS đủ       │
  │                                                                  │
  │  Q3: Revenue model?                                             │
  │  ├── Self-serve, low ARPU ($10-50/mo)  → Shared DB              │
  │  ├── Sales-led, high ARPU ($500+/mo)   → Hybrid                │
  │  └── Enterprise-only ($5K+/mo)         → DB-per-Tenant          │
  │                                                                  │
  │  Q4: Team size?                                                 │
  │  ├── < 5 engineers    → Shared DB (đơn giản nhất)                │
  │  ├── 5-20 engineers   → Hybrid                                   │
  │  └── > 20 engineers   → Bất kỳ (có đủ bandwidth)              │
  │                                                                  │
  │  Q5: DB engine nào?                                             │
  │  ├── PostgreSQL → Shared + RLS (tốt nhất cho shared)              │
  │  ├── MySQL      → Shared (no RLS → cần app-level chặt hơn)       │
  │  └── SQLite     → Per-tenant (file-based, natural isolation)      │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────────────────────┐

  │  Authentication:                                                │
  │                                                                  │

  │                      Enterprise tenants yêu cầu SSO.            │

  │  Growth           → Configurable roles + permissions            │

ANTI-PATTERNS — TRÁNH NHỮNG SAI LẦM NÀY:

  ┌────────────────────────────────────────────────────────────────────────┐
  │                                                                        │
  │  ① Trust client-supplied tenantId                                     │
  │  Sai: Header X-Tenant-ID từ client → dùng trực tiếp.                 │
  │  Đúng: tenantId từ JWT (signed, server-issued) hoặc subdomain.       │
  │                                                                        │
  │  ② Developer tự viết WHERE tenant_id = ...                            │
  │  Sai: mỗi query developer tự thêm tenant filter.                     │
  │  Đúng: framework tự inject → developer KHÔNG THỂ quên.               │
  │                                                                        │
  │  ③ Shared DB without backup enforcement                               │
  │  Sai: chỉ dùng app-level injection, không có DB-level safety net.    │
  │  Đúng: RLS (nếu PG) hoặc query validator + audit + monitoring.       │
  │                                                                        │
  │  ④ God role that crosses tenant boundary                              │
  │  Sai: "super admin" role có thể query bất kỳ tenant data.            │
  │  Đúng: platform admin access phải qua riêng audit pipeline,          │
  │        time-limited, logged, và KHÔNG dùng cùng auth flow.            │
  │                                                                        │
  │  ⑤ Over-engineer isolation quá sớm                                    │
  │  Sai: DB-per-tenant cho 10 tenants free tier.                         │
  │  Đúng: Shared DB + tốt abstraction layer → migrate khi cần.          │
  │  Abstraction layer tốt = swap strategy mà không đổi business code.     │
  │                                                                        │
  │  ⑥ Không test cross-tenant isolation                                  │
  │  Sai: chỉ test happy path trong 1 tenant.                             │
  │  Đúng: integration tests: login tenant A, try access tenant B data.  │
  │        Automated: mọi API endpoint test với 2 tenants song song.      │
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘


CHECKLIST TRƯỚC KHI GO-LIVE:

  Isolation:
  ☐ Mọi DB query tự động có tenant_id filter (không manual)
  ☐ DB-level safety (RLS / query validator / cả hai)
  ☐ Cross-tenant test suite (automated, chạy mỗi deploy)
  ☐ Audit log mọi data access với tenant context

  Authentication:
  ☐ JWT chứa tenantId, signed, short-lived
  ☐ Refresh token rotation + revocation mechanism
  ☐ Tenant resolution strategy rõ ràng (subdomain / domain / JWT)
  ☐ Rate limiting per tenant per endpoint

  Authorization:
  ☐ Roles scoped by tenant (không global roles cho business data)
  ☐ Permission check trước mọi handler (middleware)
  ☐ Platform admin access qua riêng audit pipeline
  ☐ No hardcoded role checks (dùng permission-based, không role name)
